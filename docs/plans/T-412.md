### Behavioral contract — query layer

5 queries via TanStack Query, each `staleTime: 30_000` (inherited from `main.tsx` global default per L-001) + `refetchInterval: 30_000` (per OQ-7=A explicit 30s). Per WG#X each query keys on `selectedBots` array (multi-select; empty = "all bots"):

1. `useQuery({queryKey: ["overview-positions", selectedBots], queryFn: () => apiFetch<OpenPositionListResponse>("/api/positions/" + maybeBotIdParam)})` — when `selectedBots.length === 1`, append `?bot_id=<id>`; when 0 or >1, fetch all and filter client-side via `.filter(p => selectedBots.length === 0 || selectedBots.includes(p.bot_id))`.
2. `useQuery({queryKey: ["overview-pnl-24h", selectedBots, from, to], queryFn: ...})` — `/api/analytics/pnl-series?bucket=hour&from=<24h_ago>&to=<now>` (no `bot_id` param when multi/all; backend's omit-bot_id path returns cross-bot aggregate per T-406 contract; for single-bot selection append `&bot_id=<id>`).
3. `useQuery({queryKey: ["overview-signals-received", from, to], queryFn: () => apiFetch<SignalListResponse>("/api/signals/?from=...&to=...&limit=1")})` — bare query for received count.
4. `useQuery({queryKey: ["overview-signals-accepted", from, to], ...})` — `?ingestion_status=validated&limit=1`.
5. `useQuery({queryKey: ["overview-signals-rejected", from, to], ...})` — `?ingestion_status=invalid&limit=1` per OQ-C1 (drop "duplicate" from rejected — neither truly accepted nor rejected; visible to operator via T-419 audit / T-413 raw signals feed if needed).

**Per WG — bot_id filter on positions**: `/api/positions/?bot_id=` accepts ONE bot per query. For multi-select with N>1 bots, T-412 fetches all positions cross-bot (no `?bot_id=`) and filters client-side. This is acceptable for MVP scale (BRIEF §14 expects <10 bots × <50 positions = ~500 row max). Per §0.8 anti-hypothetical: NO N parallel `?bot_id=` queries (would be 10× hammer pattern with no payoff at this scale).

**Per WG — Aggregate virtual balance tile**: T-412 renders `<OverviewTile placeholder=true title="Virtual balance" subtitle="Coming F4+" />` — operator-facing message indicates this is a known F4+ gap (see Open Questions OQ-A). NO fake "$0.00" display.

**Per WG — Alert count tile**: same pattern — `<OverviewTile placeholder=true title="Alerts (24h)" subtitle="Coming F4+" />`. See OQ-B.

**Per WG — TimeRangePicker mode**: rendered with all 5 presets + Custom button visible per OQ-D=B (not grayed-out). BUT only "24h" preset effective for queries — query window is hardcoded to 24h regardless of picker state in T-412 (Overview is per-spec a 24h dashboard). TimeRangePicker `value={range}` controlled state still updates locally to allow operator to "preview" picker UX, but `from`/`to` derived for queries are FIXED at `now - 24h` and `now`. **Important** — this is a deliberate UX gap surfaced as a comment in `index.tsx` ("Picker visible for top-bar consistency per §14.2; T-413+ sections will respect picker selection. Overview is per-spec 24h-locked."). Operator visibility into the limitation > silently grayed-out buttons.
