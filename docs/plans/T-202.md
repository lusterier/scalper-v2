## Goal

Single deliverable: **migration 0005** introducing the live execution-ledger FK chain — `orders` (regular table) + `trades` (regular table, FK to orders) + `executions` (hypertable, FK to orders + trades) — per §7.2 verbatim. Per-migration test verifies the post-0005 artifact shape end-to-end against a throwaway DB.

After this task ships, T-203 (`position_state` migration 0006), T-212 (`paper_*` migration 0008), T-216/T-217 (order pipeline + lifecycle FSM), T-218/T-219 (execution dispatcher + cumulative-delta reconcile), T-220 (P&L audit), T-221 (post-restart reconciliation) all unblock — every one of them reads or writes one of these three tables.

## Spec references

- **§7.2** DDL verbatim — `orders` (lines 956-981), `trades` (lines 983-1011), `executions` (lines 1013-1034). Plan ships these verbatim with no schema deviation.
- **§N8** forward-only Alembic migrations with per-migration test (line 145).
- **§7.4** "migration first to need timescaledb owns CREATE EXTENSION" — already satisfied at 0001; 0005 just reuses the existing extension.
- **§18.3** retention summary (lines 2427-2440) — `trades`, `executions` are listed as **forever** retention. **No retention policy. No compression policy.** This drives the migration body shape: `executions` is a hypertable but does NOT get `add_retention_policy` / `add_compression_policy` calls (contrast 0003 `ohlc_1m` and 0004 `features` which are time-bounded).
- **§5.13** Decimal — qty/price use `NUMERIC(30, 12)`, fees use `NUMERIC(20, 8)`, P&L uses `NUMERIC(20, 4)`. Brief verbatim, no precision drift.
- **§N1** UTC — every timestamp column is `TIMESTAMPTZ`. No naked timestamps.
- **§20** hazards — H-018 ("Close trade by PK only") is referenced at the schema level: BIGSERIAL `id` exists, so the upper-layer query helper (`packages/db/queries/trades.py`, T-216/T-217 owners) can do `WHERE id = $1`. Migration enforces nothing schema-side; it just provides the PK to make PK-keyed updates possible. Plan-doc records this for traceability.

retention or compression policies (contrast 0003/0004). The hypertable
shape still buys insert-rate scaling and natural time-range chunk
pruning during P&L audit window queries (T-220 reads "last 3h per
sub-account").

H-018 ("Close trade by PK only") is a query-layer invariant, not a
schema invariant. The BIGSERIAL `trades.id` exposed here is what
`packages/db/queries/trades.py` (T-216/T-217 owners) will key updates
against. Migration just provides the PK; the helper layer enforces
WHERE-id-only.
"""

## Edge cases

- **Identity vs BIGSERIAL.** Brief §7.2 spells `id BIGSERIAL`; T-001 / T-002 / T-004 precedent renders this as `sa.BigInteger() + sa.Identity(always=False)` (SQLAlchemy 2.0 idiom). Same on-disk semantics (bigint + sequence backed). Plan ships per existing precedent, NOT raw `BIGSERIAL` SQL.
- **`executions.id` is NOT a standalone PK.** It's a column inside the composite PK `(executed_at, id)` — required by TimescaleDB hypertable partitioning on `executed_at`. The auto-incrementing `Identity` clause still works inside a composite PK; the engine generates monotonic IDs but uniqueness is enforced over the composite tuple. (Pattern is verbatim from 0002 signals migration.)
- **`trades.close_order_id` has NO FK** per §7.2 verbatim. Close path may produce a row in `orders` (separate close order) OR may not (e.g., reduce-only close on the same order ID — depending on T-216b implementation). Brief leaves the FK off; plan ships per brief verbatim. Surfaced as OQ for confirmation.
- **`executions.trade_id` IS nullable** with FK. An execution event may arrive before its trade row materialises (T-216b commits the trade row in the same transaction as fills, but partial-fill / reconcile paths could land an execution before trade row exists). FK still enforced when `trade_id` is non-null.
- **No CHECK constraints on enum-like TEXT columns** (`exchange`, `side`, `order_type`, `status`, `close_reason`, `exec_type`). Brief specifies enum values via comment, not via CHECK. Mirror 0001 / 0002 precedent: comments document; constraints stay at the application layer.
- **NUMERIC precision split**: §5.13 + §7.2 split:
  - `NUMERIC(30, 12)` for qty + prices (covers full Bybit qty-step + price-tick precision; H-015 Decimal preservation).
  - `NUMERIC(20, 8)` for `executions.fee` (Bybit fees come at 8 decimal places, smaller magnitude than P&L).
  - `NUMERIC(20, 4)` for P&L + USD-denominated values (`notional_usd`, `realized_pnl`, `fees_paid`). Pin to 4 decimal places — sufficient for USD cents and basis-point operations; matches `trade_pnl_deltas.delta_amount` planned for T-220.
- **§N1 UTC**: every `TIMESTAMPTZ` is constructed via `sa.TIMESTAMP(timezone=True)`. Mirror precedent.
- **H-018 placement**: schema-level guard is the BIGSERIAL `id` PK on `trades`. Query-layer enforcement (`WHERE id = $1` only) lives in `packages/db/queries/trades.py` (T-216/T-217 owners). Plan-doc records this for traceability.
- **Alembic revision string**: `"0005"`, `down_revision = "0004"`. Standard.
- **No autogenerate**. Hand-written, mirror precedent 0001-0004.

## Decisions committed (operator-approved 2026-04-27 — "use defaults")

All ten OQs resolved with the proposed default. No overrides.

1. **OQ-1 — `trades.close_order_id`**: **default A — no FK** (per §7.2 verbatim). Close path may not always produce a separate orders row (reduce_only path; emergency-close); leaving FK off avoids edge-case insert failures. If at T-216b downstream pattern shows close_order_id always points at a real row, retro-add via ALTER TABLE in a future migration.

2. **OQ-2 — Anti-test no-policy on executions**: **default — include**. Sub-check (f) asserts ZERO `policy_retention` / `policy_compression` rows for `executions` per §18.3 forever-retention. Costs ~10 LOC; protects against accidental copy-paste of 0003/0004 retention/compression blocks.

3. **OQ-3 — NUMERIC precision smoke values**: **default A — max-precision**. Smoke INSERT uses `Decimal("0.123456789012")` qty + `Decimal("65000.123456789012")` price to pin the §5.13 12-digit fractional precision invariant. Realistic-value coverage lives at the query-helper / adapter layer (T-216+).

4. **OQ-4 — Test bot_id**: **default A — UUID-suffixed**. `f"test_t202_{uuid.uuid4().hex[:8]}"`. Defensive over simple; costs 1 LOC.

5. **OQ-5 — Test function shape**: **default — monolithic**. Single `test_migration_0005_creates_orders_trades_executions_chain` with sub-checks inline; mirror 0004.
