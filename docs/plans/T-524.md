# T-524 — Bot-level concurrent-trades caps

**Status:** Plan-stage (pre Gate 1 plan-reviewer)
**Phase:** F5 — Pre-live operational hardening cluster (ADR-0011 risk-management cluster; sibling of T-525 + T-526)
**Branch:** `feat/T-524-concurrent-caps`
**Mirror precedent:** T-526 cooldown gate (shipped 2026-05-15, commit `c1363f4`) — same pre-scoring derive-from-trades + silent-skip + Prom-counter pattern; RiskSection forward-compat container; `packages/db/queries/trades.py` extension.

## Purpose

Per-bot signal-acceptance cap that blocks new entries when the bot already holds `max_open_trades_per_bot` open positions, or when the exchange-mode realm holds `max_open_trades_global` open positions. Enforced as a pre-scoring gate at strategy-engine `consumer.py` between BRIEF §9.4 step 3b (symbol filter) and 3c (signal_id resolve), AFTER the T-526 cooldown gate. State derived from `trades` / `paper_trades` on every signal arrival (no orders.events subscribe, no in-memory counter, no restart reconcile — count is a cheap index-covered query; consistent with T-526 OQ-1=A).

## Hand verification

### Cap arithmetic worked example

`max_open_trades_per_bot=3`, `max_open_trades_global=10`. Bot `alpha`, exchange_mode=`paper`.

| State | per-bot open (paper_trades WHERE bot_id='alpha' AND status='open') | global open (paper_trades WHERE status='open') | Decision |
|-------|-------------------------------------------------------------------|------------------------------------------------|----------|
| A | 2 | 7 | not blocked (2 < 3 and 7 < 10) — signal proceeds to 3c |
| B | 3 | 7 | **blocked** reason=`max_open_trades_per_bot` current=3 cap=3 (3 >= 3; a new entry would be the 4th) |
| C | 1 | 10 | **blocked** reason=`max_open_trades_global` current=10 cap=10 (per-bot 1<3 passes; global 10>=10 trips) |
| D | 3 | 10 | **blocked** reason=`max_open_trades_per_bot` (per-bot checked first; deterministic precedence) |
| E | 5 | 12 | **blocked** reason=`max_open_trades_per_bot` current=5 cap=3 (cap lowered post-open; over-cap still blocks) |

**Block predicate**: `current_open_count >= cap` (NOT `> cap`). Rationale: `max_open_trades_per_bot=3` means "the bot may hold at most 3 concurrently"; accepting a new signal when 3 are already open would create a 4th → violate. So block when `count >= cap`.

**Disabled-knob short-circuit**: if `max_open_trades_per_bot == 0 AND max_open_trades_global == 0` → return `CapsDecision(blocked=False, ...)` BEFORE any DB query (mirror T-526 `_compute_limit` short-circuit). If only one knob is 0, only its query is skipped (per-bot query issued only when `max_open_trades_per_bot > 0`; global query only when `max_open_trades_global > 0`).

### count_open_trades SQL shape

```sql
-- per-bot (bot_id given):
SELECT count(*) FROM <trades|paper_trades> WHERE bot_id = $1 AND status = 'open'
-- global (bot_id None):
SELECT count(*) FROM <trades|paper_trades> WHERE status = 'open'
```

Index `trades_bot_status (bot_id, status)` / `paper_trades_bot_status (bot_id, status)` (migrations 0005:168 / 0008:149) covers the per-bot predicate; global `WHERE status='open'` is a bounded scan (open trades are few — risk caps keep the count small by construction). L-021: `$1` (bot_id) is column-direct equality on TEXT → L-021-safe, no `::text` cast. No other parameters (caps are Python-side int compares, NOT SQL `LIMIT`/arithmetic binds).
