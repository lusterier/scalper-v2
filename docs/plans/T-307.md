## Purpose

Land the evaluator pipeline per BRIEF ¬ß10.4 verbatim pseudocode ‚Äî the central scoring loop that takes `BotConfig` + `SignalValidated` + injected `FeatureResolver` (T-306) and produces a `ScoringResult` (T-300). ## Validation rules

1. **Per-rule iteration order**: iterate `bot_config.scoring.rules` in declaration order. `RuleResult` list mirrors that order.
2. **Resolver call per rule**: `resolver.resolve(rule_feature=rule.feature, signal=signal, max_staleness_sec=rule.max_staleness_sec)`.
3. **Resolver status mapping**:
   - `status="ok"` ‚Üí invoke `rule.condition.evaluate(ctx)` per step 4.
   - `status="data_missing"`: if `rule.required` ‚Üí emit `scoring_required_feature_missing` (info) + return `ScoringResult(decision="reject", reason="required_feature_missing", ...)` per Validation rule #11. Else: emit **H-019 `scoring_failed_open`** (warning, reason=`data_missing`) + `RuleResult(name=rule.name, weight=rule.weight, applied_weight=0.0, result="data_missing", error=resolver_result.error_info)`; continue.
   - `status="data_stale"`: if `rule.required` ‚Üí emit `scoring_required_feature_stale` (info) + return `ScoringResult(decision="reject", reason="required_feature_stale", ...)`. Else: emit **H-019 `scoring_failed_open`** (warning, reason=`data_stale`) + `RuleResult(result="data_stale", ...)`; continue.
4. **Condition.evaluate** receives `RuleContext(signal, feature_snapshot={resolved_feature_ref: feature_value}, feature_ref=resolved_feature_ref)`. Composite/series sub-conditions see THEIR rule's single feature value (T-307 v1 limitation per OQ-2 ‚Äî multi-feature composition deferred to future task).
5. **Exception during condition.evaluate**:
   - `rule.on_error == "reject"` ‚Üí emit `scoring_rule_error_reject` (info) + return `ScoringResult(decision="reject", reason="rule_error", ...)` per Validation rule #11.
   - `rule.on_error == "skip"` (default): emit **H-019 `scoring_failed_open`** (warning, reason=`condition_exception`) + `RuleResult(result="error_skipped", error={"error": repr(exc)})`; continue.
6. **Outcome handling** when condition returns `(outcome, error_info)`:
   - `outcome is True`: `applied_weight = rule.weight`; `total_score += applied_weight`.
   - `outcome is False`: `applied_weight = 0.0`. `error_info` propagated to `RuleResult.error` even on False (debugging). Result string is `"True"` or `"False"`.
7. **`applies_when` gate**: T-307 v1 SKIPS `applies_when` evaluation. `rule.applies_when: dict[str, Any] | None` is T-300 placeholder; T-308 YAML loader will narrow to `Condition` in a follow-up task. Per OQ-1, plan documents this as a v1 gap (rules always evaluated; `applies_when` ignored).
8. **Decision logic** (post-loop):
   - `mode == "passthrough"` ‚Üí `decision = "passthrough"` (regardless of total_score).
   - `total_score >= trigger_threshold` ‚Üí `decision = "execute"`.
   - else ‚Üí `decision = "reject"` with `reason = "score_below_threshold"`.
9. **`feature_snapshot` serialization**: dict keyed by resolved feature_ref; value is `{"value_num": str-or-None, "value_bool": bool-or-None, "value_json": dict-or-None}`. Decimal serialized as string per ¬ßN1. T-300 `ScoringResult.feature_snapshot: dict[str, dict[str, Any]]` opaque outer type satisfied.
10. **`config_version`** = `bot_config.version`.
11. **Early-reject `ScoringResult` field-population invariant**: when evaluator returns mid-loop (`required_feature_missing` / `required_feature_stale` / `rule_error`), the returned `ScoringResult` MUST populate all 7 required fields:
    - `decision = "reject"`
    - `reason = "required_feature_missing"` | `"required_feature_stale"` | `"rule_error"`
    - `total_score = sum(applied_weights up to rejecting rule)`  (typically 0.0 if first rule rejects, but partial sum if reject happens mid-loop)
    - `threshold = bot_config.scoring.trigger_threshold`
    - `rule_results = partial list (rules processed so far INCLUDING the rejecting one)`
    - `feature_snapshot = partial dict (entries for rules that reached condition.evaluate)`
    - `config_version = bot_config.version`
12. **Post-loop `ScoringResult.reason`**:
    - `decision = "execute"` ‚Üí `reason = None`
    - `decision = "reject"` (post-loop, score-below-threshold) ‚Üí `reason = "score_below_threshold"`
    - `decision = "passthrough"` ‚Üí `reason = None`

## Edge cases

- **Empty rules list**: `bot_config.scoring.rules == []` ‚Üí `total_score = 0.0`. If `mode == "passthrough"` ‚Üí `decision = "passthrough"`. Else: `0.0 >= threshold` (likely False) ‚Üí `decision = "reject"` reason `"score_below_threshold"`.
- **All rules return False**: `total_score = 0.0`. Same logic as empty rules.
- **Single rule reject (required + missing)**: returns immediately without populating subsequent rules' `RuleResult`. `rule_results` list is partial.
- **Single rule reject (on_error=reject + exception)**: same partial-results semantic.
- **Negative weights**: rule with `weight=-1.5` evaluates True ‚Üí `applied_weight=-1.5`; subtracts from total. Per ¬ß10.1 "negative = subtracts". Documented at T-300.
- **Mode=passthrough + reject-pre-conditions**: `data_missing+required` or `on_error=reject` exit before reaching mode check; passthrough mode does NOT override pre-evaluation rejection.
- **`error_info` from condition (False outcome with error dict)**: `error_info` IS propagated to `RuleResult.error` even when outcome is False. Composite NotCondition surprise (NOT-of-error ‚Üí True) still records error in error_info if condition reports it; in v1, `RuleResult.error` is the audit trail.
- **Condition raises during construction-time validation**: NOT T-307's concern; T-308 YAML loader catches at config-load time.
- **`feature_snapshot` size**: only resolved features (those that reached condition.evaluate) are included; `data_missing`/`data_stale`/`applies_when=False` rules do NOT add to snapshot. Operator can correlate via `RuleResult.error`.

## Hand verification

T-307 has **boolean accumulation** + scalar `total_score: float` += `applied_weight: float` arithmetic. Math-validator gate IS in scope (`packages/scoring/`).

Hand-fixtures:

- **Single-rule execute**: `BotConfig(rules=[Rule(name="r1", weight=2.5, condition=Gt(50000))], threshold=2.0, mode="active")`; signal w/ feature_value=Decimal("60000") ‚Üí resolver returns ok; condition evaluates True ‚Üí `applied_weight=2.5`, `total_score=2.5`. `2.5 >= 2.0` ‚Üí `decision="execute"`.
- **Single-rule reject**: same but feature_value=Decimal("40000") ‚Üí condition False ‚Üí `applied_weight=0.0`, `total_score=0.0`. `0.0 < 2.0` ‚Üí `decision="reject"`, `reason="score_below_threshold"`.
- **Two-rule sum**: `[r1(weight=1.5)‚ÜíTrue, r2(weight=2.0)‚ÜíTrue]`, threshold=3.0 ‚Üí `total=3.5 >= 3.0` ‚Üí `execute`.
- **Two-rule with negative**: `[r1(weight=2.0)‚ÜíTrue, r2(weight=-0.5)‚ÜíTrue]` ‚Üí total=1.5.
- **Required feature missing**: `Rule(required=True, ...)` + resolver returns `data_missing` ‚Üí immediate `decision="reject"`, `reason="required_feature_missing"`. `rule_results` list has only the missing-rule entry.
- **Optional feature missing**: `Rule(required=False)` + resolver `data_missing` ‚Üí `RuleResult(result="data_missing")`, `applied_weight=0`; loop continues to next rule.
- **Stale feature with required=True**: resolver `data_stale` + required ‚Üí `decision="reject"`, `reason="required_feature_stale"`.
- **on_error=reject + condition raises**: `decision="reject"`, `reason="rule_error"`, partial `rule_results`.
- **on_error=skip + condition raises**: `RuleResult(result="error_skipped", error={"error": "..."})`, `applied_weight=0`; loop continues.
- **Mode=passthrough + score below**: `mode="passthrough"`, total=0.0, threshold=2.0 ‚Üí `decision="passthrough"` (NOT "reject" ‚Äî passthrough overrides post-loop check).
- **Mode=passthrough + score above**: `mode="passthrough"`, total=5.0, threshold=2.0 ‚Üí `decision="passthrough"` (NOT "execute" ‚Äî passthrough overrides).
- **Empty rules**: `rules=[]`, `mode="active"` ‚Üí total=0.0, decision="reject" reason="score_below_threshold".
- **Empty rules + passthrough**: decision="passthrough".
- **`feature_snapshot` serialization**: feature_value=Decimal("50000.5") ‚Üí snapshot value `{"value_num": "50000.5", "value_bool": null, "value_json": null}`.

## Hazards

Every fail-open branch emits `bound_logger.warning(event="scoring_failed_open", ...)` to `system.log` stream (¬ßN2). Alerting-svc (F4 task) tails `system.log` JSON stream. Test pin `test_rule_error_emits_logged_event_not_silent_pass` per ¬ß20:2749 verbatim test-name.

Fail-open branches that emit `scoring_failed_open`:
1. `on_error="skip"` after `condition.evaluate(ctx)` raises Exception.
2. `data_missing` + `rule.required=False` (resolver returns missing; rule continues silently in fail-open mode).
3. `data_stale` + `rule.required=False` (resolver returns stale; rule continues silently in fail-open mode).

Counter-fail-open branches that do NOT emit (clean reject path; `data_*` events naturally surface via ScoringResult.reason):
- `data_missing` / `data_stale` + `rule.required=True` ‚Üí reject path; logged at `info` level via `scoring_required_feature_missing` / `scoring_required_feature_stale` event labels.
- `on_error="reject"` after Exception ‚Üí reject path; logged at `info` level via `scoring_rule_error_reject` event label.

Forward-pin: ¬ß10.4 brief reference to `FeatureMissingError` exception class is V1-DEFERRED. T-302/T-306 surfaced status/error_info via `(False, error_info)` tuple + `ResolverResult.status` instead of raising. T-307 adapts brief verbatim pseudocode to the existing post-T-302 contract.
