## Purpose

Land the YAML config loader for `configs/bots/<bot_id>.yaml` per BRIEF §9.4:1530 + §B.1:2909-2999 verbatim alpha.yaml example. Parses operator-authored YAML, narrows `condition: dict[str, Any]` placeholder (T-300) to a discriminated `ConditionUnion` over the 14-variant §10.2 catalog (T-302 simple 8 + T-303 series 3 + T-304 composite 4 + T-305 plugin 1), and produces a `BotConfig` instance ready for T-309 strategy-engine startup + T-310 per-bot signal consumer + T-307 evaluator.

T-308 unifies the disparate condition variants from T-302..T-305 into a single discriminated Pydantic union for YAML parse-time validation. Recursive descent through composite sub-conditions; plugin conditions wired through `plugin_registry: dict[(name, version), type[Rule]]` (T-305 loader output).

def load_bot_config(
    path: Path,
    *,
    plugin_registry: Mapping[tuple[str, str], type[Rule]] | None = None,
) -> BotConfig:
    """Parse + validate `configs/bots/<bot_id>.yaml` per BRIEF §B.1 example.

    1. yaml.safe_load(path)
    2. Build BotConfig top-level Pydantic schema (T-300 types).
    3. Per-rule: narrow rule.condition dict → Condition instance via _parse_condition().
    4. Plugin conditions: lookup (name, version) in plugin_registry → instantiate Rule.
    5. Recursive descent through composite (and/or/not/when_then_else) sub-conditions.

    Raises:
    * FileNotFoundError if path missing.
    * pydantic.ValidationError on malformed structure.
    * ValueError on unknown condition type, missing plugin, or §B.1 deviation
      that doesn't map cleanly to T-300 schema.
    """


def parse_condition(
    spec: dict[str, Any],
    *,
    plugin_registry: Mapping[tuple[str, str], type[Rule]] | None = None,
) -> Condition:
    """Recursive condition narrowing per §10.2 14-variant discriminated union.

    Public for test-pin reasons (caller can build Condition trees from YAML
    snippets without full BotConfig load). T-310 strategy-engine consumer
    uses load_bot_config; this helper is for unit tests + hot-reload paths.
    """

## Validation rules

1. **§B.1 verbatim YAML structure**: top-level keys `bot_id`, `display_name`, `created_at`, `status`, `exchange`, `signals`, `trading`, `execution`, `scoring`. T-308 parses a SUBSET — only `bot_id`, `version` (default 1 if absent), `trading.universe → BotConfig.symbols`, `scoring.{mode, trigger_threshold, rules}`. Other §B.1 sections (exchange/signals/execution) are operator-config consumed by separate services (T-309 strategy-engine reads what it needs); T-308 ignores extras via Pydantic `model_config = ConfigDict(extra="ignore")`.
2. **§B.1 vs T-300 `feature` location**: §B.1 example puts `feature: ind...` INSIDE `condition:` block; T-300 ScoringRule has `feature: str` at rule level. T-308 YAML loader extracts `feature` from condition dict at parse time + places on rule. Multi-feature conditions (ema_stack with `short`/`mid`/`long`, composite with sub-conditions) take the FIRST encountered `feature` from recursive descent of condition tree. Plugin condition without `feature` field → `ValueError("plugin rule {name}: feature required for resolver path; specify feature explicitly")`. Per pass-1 BLOCKER#2 fix Path A (operator-approved 2026-05-02): NO sentinel string; first-feature-encountered convention is uniform across all condition shapes.
3. **`condition: dict` narrowing**: `_parse_condition(spec, registry)` dispatches on `spec["type"]` to one of 14 condition classes. Unknown type → `ValueError(f"unknown condition type: {spec['type']!r}")`.
4. **Plugin condition wiring**: `spec["type"] == "plugin"` requires `plugin_registry` non-None. Lookup `(spec["name"], spec["version"])`; if missing → `ValueError(f"plugin not in registry: {name!r}/{version!r}")`. Instantiate `rule_class(spec["params"])` per T-305 contract; wrap in `PluginCondition(name=..., version=..., rule=instance)`.
5. **Composite sub-condition recursion**: `and`/`or` have `conditions: list[dict]` → recursive `_parse_condition` per element. `not` has `condition: dict` → single recursive parse. `when_then_else` has `when`/`then_`/`else_` keys (or `then`/`else` Python keywords escaped at YAML level — operator writes `then`/`else`; loader maps to `then_`/`else_`).
6. **§B.1 ema_stack mapping**: `{type: ema_stack, short, mid, long, direction}` → `EmaStackCondition(features=[short, mid, long], direction=...)`. Per pass-1 BLOCKER#1 fix Path A (operator-approved 2026-05-02): `direction` MUST be one of `"up"` | `"down"` (T-303 schema verbatim); `direction: from_signal` is **REJECTED at parse time** with `ValueError("direction='from_signal' not supported in v1; specify 'up' or 'down' explicitly. To get LONG/SHORT-aware behavior, duplicate the rule with applies_when {signal.action: LONG} + direction: up + paired SHORT version.")`. Operator updates §B.1 example to use explicit direction. SHORT-side support via `from_signal` is deferred to a follow-up task post-applies_when narrowing.
7. **`applies_when` v1 passthrough**: §B.1 has `applies_when: { signal.action: LONG }` shape. T-300 typed `applies_when: dict[str, Any] | None`. T-308 keeps as raw dict (NO narrowing); T-307 v1 ignores per OQ-1 of T-307. Documented v1 gap; T-308 narrowing follow-up planned post-T-313.
8. **`scoring.mode`**: `passthrough | active`; default `active` if absent.
9. **`scoring.trigger_threshold`**: required float.
10. **`bot_config.version`**: default `1` if absent in YAML (per T-300 schema).
11. **Operator typo at YAML key level**: Pydantic `ValidationError` surfaces verbatim (no custom wrapping).

## Edge cases

- **YAML file missing**: `FileNotFoundError`. T-309 fail-fast.
- **YAML malformed**: yaml.safe_load raises; T-308 propagates.
- **Empty `rules` list**: `ScoringConfig(rules=[])` valid per T-300 (test pin).
- **Plugin condition in YAML but no `plugin_registry` passed**: `ValueError("plugin condition requires plugin_registry")`.
- **Plugin `(name, version)` not in registry**: `ValueError`.
- **Unknown condition `type`**: `ValueError`.
- **Composite `not` with malformed sub**: nested `ValueError` propagated up.
- **`when_then_else` with `then`/`else` Python-keyword YAML keys**: loader normalizes `then` → `then_`, `else` → `else_` (operator writes natural YAML).
- **§B.1 `direction: from_signal` for ema_stack**: REJECTED at parse-time with explicit ValueError per Path A. Operator updates YAML to use `up` or `down` explicitly.
- **Single-feature condition without `feature` field at rule or condition level**: `ValueError("rule {name}: feature reference required")`.
- **Multi-feature condition (ema_stack) WITHOUT `short`/`mid`/`long`**: Pydantic ValidationError (EmaStackCondition.features Field(min_length=3, max_length=3)).
- **Composite condition (and/or/not/when_then_else) at rule level**: rule.feature uses first feature_ref encountered in the tree (recursive descent through sub-conditions). If no feature found anywhere → `ValueError("rule {name}: feature required at composite root or in at least one sub-condition")`. Per Path A — no `_multi_feature` sentinel.
- **Plugin condition without explicit `feature`**: `ValueError("plugin rule {name}: feature required for resolver path; specify feature explicitly")`.
- **Pydantic discriminated-union narrowing fails**: ValidationError surfaces with discriminator info.

## Hand verification

T-308 has **no math content** — pure schema parsing + Pydantic narrowing. Math-validator gate is in scope (`packages/scoring/`) but content is non-numeric; expected VERIFIED out of scope.

Hand-fixtures (per condition variant):

- **§B.1 `between` rule**: `{name: rsi_aligned_long, weight: 1.0, applies_when: {signal.action: LONG}, condition: {type: between, feature: ind.${signal.symbol}.15m.rsi_14, min: 55, max: 70}}` → `ScoringRule(name="rsi_aligned_long", weight=1.0, feature="ind.${signal.symbol}.15m.rsi_14", applies_when={"signal.action": "LONG"}, condition=BetweenCondition(min=55, max=70), ...)`.
- **§B.1 `ema_stack` rule (with explicit direction per Path A)**: `{condition: {type: ema_stack, short: A, mid: B, long: C, direction: up}}` → `condition=EmaStackCondition(features=["A", "B", "C"], direction="up")`. `rule.feature = "A"` (first encountered).
- **`from_signal` direction rejection**: `{condition: {type: ema_stack, short: A, mid: B, long: C, direction: from_signal}}` → `ValueError("direction='from_signal' not supported in v1; ...")` at parse time.
- **`plugin` rule with explicit feature**: `{condition: {type: plugin, name: oi_squeeze, version: 2, params: {...}, feature: ind.btcusdt.15m.oi}}` + registry has `("oi_squeeze", "2")` → `condition=PluginCondition(name="oi_squeeze", version="2", rule=OISqueezeRule({"lookback_candles": 5}))`. `rule.feature = "ind.btcusdt.15m.oi"`.
- **`plugin` rule without feature (rejected)**: `{condition: {type: plugin, ...}}` (no feature anywhere in rule or condition) → `ValueError("plugin rule {name}: feature required for resolver path; specify feature explicitly")`.
- **Composite `and`**: `{condition: {type: and, conditions: [{type: gt, feature: X, value: 50000}, {type: lt, feature: X, value: 60000}]}}` → `AndCondition(conditions=[GtCondition(value=50000), LtCondition(value=60000)])`. `rule.feature = "X"`.
- **Composite `not`**: `{condition: {type: not, condition: {type: equals, feature: Y, value: 5}}}` → `NotCondition(condition=EqualsCondition(value=5))`. `rule.feature = "Y"`.
- **`when_then_else`**: `{condition: {type: when_then_else, when: {type: gt, feature: X, value: 1}, then: {type: lt, feature: X, value: 10}, else: {type: lt, feature: X, value: 5}}}` → `WhenThenElseCondition(when=Gt(1), then_=Lt(10), else_=Lt(5))`. `then`/`else` YAML keys normalized to Python attrs.
- **Empty rules list**: `{scoring: {mode: active, trigger_threshold: 1.0, rules: []}}` → valid `BotConfig`.
- **Bot version default**: YAML omits `version` → `BotConfig.version == 1`.
