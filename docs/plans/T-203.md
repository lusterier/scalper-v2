**Composite PK `(bot_id, symbol)`** — one row per active bot-symbol pair. Naturally enforces "one open position per bot per symbol" at the schema level. Updates key off this PK (H-018 enforcement principle: WHERE `bot_id = $1 AND symbol = $2`, never `WHERE status = 'open'`).

**FK `trade_id → trades.id`** — plain `sa.ForeignKey("trades.id")` per §7.2 verbatim. Postgres default (NO ACTION) on UPDATE/DELETE: cannot delete the parent `trades` row while a `position_state` row references it. Practically irrelevant: `trades` are forever per §18.3; never deleted.

**No `status` column.** Brief design: `position_state` only contains rows for **currently-open** positions. When a position closes, the row is DELETEd (or left to be reconciled-cleanup on startup per H-020). The "state" is implicit in row presence + `remaining_qty`. (Closed-trade history lives in `trades` + `executions` ledger.)

**No discrete FSM-state column.** Brief uses **flag-based** state representation:
- `tp_hit BOOLEAN` — partial-TP filled?
- `trailing_active BOOLEAN` — trail SL engaged?
- `sl_type TEXT` — current SL kind ('protective' / 'be' / 'trail')
- `remaining_qty NUMERIC` — tracks position-size shrink across partial-TP

The (`tp_hit`, `trailing_active`, `sl_type`, `remaining_qty`) tuple uniquely identifies any FSM state T-217 needs to represent. No enum needed; T-217 derives state from flags. Surfaced as OQ-1 below for confirmation.

**No indexes beyond PK.** Brief specifies zero secondary indexes. Composite PK doubles as natural lookup index for `(bot_id, symbol)` queries; T-221 reconciliation full-table-scans (small table, one row per active bot-symbol pair); T-218 dispatcher lookup-by-trade_id is conceivable but brief is silent — F1+ opportunistic if surfaces. §0.8 anti-hypothetical.

**Server defaults:**
- `tp_hit BOOLEAN NOT NULL DEFAULT FALSE` → `server_default=sa.text("false")`
- `trailing_active BOOLEAN NOT NULL DEFAULT FALSE` → `server_default=sa.text("false")`
- `running_pnl NUMERIC(20, 4) NOT NULL DEFAULT 0` → `server_default=sa.text("0")`
- `updated_at TIMESTAMPTZ NOT NULL` — **no server_default** per §N1 (no `CURRENT_TIMESTAMP` / `NOW()`); application sets via `now_utc()`.

## Edge cases

- **Composite PK enforces "one open position per bot-symbol".** A second `INSERT` with the same `(bot_id, symbol)` raises a UniqueViolationError. T-216b therefore commits in a single transaction with `INSERT … ON CONFLICT DO NOTHING` *or* simply assumes the unique invariant; brief is silent. Plan does not pre-decide; T-216b plan-doc owns the decision. Schema just provides the PK.
- **No `updated_at` server_default** per §N1. Application-side responsibility: every UPDATE call writes `updated_at = now_utc()`. Brief precedent matches T-202 / T-001 (no `CURRENT_TIMESTAMP` in any migration).
- **FK NO ACTION default.** Brief writes plain `REFERENCES trades(id)` (no `ON DELETE CASCADE` / `ON DELETE SET NULL` / etc.). Postgres defaults to NO ACTION on UPDATE + DELETE — cannot delete the parent `trades` row while position_state row references it. Practically harmless: §18.3 retention says trades kept forever; never deleted.
- **`position_state` row LIFETIME**: row exists ONLY while position is open. Closed-trade history lives in `trades` + `executions`. T-219 close flow DELETEs the row after final close-execution + cumulative-delta apportionment.
- **No `bot_id → bots(bot_id)` FK** per §7.2 verbatim. Brief writes plain `bot_id TEXT NOT NULL`. Mirrors T-202 W#2 precedent (`executions.bot_id` is also no-FK). T-216b inserts via the same code path that already validates bot_id presence (composition root T-215 ensures every bot in adapter pool has a bots row). Anti-FK assertion in test catches accidental symmetry-add.
- **`mfe_price` / `mae_price` are NULLABLE** per §7.2. Application semantics: NULL on first row insert (no excursion observed yet); first monitor tick fills both via `now_utc()` mid-tick with current price. Brief accepts NULL initial; no DEFAULT sets a starting value (would require `entry_price` reference at INSERT time).
