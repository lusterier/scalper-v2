## Purpose

Per-bot max-drawdown hard-stop. Tracks the bot's lifetime cumulative realized P&L; when it gives back ≥ `max_drawdown_pct` of its all-time profit peak, the kill-switch latches with `trip_reason='max_drawdown'` and stays latched (hard-stop — NOT auto-cleared at UTC midnight, unlike the daily-loss latch). Pre-scoring gate in `consumer.py` AFTER the T-525a2 loss-limit gate (chain cooldown → caps → loss-limit → drawdown), silent-skip pattern. Protects against sustained equity erosion that a per-day loss limit (T-525a2) does not catch.

### NEW `packages/db/queries/trades.py::select_pnl_peak_and_current` (extend T-526/T-524/T-525a2 module)

```python
async def select_pnl_peak_and_current(
    conn: _DbExecutor, *, table_name: TradeTableName, bot_id: str,
) -> tuple[Decimal, Decimal]:
    """(peak, current) of the bot's LIFETIME cumulative realized P&L.

    current = Σ realized_pnl over ALL closed trades (the final running cumulative).
    peak    = MAX of the running prefix-sum ordered by (closed_at, id).
    Both COALESCE→Decimal('0') when the bot has zero qualifying closed trades.
    Charter (mirror sum_realized_pnl_since): status='closed' AND realized_pnl
    IS NOT NULL. NO float() (§5.13/§N1)."""
```

SQL:

```sql
SELECT
  COALESCE(MAX(running), 0)               AS peak,
  COALESCE((SELECT SUM(realized_pnl) FROM <table>
            WHERE bot_id = $1 AND status = 'closed'
              AND realized_pnl IS NOT NULL), 0) AS current
FROM (
  SELECT SUM(realized_pnl) OVER (ORDER BY closed_at, id
                                 ROWS UNBOUNDED PRECEDING) AS running
  FROM <table>
  WHERE bot_id = $1 AND status = 'closed' AND realized_pnl IS NOT NULL
) s
```

**L-021 audit**: the only parameter is `$1` (`bot_id`) — column-direct TEXT equality (`WHERE bot_id = $1`, appears twice, both column-direct) → L-021-safe, no `::text` cast. **No timestamp predicate** (lifetime — no `closed_at >= $N`) → no `::timestamptz` cast site (mirror `count_open_trades` L-021 note; distinct from `sum_realized_pnl_since` which DID need the cast). `table_name` is the `TradeTableName` Literal (f-string injection-safe). `realized_pnl` NUMERIC(20,4) → asyncpg `Decimal`; window `SUM(...) OVER (...)` + outer `MAX`/`SUM` stay NUMERIC → `Decimal`. `ROWS UNBOUNDED PRECEDING` makes the running prefix-sum explicit (deterministic; `(closed_at, id)` tiebreak mirrors `select_recent_closed_trades` ORDER BY convention).

### NEW `services/strategy_engine/app/drawdown_gate.py`

```python
@dataclass(frozen=True, slots=True)
class DrawdownDecision:
    blocked: bool
    reason: str | None              # 'max_drawdown' fresh trip; state.trip_reason if
                                     # pre-existing latch (reason-agnostic); None when not blocked
    drawdown_pct: Decimal | None    # the computed give-back ratio that tripped (audit)
    limit_pct: Decimal | None       # configured max_drawdown_pct

async def check_max_drawdown(
    *, pool, bot_id: BotId, exchange_mode: Literal["live","testnet","paper"],
    now: datetime, risk_config: RiskSection,
) -> DrawdownDecision
```

**Logic** (mirror `check_daily_loss_limit` structure; OQ-A/B/C):

1. `risk_config.max_drawdown_pct <= 0` → `_NOT_BLOCKED` **before any DB** (short-circuit; mirror siblings).
2. `async with pool.acquire() as conn:`
3. `state = await select_kill_switch_state(conn, bot_id=str(bot_id))`.
4. If `state is not None and state.tripped`:
   - `is_stale_daily_latch(state, now)` True → `await clear_kill_switch(conn, bot_id, updated_at=now)`; fall through to recompute. (Symmetric with `check_daily_loss_limit`; for a `max_drawdown` reason `is_stale` is always False — only a stale *daily* latch reaches here, and the T-525a2 loss-limit gate which runs BEFORE drawdown already stale-clears it this signal; the fresh `select` here normally already sees it cleared. Keeping the branch symmetric is safe + single-sourced.)
   - Else (non-stale tripped) → **blocked**, reason-agnostic: `reason=state.trip_reason`, `drawdown_pct=None`, `limit_pct=risk_config.max_drawdown_pct` (recompute SKIPPED — sticky; mirror T-525a2 WG#4).
5. `(peak, current) = await select_pnl_peak_and_current(conn, table_name=<dispatch>, bot_id=str(bot_id))`.
6. **`if peak <= 0: return _NOT_BLOCKED`** (OQ-A peak>0 guard — you cannot give back profit you never had; the pure-loss case is T-525a2's daily-loss territory; this also makes the division safe — no `/0`, no `/negative`).
7. `drawdown_pct = (peak - current) / peak` (Decimal − Decimal, then Decimal / Decimal — exact-context Decimal division; NO float).
8. `if drawdown_pct >= risk_config.max_drawdown_pct:` → `await upsert_kill_switch_trip(conn, bot_id=str(bot_id), trip_reason="max_drawdown", tripped_at=now, daily_anchor_date=now.date(), cumulative_loss_usd=current)` → **blocked** (`reason="max_drawdown"`, `drawdown_pct=drawdown_pct`, `limit_pct=max_drawdown_pct`). `daily_anchor_date=now.date()` is written for schema/audit consistency but is **semantically inert for max_drawdown** (`is_stale_daily_latch` returns False for `trip_reason='max_drawdown'` regardless of anchor — hard-stop, shipped T-525a1). `cumulative_loss_usd=current` reuses the audit column to store the current cumulative P&L at trip (the give-back context).
9. Else → `_NOT_BLOCKED`.

`exchange_mode` dispatch: `"trades" if exchange_mode in ("live","testnet") else "paper_trades"` (mirror siblings).

## Hand verification (§6.2 step 5 — math-validator Gate 4: Decimal cumulative + peak + give-back division + peak>0 guard)

Config `max_drawdown_pct = Decimal("0.20")` (20 % give-back of profit peak), bot `alpha`, paper.

Closed trades ordered by `(closed_at, id)`, with running prefix-sum:

| # | realized_pnl | running cumulative Σ |
|---|--------------|----------------------|
| 1 | +50.0000 | 50.0000 |
| 2 | +70.0000 | 120.0000 |
| 3 | -30.0000 | 90.0000 |
| 4 | -15.0000 | 75.0000 |

- `current` = Σ all = `Decimal("75.0000")` (the final running cumulative; also the plain `SUM`).
- `peak` = `MAX(running)` over {50, 120, 90, 75} = `Decimal("120.0000")`.
- peak > 0 → guard passes (compute).
- `drawdown_pct = (peak - current) / peak = (120.0000 - 75.0000) / 120.0000 = 45.0000 / 120.0000 = Decimal("0.375")`.
- `0.375 >= 0.20` → **TRIP**: `upsert_kill_switch_trip(reason='max_drawdown', cumulative_loss_usd=Decimal("75.0000"), tripped_at=now, daily_anchor_date=now.date())` → blocked. (`daily_anchor_date` written but inert — `is_stale_daily_latch('max_drawdown', ...)` = False always; hard-stop.)

**Boundary**: peak=100, current=80 → `(100-80)/100 = Decimal("0.20")`; `0.20 >= 0.20` → **True → trip** (reaching the limit exactly trips — mirror T-524 `>=` / T-526 `>=` / T-525a2 `<=` boundary rationale).

**peak≤0 guard (OQ-A)**: never-profitable bot, trades −20 / −30 / −10 → running {−20, −50, −60}; `current=Decimal("-60")`, `peak=MAX(−20,−50,−60)=Decimal("-20")`. `peak <= 0` → **return _NOT_BLOCKED** (no division — you cannot give back profit never earned; the −60 cumulative loss is T-525a2 daily-loss territory, not a drawdown). Also covers `peak == 0` (e.g. single break-even trade): `0 <= 0` → not blocked (no `/0`).

**current > peak impossible**: by construction `current` is one of the running values and `peak = MAX(running) ≥ current` → `(peak - current) ≥ 0` → `drawdown_pct ≥ 0` always (when peak>0). `drawdown_pct` can exceed `1.0` when `current < 0 < peak` (e.g. peak=100, current=−50 → 1.50) — legitimate; the `ge=0`-only field bound (no `le`) accommodates a threshold the operator could set anywhere.

**Exact Decimal division**: `(Decimal("3") - Decimal("1")) / Decimal("3")` = `Decimal("0.6666666666666666666666666667")` (Python default 28-digit Decimal context, ROUND_HALF_EVEN). The comparison `Decimal("0.6666666666666666666666666667") >= Decimal("0.66")` → True. No `float()` anywhere — subtraction exact, division uses Decimal context (deterministic, reproducible), comparison Decimal-vs-Decimal. (Unit test #9 pins this exact value.)
