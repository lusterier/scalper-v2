## Purpose

Ship Section 8 Audit log viewer per BRIEF §14.3:2067 — chronological `audit_events` table with filters (actor / event_type / correlation_id / time-range). Single TanStack Router file `routes/audit.tsx` per OQ-1=A (list-only with inline row-expand for drill-down; composite-PK URL deferred — small 10-field row + same-page expand simpler than `/audit/$eventId?occurred_at=` route).

**Per OQ-2=A — filter set**: 4 filters (actor + action_prefix + entity_type + TimeRangePicker). Backend doesn't expose `correlation_id` as filter param (gap vs BRIEF wording — backend has entity_type / entity_id / actor / action_prefix / from-to). T-419 OMITS correlation_id filter input field; F4+ separate task adds backend support. **Per OQ-3=A — URL search-param consumer**: T-411 `CorrelationIdChip` navigates to `/audit?correlation_id=<id>`. T-419 reads `?correlation_id=` URL search param + applies client-side filter on loaded events (best-effort; only filters current 50-row page). Inline notice rendered: "Filtering by correlation_id=X (client-side; only current page)".

**Per OQ-4=A — before_state / after_state rendering**: row-expand renders `<pre>JSON.stringify(state, null, 2)</pre>` for raw JSONB inspection. Flexible-shape dicts (e.g., bot_config row dump) — pretty-print is operator-friendly without imposing schema assumptions.

T-419 reuses T-411 DataTable + T-414 custom pagination + format-time helpers + CorrelationIdChip from T-411. New: row-expand inline drill-down (NOT shadcn Dialog — same-page UX preferred for high-frequency audit row inspection).

**Explicitly out of scope for T-419**: backend `correlation_id` filter (F4+ separate task `T-F4+ /api/audit/?correlation_id=`); separate detail route `/audit/$eventId` (per OQ-1=A — inline expand sufficient); CSV export (F5+); deep-linkable expand state in URL (F5+); multi-row expand (T-419 single-row only); diff-style before/after rendering (F5+ — line-diff highlighting); audit event search by full-text (F5+); per-user audit retention policy admin (F5+); error boundary (T-420+); CEST/UTC toggle (F5+); mobile responsive; Storybook for AuditExpandRow.

### Behavioral contract — query layer

**Audit list** (`/audit`):
- `useQuery(["audit-list", filters, offset], () => apiFetch<AuditEventListResponse>(buildAuditUrl(...)))` — operator filter-driven; no auto-refetch.

### Behavioral contract — URL search-param consumer (per OQ-3=A)

`Route.useSearch()` returns `{correlation_id?: string}`. When non-empty:
- Apply client-side filter: `events.filter(e => e.correlation_id === correlationFilter)` AFTER backend fetch.
- Render notice above DataTable: `Filtering by correlation_id=<value> (client-side; only current page)` with "Clear" button → `navigate({to: "/audit", search: {}})`.
- Backend fetch URL UNCHANGED — no `?correlation_id=` query param sent (backend doesn't accept it; would 422).

### Behavioral contract — row-expand drill-down (per OQ-1=A + OQ-4=A)

{expandedId !== null && (

### Behavioral contract — `before_state` / `after_state` rendering

```tsx
function StateBlock({ label, state }: { label: string; state: Record<string, unknown> | null }): React.JSX.Element {
  if (state === null) {
    return (
      <details>
        <summary>{label}</summary>
        <div className="text-sm text-muted-foreground">
          (no {label} — {label === "before_state" ? "first version" : "entity removed"})
        </div>
      </details>
    );
  }
  return (
    <details open>
      <summary>{label} ({Object.keys(state).length} keys)</summary>
      <pre className="max-h-64 overflow-auto bg-muted/30 p-2 font-mono text-xs">
        {JSON.stringify(state, null, 2)}
      </pre>
    </details>
  );
}
```

## Write-time guidance

(plan-reviewer Gate 1 APPROVE 2026-05-05; 7 WG items — must be addressed during implementation; brief-reviewer Gate 3 verifies each verbatim.)

1. **STALE_MS na liste — zladiť s precedensom**. Existujúce list routy (`scoring.index.tsx`, `trades.index.tsx`) `STALE_MS`/`staleTime` NEpoužívajú (filter-driven query, refetch len pri zmene filtra). Plán uvádza `STALE_MS=30_000` v §Hazards/L-001 echo. Buď konštantu vynechať (mirror scoring.index pattern) ALEBO ju skutočne aplikovať na `useQuery({staleTime: STALE_MS})` aby nešla "navyše/dead". Drift-checker by mal flagnúť unused const.

2. **Korekcia justifikácie v komentári pri `buildAuditUrl`**. Plán v §"behavior contract — URL search-param consumer" tvrdí *"backend doesn't accept it; would 422"*. Reálne FastAPI/Starlette **ignoruje** neznáme query params (žiadny `extra="forbid"` v `analytics_api`). Skutočný dôvod prečo `?correlation_id=` neposielať na backend: backend filter neexistuje, fetch by sa zbytočne re-cachoval per non-deterministic URL. Komentár v kóde nech znie *"backend lacks correlation_id filter; param would be silently ignored — keep fetch URL deterministic"*. Test #7 ostáva v platnosti formulovaný correctly.

3. **Dotyk routeTree po pridaní `/audit`**. T-411 `CorrelationIdChip.tsx:91` má `to: "/audit" as never` cast (komentár: *"T-419 will retire this `as never` cast"*). T-419 musí: (a) `pnpm dev` regeneruje `routeTree.gen.ts`, (b) odstrániť `as never` casty v `NavigatingChip` (`to:` aj `search:`) raz `/audit` route je súčasťou typed tree. Bez tejto retire sa porušuje T-411 explicit promise; brief-reviewer to overí.

4. **AuditEvent shape mirror — exactly 10 polí**. `services/analytics_api/app/models/audit.py:AuditEventResponse` má polia v poradí: `id, occurred_at, actor, action, entity_type, entity_id, before_state, after_state, correlation_id, meta`. Plán `api-types.ts` má rovnaké poradie + správne nullable (`before_state: dict|None`, `after_state: dict|None`, `correlation_id: str|None`, `meta: dict` non-null). DRY-up: ak existuje pomocný `Record<string, unknown>` typ alias v `api-types.ts`, použiť ho; ak nie, nezavádzať.

5. **`?correlation_id=` empty/whitespace edge case**. `validateSearch` má vrátiť `undefined` aj pre prázdny string (`""`) — inak `events.filter(e => e.correlation_id === "")` by filtroval na nikto. Plán v signature `correlation_id?: string` má, ale guard chýba: `typeof search.correlation_id === "string" && search.correlation_id.length > 0 ? search.correlation_id : undefined`.

6. **Test #6 vs Test #7 disambiguation**. Test #6 ("client-side filter applied") vyžaduje aby fixture `/api/audit/` MSW response obsahoval EVENTS s aspoň jedným matching `correlation_id` aj non-matching, a po filtrovaní DataTable ukáže iba matching. Test #7 ("backend fetch URL UNCHANGED") vyžaduje MSW handler verify že prijatý request `request.url` NEobsahuje `correlation_id=`. Tieto dva testy majú odlišné MSW signatúry — explicitne v T-419 testfile dokumentovať.

7. **Inline expand single-state — ak `events` array zmení (nová page), expand collapse**. Plán hovorí `expandedId: number | null`. Pri `setPage(p+1)` alebo zmene filter-u staré `expandedId` môže nezodpovedať žiadnemu rowu na novej page → `events.find(...)` vráti `undefined` → `event!` non-null assertion zlyhá runtime-e. Pridať reset: `useEffect(() => setExpandedId(null), [filters, page])` ALEBO explicit guard `events.find(...) ?? null` + null-render ak nezhoda.
