## Purpose

Eliminate the WS-fill-before-persist race in the execution dispatcher that terminates a bot's exchange execution stream and strands an untracked, SL-less position. The open-fill WS event for a freshly REST-placed order can arrive at `_process` before the `placement_persist` tx commits `orders`/`trades`/`position_state` ‚Üí `_derive_exec_type ‚Üí ("unknown",None,None)` ‚Üí `:192-200 raise RuntimeError("unattributable fill")` ‚Üí `run_dispatcher_for_bot :369 except` ‚Üí `dispatcher_stream_terminated` + re-raise ‚Üí per-bot dispatcher task dead until restart ‚Üí untracked SL-less position. Per ADR-0018.

## Design ‚Äî 3 distinct, separately-logged outcomes

Before the existing `async with self._pool.acquire() as conn, conn.transaction():` write-tx (`dispatcher.py:133`), a **bounded attribution-readiness retry**: each attempt briefly acquires a conn, probes `select_order_id_by_exchange_id(message.exchange_order_id)` + the position_state/trade attribution `_derive_exec_type` uses; attributable ‚Üí break; else release conn, `await asyncio.sleep(backoff)` (**conn-free; NEVER sleeping while holding a pooled conn / open tx**), retry up to `execution_dispatcher_unattributable_retry_attempts`. In-process within ONE `consume`/`_process` invocation ‚Üí H-009 dedup ring (keyed `exchange_exec_id`) sees exactly one processing; no redelivery / double-apply.

- **Outcome 1 ‚Äî attributable within window:** proceed into the EXISTING write-tx body UNCHANGED (exec_type re-derived in-tx as today; full read+insert+update atomicity preserved).
- **Outcome 2 ‚Äî gate window exhausted (genuine orphan / persistence never arrived):** `bound_logger.error("execution.dispatcher_unattributable_fill_skipped", ...)` + `insert_trading_event(event_type="dispatcher_unattributable_fill_skipped", ...)` audit row (¬ßN2 audit-grade; `trail_audit.py` emit-only precedent; **NO NATS event** ‚Äî ¬ß0.8, no consumer) + `return` (skip; **NO raise**) ‚Üí `run_dispatcher_for_bot` async-for CONTINUES ‚Üí per-bot stream **survives**.
- **Outcome 3 ‚Äî gate PASSED but in-tx `_derive_exec_type` STILL `("unknown",None,None)`:** the existing `:192-200` `else: raise RuntimeError("unattributable fill...")` is **KEPT as a defensive in-tx invariant ‚Äî NOT removed**, re-keyed to a DISTINCT log `execution.dispatcher_unattributable_fill_post_gate` ‚Üí `RuntimeError` ‚Üí tx rollback ‚Üí `dispatcher_stream_terminated`. Fail-stop is CORRECT here ‚Äî the pre-tx gate probe and the in-tx re-derive read different DB snapshots at different instants (non-atomic; a concurrent close‚Üíreopen / partial-visibility window can make a gate-passed event in-tx-unattributable); SAME class/behaviour as the `:161/:176/:190/:261` in-tx integrity halts T-559 explicitly leaves out of scope; recovery = restart + T-221, unchanged. Removing it would convert a gate-missed genuine-unattributable into a silent mis-attribution ‚Äî rejected. The distinct key separates Outcome-2 (skip+survive) from Outcome-3 (fail-stop integrity anomaly).
