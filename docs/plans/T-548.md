# T-548 — rate-limiter NATS KV key separator `:`→`.` (nats-py 2.14.0 InvalidKeyError)

**Hazard:** H-025 (Bybit rate-limit mitigation) — semantically UNCHANGED; key-string format only, no §20 regression, §N10 intact.

## Verified root cause (from installed .venv, not hypothesis)

`nats-py 2.14.0` — `nats/js/kv.py:38` `VALID_KEY_RE = re.compile(r"^[-/_=\.a-zA-Z0-9]+$")` (**module-level**, NOT `KeyValue.VALID_KEY_RE` — that attribute does not exist). Real gate `kv.py:40-43 _is_key_valid(key)`: False if `len==0` or `key[0]=="."` or `key[-1]=="."`, else `VALID_KEY_RE.match`. Enforced at `kv.py:~147` (get) / `~210` (put): `if validate_keys and not _is_key_valid(key): raise InvalidKeyError` (default `validate_keys=True`). Colon `:` ∉ regex class → the entire rate-limiter keyspace (`bybit:ip:pause`, `bybit:ip:global`, `f"bybit:{sub_account}:{endpoint_group}"`) is invalid → `InvalidKeyError` on the first `kv_get`/`kv_put` of a real live Bybit adapter (`acquire()`→`_wait_if_paused`). Latent: only a live adapter exercises this; ci-fast mocks the bus; real-NATS integration tests are skipped (no NATS_TEST_URL). `infra/nats/bootstrap.sh:85` creates the `rate_limits` bucket (10s TTL) — NOT the cause (client-side key validation).

## Fix: separator `:`→`.` (OQ1 default)

`.` ∈ VALID_KEY_RE; keys `bybit.ip.pause` / `bybit.ip.global` / `bybit.<sub>.<group>` start `b`, no leading/trailing `.` → pass `_is_key_valid`. Code does flat get/put only (verified: no `kv ls`/prefix/watch/range — only integer `range(_MAX_CAS_RETRIES)`), so `.` adds no NATS subject-hierarchy semantic change.

## Alternatives (§0.6 — recorded decision)

- **(A) CHOSEN — rename `:`→`.`** across rate-limiter sites + tests + spec footnotes. Keys correct-by-construction; client validation stays ON globally; idiomatic.
- **(B) REJECTED — `validate_keys=False` at `packages/bus/client.py:301` (kv.get) + `:399` (kv.put)** (~2 LOC, keeps colon convention). Rejected: that seam is the single shared bus KV path for ALL buckets (`rate_limits` + `config_runtime` + feature-registry-seen ADR-0012 + any future) — disabling validation there is a **platform-wide key-format validation regression** for a 2-LOC saving, and leaves keys invalid for the pinned nats-py client AND the `nats` CLI used by bootstrap.sh/ops/integration `_clear_kv_keys` (cross-tool fragility). (A) is the correct trade.
