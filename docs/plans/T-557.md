## Problem (code-confirmed + empirically observed in the T-556 live-diagnostic)

`services/execution/app/placement.py:403` `tp_size = compute_tp_size(quantized_qty, request.tp_qty_pct)` = raw `quantized_qty * tp_qty_pct` (`placement_persist.py:117-119`) with **no qty_step quantization and no min_order_qty validation** — unlike the entry qty, which IS validated via `quantize_qty(working_qty, instrument_info)` (`placement.py:276`, T-529/H-036). When `tp_size < min_order_qty`, Bybit rejects the Partial `set_trading_stop` (`placement.py:476-481`) with `OrderRejected("The number of contracts exceeds minimum limit allowed")`; the generic catch at `:482-488` logs `execution.tp_set_failed_continuing_with_sl_only` and continues → the position is left with SL but NO TP, while `persist_placement_tx` still persists the intended `tp_price`/`tp_size`. Observed live 2026-05-18 (demo bot: qty 0.001 × tp_qty_pct 0.5 = 0.0005 < min_order_qty 0.001 BTC). Affects step 7 (paper) and step 9 (live/demo/testnet).

## Operator decisions (baked — no open questions)

- OQ-1 = **C: pre-flight reject before entry** — validate the would-be tp_size BEFORE `place_market_order`; if unsatisfiable → log ERROR + return, NO position opened (strongest capital-safety: never open a position that cannot honor its configured TP).
- OQ-2 = **symmetric across modes** — naturally satisfied: the gate is PRE-flight (before the step-7 paper / step-8/9 live fork), one code path covering paper + live/demo/testnet uniformly.
- OQ-3 = **defer DB-persistence drift to triage finding #2** — T-557 does NOT touch `persist_placement_tx` / `position_state` / T-217 monitor (moot for the reject case: no position opens).

## NEW H-037 hazard text (BRIEF §20, inserted after the H-036 block, before §20→§21 boundary)

```
### H-037 — Partial-TP size MUST be quantized + min-lot validated pre-flight before place_market_order

**Context.** `services/execution/app/placement.py` computed the partial-TP order size as raw `compute_tp_size(quantized_qty, tp_qty_pct)` = `quantized_qty * tp_qty_pct` with NO qty_step quantization or min_order_qty validation — unlike the entry qty (H-036). When `tp_size < min_order_qty` (or not qty_step-aligned), Bybit rejects the Partial `set_trading_stop` ("number of contracts exceeds minimum limit allowed"); the generic catch logged `execution.tp_set_failed_continuing_with_sl_only` and continued → a real position left with SL but NO TP (capital risk; DB persisted the unplaceable tp_price/tp_size). Surfaced by the T-556 live-diagnostic 2026-05-18 (demo: qty 0.001 × tp_qty_pct 0.5 = 0.0005 < min_order_qty 0.001 BTC).

**Policy.** Every order-placement path MUST validate the would-be partial-TP size BEFORE `adapter.place_market_order`: `tp_size = quantize_qty(compute_tp_size(quantized_qty, tp_qty_pct), instrument_info)` (same `quantize_qty` qty_step floor + dual min_order_qty check as H-036). If `quantize_qty` raises `QtyValidationError` → log ERROR `execution.tp_size_below_min_order_qty` + increment the unsatisfiable-TP counter + return WITHOUT placing the order (no set_leverage, no place_market_order, no NATS publish, no rate-limit token) — no position is opened that cannot honor its configured partial TP. The qty_step-aligned validated tp_size is threaded to all downstream sites (paper/live set_trading_stop, persist_placement_tx). The generic post-place `execution.tp_set_failed_continuing_with_sl_only` catch is retained for OTHER (auth/network/transient) TP-set failures.

**Test.** `test_placement.py`: tp_size < min_order_qty → handler returns before set_leverage/place_market_order (both not called, no NATS); tp_size needs qty_step round-down → proceeds with the aligned tp_size to downstream set_trading_stop(Partial); tp_qty_pct=1.0 boundary → proceeds.

H-037 numbering note: F6 triage-finding-#1 (surfaced by the T-556 live-diagnostic 2026-05-18; operator-prioritized ahead of triage C as direct capital risk). Sibling-class of H-036 (same pre-flight instrument-filter-validation discipline) but a distinct failure mode (partial-TP size, not entry qty), distinct origin (T-556 live-diagnostic, NOT the 2026-05-08/09 operator audit), distinct policy (pre-flight reject of the whole signal vs entry-qty round-down). The H-030..H-036 "7 of 7 audit cluster CLOSED" attestation is byte-unchanged — H-037 is a separate post-MVP F6 hazard, NOT a reopening of that cluster.
```

## Hand verification (math-validator Gate-4 — ACTIVE: services/execution/ in trigger list + position-sizing arithmetic)

`tp_size = quantize_qty(quantized_qty * tp_qty_pct, info)`; `quantize_qty` (H-036, audited, Decimal-only §5.3): `if q < min: raise; rounded = (q // step) * step; if rounded < min: raise; return rounded`.

- **Reject (the live-diagnostic case)**: quantized_qty=`0.001`, tp_qty_pct=`0.5` → raw=`0.0005`; `0.0005 < 0.001` (min) → `QtyValidationError` → **pre-flight reject, no order**. ✓ matches the observed Bybit `OrderRejected`.
- **Pass + round-down**: quantized_qty=`0.003`, tp_qty_pct=`0.5` → raw=`0.0015`; `0.0015 ≥ 0.001`; `0.0015 // 0.001 = 1`, `1 * 0.001 = 0.001`; `0.001 ≥ 0.001` → tp_size=`0.001` (carried downstream). ✓
- **Pass aligned**: quantized_qty=`0.004`, tp_qty_pct=`0.5` → raw=`0.002`; `0.002 // 0.001 * 0.001 = 0.002` ≥ `0.001` → tp_size=`0.002`. ✓
- **Boundary tp_qty_pct=`1.0`**: quantized_qty=`0.001` → raw=`0.001` = min → passes (entry qty already ≥ min, validated at `:276`). ✓
