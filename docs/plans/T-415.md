## Purpose

Ship Section 4 Backtest lab per BRIEF §14.3:2063 — list runs + new-run form + status + results + comparison view. Two TanStack Router files:

1. **`ui/src/routes/backtests.index.tsx`** — list of backtest runs (DataTable from T-411) + filters (BotSelector single + status dropdown + TimeRangePicker tied to `started_at`) + custom pagination + collapsible "+ New backtest run" form card per OQ-2=A. Click row → drill-down at `/backtests/$runId`.
2. **`ui/src/routes/backtests.$runId.tsx`** — single-page detail card per OQ-4=A: header (id + name + StatusBadge + submit button) + run metadata (bot + dates + config_hash + started_at + finished_at) + config_yaml collapsible preview + summary JSON pretty-print or "F5+ worker pending" placeholder + notes. 404 handling via `error.message.includes("404")` substring detect (T-414 WG#2 echo).

**Per OQ-3=B — comparison view DEFERRED** to F5+ separate task. F4 backend writes only `status='queued'` rows (per T-407 backend docstring) + `summary` always null. Comparison view UX would render 2× null placeholders for any 2 runs — operator-facing message would be misleading. Operator-facing decision: defer comparison to F5+ task when worker populates summaries; T-415 shipped without comparison saves ~80 LOC + cosmetic placeholder UX.

T-415 reuses T-414 patterns: `<TimelineSection>` primitive (loading/error/placeholder), `formatUtcDateTime`, custom pagination block placement, 4-filter row with `?status=` + `?bot_id=` + `?from=`/`?to=`. F4-specific deviations: backtest filter `from`/`to` filter `started_at` (not `closed_at` — different semantics from T-414 trades), so no open-status branch needed.

**Explicitly out of scope for T-415**: Comparison view (per OQ-3=B — F5+ separate task); inline YAML linting / live Pydantic validation in form (T-416 strategy editor ships YAML editor with validation); per-run cancel / retry / restart admin actions (F5+ worker concern); per-run resource cost estimate (F5+); per-run progress bar / partial results (F5+ worker SSE); CSV export of run list (F5+); deep-linkable filter state in URL (F5+); bulk delete / archive (F5+); error boundary (T-416+); skeleton loaders unification; CEST/UTC toggle (F5+); mobile responsive (per §14.2:2056); Storybook for backtest detail card; `summary` JSONB schema-typed renderer (F4 always null; F5+ defines schema).

### Behavioral contract — query layer

- `useQuery(["backtests-list", filters, offset], () => apiFetch<BacktestRunListResponse>(buildBacktestsUrl(...)), { refetchInterval: undefined })` — operator filter-driven; F4 has no live status updates (worker is F5+; SSE binding TBD when worker ships).

## Write-time guidance

(plan-reviewer Gate 1 APPROVE 2026-05-05; 8 WG items — must be addressed during implementation; brief-reviewer Gate 3 verifies each verbatim.)

1. **§N3 active control mechanism — test 7 must use mock fetch spy + call-count assertion, NOT just `expect(button).toBeDisabled()`.** Pattern: render form, click Submit (mutation goes to isPending), click Submit second time, assert `mockFetch.mock.calls.filter(c => c[0].includes("/api/backtests/") && c[1]?.method === "POST").length === 1`. Passive disabled-attribute assertion is insufficient — operator's §N3 concern is whether double-submit results in second backend call, not whether the UI looks disabled. Test 7 description in plan §`Test strategy` line 228 should explicitly say "asserts mock POST called exactly once after two button clicks within isPending window".

2. **§N1 form POST body Z-suffix — test 5 (NewRunForm happy path) must explicitly assert that POST body JSON has `date_range_start` + `date_range_end` ending in "Z".** Mechanism: capture `mockFetch.mock.calls[0][1].body` (parsed JSON string); regex assert `/T\d{2}:\d{2}:\d{2}(\.\d{3})?Z$/.test(body.date_range_start)` and same for `date_range_end`. Plan §`Behavioral contract — datetime conversions` documents the conversion path (datetime-local naive → `new Date(input.value)` → `.toISOString()` Z-suffix), but plan test 5 only says "POST returns 202 + form resets + list refetches" — operator's §N1 active control needs the URL-side test 4 mechanism mirrored on POST body.

3. **`<input type="datetime-local">` browser-tz conversion edge — explicit comment in NewRunForm submit handler.** `new Date("2026-05-05T12:00")` interprets the naive string in browser-local tz; `.toISOString()` then converts to UTC. If operator's machine is CEST (UTC+2), input "12:00" becomes "10:00:00Z" in body — this is correct behavior but non-obvious. Add inline comment in submit handler referencing §N1 + the browser-local→UTC conversion path so future maintainers don't "fix" this by stripping the conversion.

4. **StatusBadge cva extension — preserve T-411 inline cva pattern (no centralized STATUS_COLORS map).** Add `backtest: ""` to `kind` variants block + 4-line `if (kind === "backtest")` branch in `toneFor` helper with mapping queued→yellow / running→blue / completed→green / failed→red. Update `BadgeKind` union to `"bot" | "signal" | "trade" | "backtest"`. NO new exported tone enum, NO STATUS_COLORS map, NO factory abstraction — premature for 4-kind footprint per T-411 OQ-4=A precedent already baked into the file.

5. **Summary placeholder string verbatim — pin the message text.** Plan §`Behavioral contract — result summary display` says `"F5+ worker pending — backtest summary will populate after compute"`. Use this string verbatim in implementation; test 11 should assert the exact string via `screen.getByText(/F5\+ worker pending/i)` so future drift in wording is caught. Mirrors T-414 precedent ("Coming F4+/F5+" pinned per WG#3).

6. **PAGE_SIZE / STALE_MS / WINDOW_30D_MS named module constants per L-001 + T-414 WG#5 echo.** Each with rationale comment ref backend `_DEFAULT_LIMIT=50` (PAGE_SIZE), useQuery global staleTime alignment (STALE_MS), TimeRangePicker default window per BRIEF §14.2 (WINDOW_30D_MS).

7. **`buildBacktestsUrl` filter status="all" omits `?status=` — assert via negative URL test.** Plan test 3 says "status filter appends ?status= (status='all' omits param)". Mechanism: when status="all", assert `mockFetch.mock.calls[0][0]` does NOT contain "status=" substring (negative regex `expect(url).not.toMatch(/[?&]status=/)`). Mirrors T-414 WG#4 negative URL assertion pattern. Without this, a regression where `status=all` is silently sent to backend (and backend may 422) goes undetected.

8. **Mid-write split contingency at ~380 src LOC threshold — invoke drift-checker if drill-down route still unwritten.** Plan §0.3 budget paragraph documents this; restate in implementation order — write `routes/backtests.index.tsx` first, then `lib/api-types.ts` extension + `StatusBadge` extension, then `routes/backtests.$runId.tsx` last; if at ~380 LOC the drill-down is unwritten OR the NewRunForm is unwritten, STOP and run drift-checker before continuing. Per L-007 mid-write split is operationally expensive.
