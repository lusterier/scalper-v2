## Purpose

Detect out-of-FSM (manual Bybit-UI / out-of-band) modification of a tracked position's stop-loss. The PositionLifecycle monitor (`run_position_monitor_for_trade`, T-217a/b) is the single per-trade owner of SL state: it sets SL via `adapter.set_trading_stop` (BE/trail) and records its intent in `position_state.sl_price` (`update_position_state_sl`). T-535 adds, at the start of each monitor tick, a read of the live exchange SL (`Position.sl_price`, T-534a) and compares it to the FSM's last-known DB value (`ps.sl_price`). A mismatch ⇒ an out-of-band change happened ⇒ emit a `sl_overwrite_detected` `trading_events` audit row + WARN log. The FSM then continues normally (OQ-2=A — its own next BE/trail action re-asserts the bot's SL; the bot remains authoritative). Completes the SL/TP-verification cluster's *modification* arm (T-534b2 watchdog owns *removal*/naked-position; T-535 owns *modification*/overwrite — clean disjoint split). Ships the formal **H-029 §20 catalog allocation** (slot reserved by ADR-0011; allocated at the implementing task per §0.8; mirror T-525a1 H-027 / T-534b2 H-028 mechanism).

   H-029 entry text:
   > ### H-029 — SL-overwrite detection must not false-positive on legitimate FSM trail/BE updates
   >
   > **Context.** The PositionLifecycle monitor (`run_position_monitor_for_trade`) is the single per-trade owner of stop-loss state: it moves the SL on BE-trigger / trail via `set_trading_stop` and records its intent in `position_state.sl_price`. An out-of-band SL change (operator edits the SL in the Bybit UI, an external tool, a partial `set_trading_stop` desync) silently diverges the live exchange SL from the bot's tracked intent — the operator believes the bot's risk plan is in force when it is not. T-535 detects this by comparing the live exchange `Position.sl_price` (T-534a) to `position_state.sl_price` at each monitor tick. The hazard (ADR-0011 anticipated): the bot's OWN legitimate trail/BE update also changes the exchange SL — a naive periodic compare would false-positive on every trail step and emit spurious `sl_overwrite_detected` rows, drowning the real signal and eroding trust in the alert.
   >
   > **Policy.** The compare runs at the START of the lifecycle monitor tick (after `select_position_state`, before the BE/trail block), inside the single per-trade `asyncio.Task`. Because that task is the ONLY in-FSM writer of the SL and its tick is strictly sequential (`set_trading_stop` then `update_position_state_sl` complete within one tick before the next tick's compare runs), a legitimate trail/BE update leaves `exchange_sl == position_state.sl_price` at the next tick-start → no false-positive (exact `Decimal` equality is sufficient and correct under this single-task steady-state model — no tolerance needed). Only an out-of-FSM change makes them differ. A failed/transient `get_positions` read is NEVER treated as evidence of an overwrite (skip the tick, no emit — same false-positive-on-uncertainty guard as H-028). SL *removal* (`exchange_sl is None`) and position-gone are explicitly OUT of scope (T-534b2 watchdog/H-028 owns naked-position; T-221 owns orphan reconcile) — T-535 owns only *modification*. Detection is emit-only + WARN log (`sl_overwrite_detected` `trading_events` row); non-destructive — the FSM stays authoritative and re-asserts its SL on its next BE/trail action (OQ-2=A). Known benign edge: if the task is cancelled exactly between `set_trading_stop` and `update_position_state_sl` (shutdown), a restart may emit one spurious audit row before the FSM re-syncs — accepted (audit-only, operator-visible, rare). Known limitation (OQ-3=A): if Bybit normalizes the SL to the symbol price-tick and echoes a value ≠ what was sent, exact equality would false-positive every tick — mitigation (tick-tolerant compare needing a `priceTick` source) deferred to an F5+ ticket, raised only if testnet/live surfaces it.
   >
   > **Test.** `test_sl_overwrite_detected_emits_trading_event` (exchange_sl ≠ ps.sl_price → one `insert_trading_event(event_type="sl_overwrite_detected")` with str-cast prices + WARN; FSM continues) + `test_sl_overwrite_no_false_positive_on_matching_sl` (exchange_sl == ps.sl_price, incl. post-legit-trail steady state → no emit) + `test_sl_overwrite_skipped_on_get_positions_error` (transient → no emit) + `test_sl_overwrite_skipped_when_exchange_sl_none` (removal/paper → defer, no emit).

## H-029 false-positive analysis (load-bearing — the hazard's core)

- `run_position_monitor_for_trade` is a **single per-trade `asyncio.Task`**. The SL is mutated ONLY by this task (BE/trail `else:` branch: `set_trading_stop` then `update_position_state_sl`, sequential, same task, no cancellation point between in the branch).
- Legitimate trail/BE at tick N: after the branch, exchange SL = `new_sl` AND `position_state.sl_price` = `new_sl`.
- T-535 check at tick N+1 start: reads exchange `get_positions().sl_price` (= `new_sl`) and `ps.sl_price` (= `new_sl` from DB) → **equal → no emit**. No false-positive. ✓
- Out-of-band change between tick N and N+1: exchange SL = X ≠ `new_sl` = `ps.sl_price` → **emit**. Correct true-positive. ✓
- Because the same single task performs both the FSM SL write and the next tick's compare, there is **no concurrency window** where the compare observes a half-applied FSM update — exact `Decimal` equality is correct and H-029-safe (no tolerance required).
- Transient `get_positions` failure → skip, no emit (uncertainty ≠ overwrite — H-028 sibling principle).
- Benign edge (documented): task cancelled exactly between `set_trading_stop` and `update_position_state_sl` (shutdown) → on restart one spurious audit row possible before re-sync. Audit-only, non-destructive (OQ-2=A), operator-visible, rare — accepted.
