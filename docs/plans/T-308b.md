## Purpose

Replace the `ScoringRule.model_construct(...)` bypass in `yaml_loader.py:_build_rule` (line 320) with a normal `ScoringRule(...)` constructor call so that **all** ScoringRule fields are Pydantic-validated at YAML parse time, not just `weight` (the one field guarded by a hand-rolled pre-check at line 317-319).

## Problem (from T-308 brief-reviewer pass-1 CONCERN#1)

Today `_build_rule` bypasses Pydantic validation:

```python
return ScoringRule.model_construct(  # ← skips ALL validation
    name=rule_yaml["name"],
    weight=float(weight_raw),  # weight pre-checked manually above
    feature=feature,
    applies_when=rule_yaml.get("applies_when"),
    condition=condition,  # Condition instance, not dict
    on_error=rule_yaml.get("on_error", "skip"),
    required=rule_yaml.get("required", False),
    max_staleness_sec=rule_yaml.get("max_staleness_sec"),
)
```

Reason for bypass: `ScoringRule.condition: dict[str, Any]` (T-300 placeholder typing). The yaml_loader assigns a Pydantic **Condition instance** (e.g. `EqualsCondition`, `AndCondition`) — Pydantic strict-mode rejects assigning a Condition where a dict is declared.

Resulting validation gaps in production:

| Field | Declared type | YAML typo example | Today's behaviour |
|---|---|---|---|
| `name` | `str` | `name: 42` (int) | Silently accepted as int; downstream stringification |
| `applies_when` | `dict \| None` | `applies_when: "all"` (str) | Silently accepted as str |
| `on_error` | `Literal["skip","reject"]` | `on_error: "blow_up"` | Silently accepted; T-307 evaluator falls through to "skip" branch |
| `required` | `bool` | `required: "yes"` (str) | Silently accepted as str (truthy in Python) |
| `max_staleness_sec` | `int \| None` | `max_staleness_sec: "60s"` | Silently accepted as str |

Operator typo in `configs/bots/<id>.yaml` does not surface at startup; either crashes at first signal in T-310, or worse, silently mis-evaluates rules.

## Scope

### In scope (T-308b)

1. **Re-type `ScoringRule.condition: dict[str, Any]` → `condition: Any`** in `packages/scoring/types.py`. The original T-300 docstring claimed T-302..T-305 would narrow this to a discriminated Pydantic union, but the architecture took Path C (RuleContext-based Protocol) — the placeholder is permanently a placeholder. `Any` is honest typing.

2. **Add `strict=True` to `ScoringRule.model_config`** so coercion-friendly fields (`required`, `max_staleness_sec`) reject string inputs instead of auto-coercing. Mirrors T-302/T-303 condition models which already use `strict=True` (per `simple.py:42` etc.).

3. **Replace `ScoringRule.model_construct(...)` → `ScoringRule(...)`** in `_build_rule`.

4. **Keep the manual weight pre-check at lines 316-319** as belt-and-suspenders. Pydantic strict-mode rejects bool→float coercion in 2.x default, but the manual check provides a clearer error message ("weight must be a number; got True") than Pydantic's generic "Input should be a valid number". Keeping the pre-check is L-008 active control (defensive validators).

5. **Update T-300 ScoringRule docstring** to remove the stale "T-302..T-305 narrow them to discriminated Pydantic unions" claim. Replace with: "T-302..T-305 ship Condition Protocol classes; T-308 yaml_loader populates `condition` with a Condition instance via `parse_condition`. The `Any` typing reflects this — runtime narrowing happens via `@runtime_checkable Condition` (T-302), not Pydantic discrimination."

### Out of scope

- Changing `ScoringConfig.rules: list[ScoringRule]` validation behaviour. ScoringConfig accepts already-built ScoringRule instances; no per-rule re-validation needed.
- Changing `BotConfig.bot_id` regex / `BotConfig.version` Field constraints — already validated.
- Adding a discriminator for `condition` (would require redesigning T-302..T-305 condition catalog as Pydantic discriminated union; out of scope; T-300 docstring placeholder claim was aspirational).
- Re-running plan-reviewer on T-300 / T-302..T-305 / T-307 (none of those tasks change; T-308b only changes types.py + yaml_loader.py).

## Public interface change

### Before (`packages/scoring/types.py:101-121`)

```python
class ScoringRule(BaseModel):
    """One rule within a bot's scoring config (§10.1).

    ``applies_when`` and ``condition`` are typed as ``dict[str, Any]``
    placeholders at T-300; T-302..T-305 narrow them to discriminated
    Pydantic unions over the §10.2 condition catalog ...
    """

    name: str
    weight: float
    feature: str
    applies_when: dict[str, Any] | None = None

    on_error: Literal["skip", "reject"] = "skip"

```

### After

```python
class ScoringRule(BaseModel):
    """One rule within a bot's scoring config (§10.1).

    ``applies_when`` is typed ``dict[str, Any] | None`` — raw YAML
    pass-through; T-307 evaluator v1 ignores it (see grep-anchor
    comment at evaluator.py:107).

    ``condition`` is typed ``Any`` — yaml_loader (T-308) populates with
    a Condition instance from `parse_condition`. Runtime narrowing via
    ``@runtime_checkable Condition`` Protocol (T-302); Pydantic does not
    narrow at construction. The ``Any`` typing is intentional:
    T-300/T-302..T-305 architecture took the Path C RuleContext route,
    not the discriminated-union route the original T-300 docstring
    aspired to.
    """

    model_config = ConfigDict(frozen=True, strict=True)

    name: str
    weight: float
    feature: str
    applies_when: dict[str, Any] | None = None
    condition: Any  # ← Path A1
    on_error: Literal["skip", "reject"] = "skip"
    required: bool = False
    max_staleness_sec: int | None = None
```

## Edge cases

- **Pydantic strict-mode + `int | None` field with `None` from YAML**: strict-mode allows None for `int | None` per Pydantic 2 default. Verified via test #6 happy-path.
- **`Literal["skip", "reject"]` in strict-mode**: rejects any other str. Pydantic Literal validator pre-dates strict.
- **`condition: Any` accepting `None`**: Pydantic accepts None for `Any` field. The yaml_loader contract requires `parse_condition` to return a non-None Condition (raises on unknown type). So None never reaches ScoringRule(...) in practice. No test for this — exercising would require breaking parse_condition contract.
- **`weight: float` strict-mode**: rejects int→float coercion. But existing test pin `test_load_bot_config_happy_paths_*` uses `weight: 1.0` (float literal in YAML). Verify YAML loader receives float from PyYAML for `weight: 1.0`.
- **PyYAML int vs float**: `weight: 1` (no decimal) → PyYAML parses int → ScoringRule(weight=1) under strict-mode rejects int. **Risk**: existing fixtures may use `weight: 1` and break. **Mitigation**: keep manual weight pre-check `float(weight_raw)` BEFORE ctor call — pre-check passes int through `float(1) == 1.0` and Pydantic receives a true float. The pre-check coerces before strict-validates, preserving fixture compatibility.

## Path A1 vs alternatives (decision: Path A1)

**Path A1 (chosen):** condition: Any + strict=True. Minimal change. Keeps weight pre-check.

**Path B (rejected):** Add `field_validator` accepting Condition runtime types. Requires `arbitrary_types_allowed=True`. Invasive — changes Pydantic semantics globally for ScoringRule. Cost > benefit.

**Path C (rejected, original backlog suggestion):** Hand-rolled per-field pre-check mirroring weight check. Expands the manual-validation surface to 6 fields. Maintenance burden — each new ScoringRule field needs a new manual check. Pydantic exists for this exact reason.
