# T-526 — Cooldown after loss + losing-streak cooldown

**Status:** Plan-stage (pre Gate 1 plan-reviewer)
**Phase:** F5 — Pre-live operational hardening cluster (per ADR-0011)
**Branch:** `feat/T-526-cooldown-gate`
**Mirror precedent:** T-310b consumer pre-scoring skip pattern (signal_expired / signal_outside_universe — silent skip + trading.log info).

## Purpose

Per-bot signal-acceptance gate that suppresses signal processing for a configurable duration after recent losses. Two YAML-configurable timers combined via `max()`:

1. **`cooldown_after_loss_minutes`** — fixed cooldown from the last loss. Every new loss extends the timer naturally because each subsequent loss has its own `last_loss_at`.
2. **`cooldown_after_streak_n_losses_minutes`** — kicks in when `streak_count >= cooldown_after_streak_n_losses` (consecutive losses; win resets counter to 0).

Effective `cooldown_until = max(loss_until, streak_until)`. Reject reason indicates the binding cooldown (`cooldown_after_loss` / `cooldown_after_streak` / `cooldown_after_loss_and_streak`).

State is **derived from the `trades` / `paper_trades` table on each signal arrival** — no new persistent state, no message-bus subscriptions, no restart reconciliation. Mirror BRIEF §9.4 pre-scoring skip pattern (signal_expired / signal_outside_universe).

:class:`ShadowConfig` precedent (net-new feature catches operator typos at YAML load via
unknown-key rejection). For `int Field(ge=0)` fields, type-coercion strictness is functionally

discipline to RiskSection is consistent. Test `test_risk_section_rejects_extra_keys` pins
the typo-catch behavior.

## Hand verification

### Cooldown semantic worked example

Config: `cooldown_after_loss_minutes=10`, `cooldown_after_streak_n_losses=3`, `cooldown_after_streak_n_losses_minutes=60`.

Trades (closed_at DESC):

| Index | realized_pnl | closed_at | Notes |
|-------|--------------|-----------|-------|
| 0     | -5.00        | 10:00:00  | newest |
| 1     | -3.00        | 09:50:00  | |
| 2     | -2.00        | 09:30:00  | |
| 3     | +10.00       | 09:00:00  | streak boundary |

**Streak count**: walk from index 0 while `realized_pnl < 0`; counter increments → index 0 (loss, count=1) → index 1 (loss, count=2) → index 2 (loss, count=3) → index 3 (win, **stop**). Final `streak_count=3`. `last_loss_at=10:00:00`.

**At now=10:05:00**:
- `loss_cooldown_until = 10:00 + 10min = 10:10:00`
- `streak_cooldown_until = 10:00 + 60min = 11:00:00` (because `streak_count=3 >= cfg.streak_n_losses=3`)
- `effective_cooldown_until = max(10:10, 11:00) = 11:00:00`
- Both binding → `reason='cooldown_after_loss_and_streak'`
- `CooldownDecision(active=True, reason='cooldown_after_loss_and_streak', cooldown_until=11:00, streak_count=3, last_loss_at=10:00)`

**At now=10:11:00**:
- `loss_cooldown_until = 10:10 < now → loss no longer binding`
- `streak_cooldown_until = 11:00 > now → still binding`
- `reason='cooldown_after_streak'` (single binding)
- `cooldown_until=11:00`
- `active=True`

**At now=11:01:00**:
- Both `cooldown_until < now` → all expired
- `active=False`

### LIMIT calculation

`limit = max(1 if cfg.cooldown_after_loss_minutes > 0 else 0, cfg.cooldown_after_streak_n_losses)`.

If both disabled (`= 0`) → early-return before SELECT (per WG#1).
If only loss enabled → LIMIT 1 (need last_loss_at only).
If only streak enabled with n=5 → LIMIT 5.
If both enabled with streak_n=10 → LIMIT 10 (covers loss check too).

### Zero P&L row semantic

Per OQ-2=A (`realized_pnl < 0` strict):
- `realized_pnl=-0.50` → counts as loss (counter += 1).
- `realized_pnl=0.00` → NOT a loss; streak counter resets to 0 at this row.
- `realized_pnl=+0.01` → NOT a loss; streak counter resets to 0 at this row.
- `realized_pnl=NULL` (open trade) → impossible here because SELECT filters `status='closed'`.
