## Goal

Add `backtest_trades` table per BRIEF §12.2:1969-1971 (*"All generated trades in `backtest_trades`, linked by run_id"*) so that T-507 CLI + T-509 worker have a persistence target and T-516 UI has a query surface. Schema mirrors live `trades` (BRIEF §7.2:983-1009) with backtest-specific adaptations: FK to `backtest_runs(id)` cascade-delete, no FK on `bot_id` (mirror backtest_runs convention — archived bots may still own backtest history), no FK on `open_order_id` / `close_order_id` (paper backtest doesn't write to live `orders` table).

Plus a minimal read helper in `packages/db/queries/analytics.py` (mirror `BacktestRunRow` location): `BacktestTradeRow` dataclass + `select_trades_by_run(conn, run_id, limit, offset)` for T-516 consumer.

When adding `insert_backtest_trade` (T-507) and `update_backtest_trade_close` (T-509), pass Python dicts directly to asyncpg for the `meta` JSONB binding — **NO `json.dumps` at the call site**; pre-convert any UUID/datetime/Decimal in the dict via a `_to_jsonable` helper. ## Schema design

### `backtest_trades` columns (22 columns)

| # | Column | Type | Nullable | Notes |
|---|--------|------|----------|-------|
| 1 | `id` | `BIGSERIAL` | NOT NULL | PK |
| 2 | `run_id` | `UUID` | NOT NULL | **FK** `REFERENCES backtest_runs(id) ON DELETE CASCADE` |
| 3 | `bot_id` | `TEXT` | NOT NULL | NO FK (mirror backtest_runs convention) |
| 4 | `signal_id` | `BIGINT` | NULL | replayed-signal trace; null if synthesized |
| 5 | `open_order_id` | `BIGINT` | NULL | NO FK (paper backtest doesn't write live `orders`) |
| 6 | `close_order_id` | `BIGINT` | NULL | NO FK (same reason) |
| 7 | `symbol` | `TEXT` | NOT NULL | |
| 8 | `side` | `TEXT` | NOT NULL | `'buy'` or `'sell'` |
| 9 | `entry_price` | `NUMERIC(30, 12)` | NOT NULL | mirror live trades (§N1 — Decimal as string per §5.3) |
| 10 | `exit_price` | `NUMERIC(30, 12)` | NULL | null while open |
| 11 | `qty` | `NUMERIC(30, 12)` | NOT NULL | |
| 12 | `notional_usd` | `NUMERIC(20, 4)` | NOT NULL | qty × entry_price; computed at INSERT (mirror live) |
| 13 | `realized_pnl` | `NUMERIC(20, 4)` | NULL | null while open |
| 14 | `fees_paid` | `NUMERIC(20, 4)` | NULL | null while open |
| 15 | `close_reason` | `TEXT` | NULL | `'tp'` / `'sl'` / `'be'` / `'trail'` / `'manual'` / `'emergency'` / `'reconcile'` (mirror live) |
| 16 | `opened_at` | `TIMESTAMPTZ` | NOT NULL | |
| 17 | `closed_at` | `TIMESTAMPTZ` | NULL | null while open |
| 18 | `status` | `TEXT` | NOT NULL | `'open'` / `'closed'` / `'error'` |
| 19 | `mfe_pct` | `DOUBLE PRECISION` | NULL | maximum favorable excursion |
| 20 | `mae_pct` | `DOUBLE PRECISION` | NULL | maximum adverse excursion |
| 21 | `confidence_score` | `DOUBLE PRECISION` | NULL | scoring engine output snapshot |
| 22 | `meta` | `JSONB` | NOT NULL DEFAULT `'{}'::jsonb` | flexible per-row attrs |

### Indexes (3 btree)

- `backtest_trades_run_id` — `(run_id)` — primary lookup; supports T-516 "all trades for run X".
- `backtest_trades_run_closed` — `(run_id, closed_at DESC) WHERE status = 'closed'` — partial index for chronological closed-trade queries (mirror live `trades_closed_at`).
- `backtest_trades_run_status` — `(run_id, status)` — filter open vs closed within run.

NOT a hypertable per OQ-2 default A.

### FK cascade behavior

`run_id UUID NOT NULL REFERENCES backtest_runs(id) ON DELETE CASCADE` per OQ-3 default A. Rationale: when operator deletes a backtest run (e.g., experimental misconfiguration), all child trades go with it. Cascade is the cleanest semantic; alternative (RESTRICT) would require manual two-step deletion in any UI delete-run flow. F5+ may add soft-delete (`deleted_at` column) if forensic preservation matters; currently runs are operator-controlled and re-runnable.

### `select_trades_by_run` async function

```python
async def select_trades_by_run(
    conn: _DbExecutor,
    *,
    run_id: UUID,
    limit: int = 100,
    offset: int = 0,
) -> list[BacktestTradeRow]:
    """Return paginated trades for a backtest run (ORDER BY opened_at ASC).

    Default limit 100; T-516 UI may paginate but typical backtest produces
    < 500 trades. ASC ordering matches chronological replay sequence.
    """
```

SQL:
```sql
SELECT id, run_id, bot_id, signal_id, open_order_id, close_order_id,
       symbol, side, entry_price, exit_price, qty, notional_usd,
       realized_pnl, fees_paid, close_reason, opened_at, closed_at,
       status, mfe_pct, mae_pct, confidence_score, meta
FROM backtest_trades
WHERE run_id = $1
ORDER BY opened_at ASC
LIMIT $2 OFFSET $3
```
