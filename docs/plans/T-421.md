**Per OQ-1=A — 4 separate dashboards**:
1. `service-health.json` — per-service `up` + scrape duration + process metrics (CPU/memory/fds via Python `prometheus_client` defaults) + spec'd metrics (`open_positions{bot_id}`, `virtual_balance{bot_id}`, `ws_connected{stream}`, `db_pool_saturation{service}`) per BRIEF §15.4:2156.

3. `pg.json` — `db_pool_saturation{service}` (per-service exposed metric) + connection-count + slow-query placeholders for postgres_exporter.
4. `host.json` — `node_*` metrics (CPU / memory / disk / network) — empty until F5+ adds node-exporter scrape.

**Per OQ-2=B + OQ-3=B + OQ-4=B — empty-metric placeholders are valid PromQL**: tests assert PromQL parses, NOT that metrics return data (per `test_dashboard_query_parses` docstring "asserts only `status='success'` — result data is not inspected, so this stub is robust to scrape-coverage drift"). Dashboards reference spec'd metric names that may currently be empty; populated when F5+ exporter sidecars land.

**Per OQ-5=A — service-health dashboard renders BOTH** existing scrapes (process_*) AND spec'd metrics (open_positions etc. empty until services emit).

**Explicitly out of scope for T-421**: prometheus-nats-exporter sidecar (F5+ — adds scrape entry + container; T-421 ships dashboard placeholder); postgres_exporter sidecar (F5+); node-exporter sidecar (F5+); Grafana templating (`$var` / `${var}`) — `tests/grafana/conftest.py` line 25 explicitly defers template-substitution to "F1's first var-bearing dashboard task" (T-421 stays var-free); per-bot drill-down (BRIEF §15.4 line 2161 "trading dashboards live in custom UI, not Grafana"); Alertmanager rules (BRIEF §15.5 separate scope; F5+); error boundary equivalent (Grafana shows zero-data panels gracefully); CEST/UTC toggle (F5+ — Grafana timezone config); per-dashboard dark/light theme override; Prometheus alerting rules (F5+); unified panel theme overrides — use Grafana defaults.
