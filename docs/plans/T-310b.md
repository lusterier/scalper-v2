## Purpose

Wire the strategy-engine signal consumer end-to-end. Subscribe to `signals.validated`, run per-signal evaluation (TTL → symbol filter → resolve signal_id → evaluate via T-307 → write `scoring_evaluations` audit row → publish OrderRequest or SignalRejected per Decision), close BRIEF §9.4:1532-1543 main loop.

T-310b wires the existing T-310a-shipped surfaces:
- BotConfig (T-308 + T-310a sections) read from `app.state.bot_config`
- FeatureResolver (T-306) read from `app.state.resolver`
- T-307 `evaluate(...)` for scoring
- T-310a `select_signal_id_by_idempotency_key` for SignalValidated.idempotency_key → signals.id
- T-301 `insert_scoring_evaluation` for audit row
- T-310a `SignalRejected` schema + `subject_for_signals_rejected` helper for reject publish
- `OrderRequest` schema (T-216a) + `subject_for_orders_request` helper for execute publish

## Scope

### In scope (T-310b)

1. **`services/strategy_engine/app/consumer.py`** (~200 LOC) — handler factory + per-signal evaluate loop:
   - `make_signal_handler(*, bot_id, bot_config, resolver, pool, bus, bound_logger, now_fn, max_signal_age_seconds) -> Handler` closure factory
   - Inner `_handle(envelope: MessageEnvelope)` async function performs §9.4 steps 3a-3h:
     - **`now` snapshot at handler entry** — single `now = now_fn()` call reused by 3a (TTL), 3c (`received_at_lower_bound = now - timedelta(seconds=max_signal_age_seconds)`), and 3f (`rejected_at=now`). Single snapshot keeps tests deterministic + avoids microsecond drift across the 3-step chain.
     - **3a TTL check** — `if signal.expires_at < now: log signal_expired, return` (brief §9.4:1536 verbatim — strict `<`; equal-to-now passes)
     - **3b symbol filter** — `if signal.symbol not in bot_config.symbols: log signal_outside_universe, return`
     - **3c+3d resolve signal_id + evaluate** — lookup signal_id via T-310a helper using `received_at_lower_bound = now - timedelta(seconds=max_signal_age_seconds)`; if None log `signal_id_unresolved` + return (race or expired); call `await evaluate(bot_config=, signal=, resolver=, bound_logger=)`; receive `ScoringResult`
     - **3e/3f/3g decision dispatch**:
       - `decision == "execute"` or `decision == "passthrough"` → build `OrderRequest` per §9.4:1540 + publish to `orders.requests.<bot_id>`
       - `decision == "reject"` → build `SignalRejected` + publish to `signals.rejected.<bot_id>` (T-310a schema)
     - **3h scoring_evaluations write** — call T-301 `insert_scoring_evaluation` for audit before ack-equivalent (handler return). Publish-after-persist per T-200 Q2 — write to DB FIRST (publish-after-persist invariant: even if publish fails, audit row exists).
   - **publish-after-persist ordering**: scoring_evaluations INSERT inside tx → tx commits → publish OrderRequest / SignalRejected OUTSIDE tx. Mirror T-216b2 + T-219 precedent.
   - Error handling per branch:
     - `signal_id_unresolved` (lookup returns None) — log INFO + return; race condition / expired window
     - `evaluate(...)` raises — log ERROR `scoring_evaluator_crashed` + return; H-019 per-rule fail-open is T-307's; consumer-level crash is bug
     - DB INSERT raises — log ERROR `scoring_evaluations_insert_failed` + return (no publish; signal will be re-processed if signal-gateway re-emits — but `signals_idempotency` UNIQUE prevents duplicate INSERT and T-310b's lookup returns same id)
     - Publish raises (`PublishError` / `NotConnectedError`) — log ERROR `orders_request_publish_failed` or `signals_rejected_publish_failed` + return; audit row already written so operator can see decision was made even if downstream missed
   - **`OrderRequest` field mapping** (Decision==execute or passthrough):
     - `bot_id` ← `bot_config.bot_id`
     - `signal_id` ← T-310a-resolved id
     - `symbol` ← `signal.symbol`
     - `side` ← side mapping (see below)
     - `order_type` = `"market"` (v1 default; OrderRequest schema default)
     - `qty` / `leverage` / `sl_pct` / `tp_pct` / `tp_qty_pct` / `be_trigger` / `be_sl_level` / `trail_pct` ← `bot_config.execution.*`
     - `exchange_mode` ← `bot_config.exchange.mode`
   - **`signal.action` value-set handling** — `SignalValidated.action` is Pydantic-validated `Literal["LONG", "SHORT", "CLOSE"]` at signal-gateway publish (T-015b1) so unknown values are unreachable in T-310b. Three reachable values:
     - `"LONG"` → `side = "buy"`
     - `"SHORT"` → `side = "sell"`
     - `"CLOSE"` → log `consumer.close_action_unsupported_v1` + return (v1 limitation per §0.8 — close-action support requires own-bot position lookup, deferred to F4+).

2. **`services/strategy_engine/app/main.py` lifespan extension** (~30 LOC):
   - Step 6 (after state attach, before yield): subscribe to `signals.validated` with the bot-bound handler from `make_signal_handler(...)`. **Single subscription per process** — strategy-engine instance handles ONE bot per BRIEF §9.4:1530 ("One Docker container per active bot").
   - **DI signature**: `await bus.subscribe("signals.validated", make_signal_handler(...))`. Subscribe failure crashes lifespan (mirror T-216a WG#7 fail-fast).
   - **Settings extension**: add `signal_max_age_seconds: int = 600` to `services/strategy_engine/app/config.py` for the T-310a `received_at_lower_bound` parameter (configurable per §N9; 600s default = 10 min covers TTL 120s + retry budget + clock skew).

3. **`services/strategy_engine/app/deps.py` extension** (~10 LOC):
   - `get_resolver(request)` provider returning `FeatureResolver` from `app.state.resolver` (T-309 deferred per §0.8; T-310b owns the consumer wiring so the provider lands here).

## Edge cases

- **`MessageEnvelope.payload` not parseable as `SignalValidated`** — Pydantic ValidationError caught at handler entry → log + return (test #14)
- **`signal.expires_at == now`** — boundary; spec §9.4:1536 says `expires_at < now` (strict `<`) so equal-to-now is NOT expired. T-310b uses brief verbatim `<`. Tested via `test_expired_signal_is_dropped_at_strategy_engine` with `expires_at = now - 1s` (clearly past).
- **`signal_id` lookup returns None** — race window (signal-gateway's INSERT not yet visible OR signal too old per `received_at_lower_bound`). Log + skip. Tested.
- **`Decimal` precision in OrderRequest** — `bot_config.execution.qty: Decimal` passed straight to `OrderRequest.qty: Decimal`. No coercion in T-310b. Test pin verifies field-by-field equality.
- **CLOSE action** — operator may publish a CLOSE signal via TradingView. Out of scope for v1 (would need own-bot position lookup). Log + return.
- **Subscribe failure at lifespan** — crashes lifespan, service exits. Mirror T-216a fail-fast (subscribe failure during startup = misconfigured infra; runtime-non-recoverable).

## §N invariants

- **§N1 UTC** — `now_fn=lambda: datetime.now(UTC)` injected at lifespan call site; `signal.expires_at` already UTC-validated at SignalValidated layer (T-015b1); `received_at_lower_bound = now_fn() - timedelta(seconds=max_signal_age_seconds)`; `rejected_at = now_fn()` in SignalRejected envelope.
- **§N2 logging** — 8 distinct event labels routed across §N2 streams (`trading.log` for per-signal decision audit, `system.log` for service-level errors, `audit.log` for post-persist anomalies):
  - `signal_expired` (info, **`trading.log`**, H-008) — per-signal lifecycle event
  - `signal_outside_universe` (info, **`trading.log`**) — per-signal lifecycle event
  - `signal_id_unresolved` (info, **`trading.log`**) — per-signal lifecycle event
  - `consumer.close_action_unsupported_v1` (info, **`trading.log`**) — per-signal lifecycle event
  - `scoring_evaluator_crashed` (error, **`system.log`**) — service-internal exception (programming bug; T-307 evaluator should not raise)
  - `scoring_evaluations_insert_failed` (error, **`system.log`**) — DB write fail
  - `consumer.signal_validated_validation_failed` (error, **`system.log`**) — envelope-parse fail (programming/upstream bug)
  - `orders_request_publish_failed` / `signals_rejected_publish_failed` (error, **`audit.log`** + `system.log`) — post-persist publish miss; audit-log entry pins the decision-was-made-but-downstream-missed contract for operator visibility (audit row already in `scoring_evaluations`)
- **§N3 idempotency** — `insert_scoring_evaluation` already `@non_idempotent` (T-301); T-310b consumer body mirrors T-216a precedent — handler entry deduplicates implicitly via NATS server-side `signals.validated` stream `duplicate_window` (T-015b1 message_id_for + signal-gateway publish). T-310b adds **no in-process dedup ring** — single-bot single-process; signal repeats from upstream are server-deduped. (If F5+ adds JetStream durable consumer, dedup ring may be needed at strategy-engine; T-310b §0.8.)
- **§N6 no globals** — closure factory; bot_id/bot_config/resolver/pool/bus/logger bound at registration, not module-level.
- **§N7 hexagonal** — DB via `packages.db.queries.scoring`/`signal_gateway`; bus via `packages.bus.NatsClient`; scoring via `packages.scoring.evaluate`; thin consumer layer.
- **§N9 configurability** — `signal_max_age_seconds` env-tunable; bot-side qty/leverage/etc. read from BotConfig (per-bot YAML).
