## Purpose

Two read-only endpoints for the dashboard's feature inspector section (BRIEF §9.6:1627 + §14.3:2065):

1. **`/api/features/latest`** — name-prefix-filtered KV view, one row per `(feature_name, symbol)` pair. Backs the "feature browser" UI: operator filters by name prefix (e.g., `ind.btcusdt`) and sees the latest computed value + source_version per matching feature.
2. **`/api/features/history`** — time-series for one `(feature_name, symbol)` pair over a range. Backs the chart UI.

Per §0.8 anti-hypothetical:
- NO staleness flag inline / separate endpoint — UI computes `stale = age > threshold` client-side per its own per-feature policy (per OQ-6 default C; backend stays policy-free)
- NO `value` polymorphic discriminator — three columns (`value_num`, `value_bool`, `value_json`) preserved faithful to schema; UI knows which to read per feature definition (per OQ-5 default A)
- NO live SSE feature updates → T-408
- NO feature-name search by full-text / regex → opportunistic per UI demand

T-404 is read-only; audit emission N/A.

@dataclass(frozen=True, slots=True)
class FeatureRow:
    """Full projection of features row (7 fields per §7.2:903-911)."""
    feature_name: str
    symbol: str
    computed_at: datetime
    value_num: float | None       # DOUBLE PRECISION → float per §5.13 (statistical, not money)
    value_bool: bool | None
    value_json: dict[str, Any] | list[Any] | None  # JSONB; could be object or array
    source_version: str


_LATEST_FEATURES_NO_FILTER_SQL = """
    SELECT DISTINCT ON (feature_name, symbol)
           feature_name, symbol, computed_at, value_num, value_bool,
           value_json, source_version
    FROM features
    ORDER BY feature_name, symbol, computed_at DESC
    LIMIT $1 OFFSET $2
"""

_LATEST_FEATURES_WITH_PREFIX_SQL = """
    SELECT DISTINCT ON (feature_name, symbol)
           feature_name, symbol, computed_at, value_num, value_bool,
           value_json, source_version
    FROM features
    WHERE feature_name LIKE $1
    ORDER BY feature_name, symbol, computed_at DESC
    LIMIT $2 OFFSET $3
"""

_COUNT_LATEST_FEATURES_NO_FILTER_SQL = """
    SELECT COUNT(*) AS n
    FROM (
        SELECT DISTINCT feature_name, symbol FROM features
    ) AS t
"""

_COUNT_LATEST_FEATURES_WITH_PREFIX_SQL = """
    SELECT COUNT(*) AS n
    FROM (
        SELECT DISTINCT feature_name, symbol FROM features WHERE feature_name LIKE $1
    ) AS t
"""

## Edge cases

- **`?prefix=`** (empty string) — treat as None (no filter); pin via `prefix or None` normalization in router OR pass through to query helper which checks `if prefix`. Default: helper checks `prefix is None` only; router passes empty string through if Query default behavior accepts it. Pin in test which path is taken.
- **`?prefix=ind.btcusdt`** (no trailing %) → query helper appends `%` for SQL LIKE; caller doesn't worry about LIKE syntax.
- **`?prefix=ind.btcusdt%`** (caller already added `%`) → query helper appends another `%` → SQL LIKE matches `ind.btcusdt%%` which is equivalent to `ind.btcusdt%`. Acceptable degenerate input.
- **`/api/features/history` with no `?from=` or `?to=`** → return all history rows for the feature, ordered DESC, limited by limit/offset
- **Feature with only `value_json` set, value_num + value_bool NULL** → response has `value_num: null, value_bool: null, value_json: {...}`
- **Feature with `value_json: []`** (array, not dict) → response has `value_json: []` — Pydantic accepts `dict | list | None` union
- **`computed_at` in future** (clock skew between feature-engine and analytics-api hosts) → no special handling; data shown as-is
- **Empty features table** → both endpoints return 200 with empty list
