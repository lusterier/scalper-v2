# T-300: `packages/scoring/types.py` foundational Pydantic types

**Status**: DRAFT for plan-reviewer (2026-05-02)

**Trace**: F3 §10.1, §10.4, §7.2:1036 (scoring_evaluations columns derived from these types)

## Purpose

Land the foundational Pydantic types for `packages/scoring/`. These types describe (a) the shape of a per-bot YAML config and (b) the shape of an evaluator's per-signal output. Subsequent F3 tasks consume these types:

- T-301 Migration 0010 columns mirror `ScoringResult` JSONB serialization (`rule_results`, `feature_snapshot`).
- T-302..T-305 condition modules narrow `ScoringRule.condition` from placeholder `dict[str, Any]` to a discriminated Pydantic union.
- T-306 resolver consumes `ScoringRule.feature` reference + `max_staleness_sec`.
- T-307 evaluator pipeline returns `ScoringResult` per §10.4 verbatim pseudocode.
- T-308 YAML loader parses to `BotConfig`.

This task introduces zero I/O and zero business logic. Pure schema + validation.

## Public interface

`packages/scoring/types.py` (ALL public via `__init__.py` re-export):

```python
Decision: TypeAlias = Literal["execute", "reject", "passthrough"]


class RuleResult(BaseModel):
    """Per-rule evaluation outcome (§10.4 result row)."""
    name: str
    weight: float
    applied_weight: float
    result: str  # "True" | "False" | "n/a" | "skipped" | "error_skipped" | "data_missing" | "data_stale"
    error: dict[str, Any] | None = None


class ScoringResult(BaseModel):
    """Top-level evaluator output (§10.4 return type)."""
    decision: Decision
    total_score: float
    threshold: float
    rule_results: list[RuleResult]
    feature_snapshot: dict[str, dict[str, Any]]  # opaque inner shape; T-307 owns serializer
    config_version: int  # mirrors BotConfig.version for scoring_evaluations.config_version
    reason: str | None = None  # §10.4:1749/1754 reject reason; None on execute/passthrough


class ScoringRule(BaseModel):
    """One rule within a bot's scoring config (§10.1)."""
    name: str
    weight: float
    feature: str  # templated reference, e.g. "ind.${signal.symbol}.15m.ema_20"
    applies_when: dict[str, Any] | None = None  # T-302 narrows to ConditionUnion | None
    condition: dict[str, Any]                   # T-302 narrows to ConditionUnion (REQUIRED)
    on_error: Literal["skip", "reject"] = "skip"
    required: bool = False
    max_staleness_sec: int | None = None  # None → resolver default 2 × interval_seconds


class ScoringConfig(BaseModel):
    """Scoring section of BotConfig (§10.4)."""
    mode: Literal["active", "passthrough"] = "active"
    trigger_threshold: float
    rules: list[ScoringRule]


class BotConfig(BaseModel):
    """Top-level bot YAML config (§9.4 + §10)."""
    bot_id: str
    version: int = Field(default=1, ge=1)  # audit trail for scoring_evaluations.config_version
    symbols: list[str]                      # bot's trading universe
    scoring: ScoringConfig
```

All models `frozen=True` via `model_config = ConfigDict(frozen=True)` per §5.3 immutability convention.

## Validation rules

Beyond Pydantic field-type checks, the plan requires these explicit validators:

1. `BotConfig.bot_id` non-empty string + matches `^[a-z][a-z0-9_-]*$`. **T-300 task-level convention** (NOT verbatim brief) — brief §7.2:850 declares `bots.bot_id TEXT` with examples `'alpha', 'beta'`; charset regex codifies env-var compatibility (`BOT_<ID>_BYBIT_API_KEY` per ADR-0004 H-022) + FK reference safety. Operator-approved Path A 2026-05-02.
2. `ScoringConfig.rules`: rule names unique within bot config (`ValueError("duplicate rule name: <X>")`).
3. `ScoringRule.weight` is `float`; can be negative (subtractive) per §10.1.
4. `ScoringResult.total_score` ≈ sum of `applied_weight` across `rule_results` — out of T-300 scope (cross-field invariant; T-307 evaluator owns + property-tests).
5. `RuleResult.error` ↔ `result` correlation (e.g. `error` non-None ⇒ `result ∈ {"skipped", "error_skipped", "data_missing", "data_stale"}`) is **out of T-300 scope** — evaluator-side convention enforced/tested at T-307. T-300 keeps `error` and `result` as independent fields (caller-set).

## Edge cases

- **Empty rules list**: `ScoringConfig.rules = []` is valid (operator-driven; bot effectively passthrough always-execute when total_score=0 below threshold or scoring.mode=passthrough). Test pin documents.
- **Mode=passthrough + threshold present**: brief §10.5 says "scoring is evaluated normally"; threshold field is required but doesn't gate decision. Test pin: passthrough mode preserves total_score computation but `decision='passthrough'` regardless.
- **Negative weights**: rule with `weight=-1.5` valid. Subtracts from total. Test pin.
- **Rule name collision**: two rules with same `name` → ValueError at BotConfig construction.
- **Forward-ref placeholder**: `condition: dict[str, Any]` accepts any dict; T-302 narrowing will replace with `Annotated[Union[...], Field(discriminator="type")]`. Plan-time comment in code documents this.
- **`feature_snapshot` shape**: T-300 keeps it opaque (`dict[str, dict[str, Any]]`); the inner value-dict shape (Decimal-as-str for `value_num` per §N1, etc.) is **T-307 evaluator's responsibility** — `FeatureValue` from `packages/features/types.py` is a frozen `@dataclass`, not Pydantic, so the T-307 serializer (helper added at that task) maps `FeatureValue → jsonable dict` with explicit Decimal-string preservation. T-300 test only verifies that `ScoringResult` round-trips through `model_dump()` with arbitrary nested dict content.
