# T-527a — §B.1 sizing config foundation (foundation leaf, no consumer)

**Status:** Plan-reviewer Gate-1 APPROVE + 5 WG (verbatim below). Implemented. **Plan-vs-reality correction (T-536/T-533a/T-533b1 tooling-correction class, L-018, in-body, drift-checker to adjudicate):** WG#3 / §Scope #2 said `_parse_sizing` "mirror `_parse_shadow` EXACTLY". `_parse_shadow` reads keys explicitly and never `**spec`-splats → an unknown `shadow:` key is *silently dropped*. For `sizing` that is a capital-safety hazard: the operator OQ-2=A-**deferred** `tier_promotion`/`tier_demotion` (or any typo) MUST be *loudly rejected* at YAML load (silently ignoring a `tier_promotion:` the operator wrote — believing promotion is active — is exactly what WG#2's "extra='forbid' rejects them at YAML load" guards against; surfaced by the failing `test_yaml_loader_sizing_rejects_deferred_tier_promotion` + `..._rejects_typo_key`). Corrected to mirror **`_parse_risk`'s `RiskSection(**coerced)` splat** instead (preserve every key → `SizingSection` `extra="forbid"` catches unknowns), adapted for the nested per-element `_to_decimal` coercion the flat `risk:` block does not need. This makes WG#2 literally true and is the safer design. L-007 split leaf 1/2 of T-527 (split-record `docs/plans/T-527.md`). **T-527b** (the sizing compute) is blocked-by this and is the first consumer.
**Phase:** F5 (ADR-0011 pre-live operational hardening — position-sizing cluster; TASKS.md:222-225).
**Operator OQs (resolved, baked — T-527 plan-stage 2026-05-16, all default A):** OQ-1=A 2-leaf split (this = config foundation); OQ-2=A tier_promotion/tier_demotion deferred → NEW backlog ticket, not modeled here; OQ-3=A `size`/`max_notional` = notional USD (binds T-527b; schema types them Decimal-USD); OQ-4=A tier by `total_equity`, sub-lowest-tier → skip (binds T-527b; schema validator guarantees a well-defined lowest tier). No unresolved questions.

## Purpose

Model the BRIEF-deferred §B.1 `sizing:` block as a typed Pydantic config section + parse it from YAML + thread the scoring `score` producer→wire onto `OrderRequest`, **with zero behavior change and no consumer**. This is the foundation T-527b builds on (T-527b computes `qty` from `sizing` + `score` + account balance). Today `OrderRequest.qty = bot_config.execution.qty` (static, `consumer.py:428`); the §B.1 `score_multipliers` need the numeric score which is currently **dropped** after the accept decision (`_publish_order_request` `consumer.py:394` does not receive `result`). T-527a closes the schema gap + the score-plumbing gap; it does NOT change how `qty` is chosen (still `execution.qty`). A bot YAML with no `sizing:` block stays byte-identical in behavior.

## Scope — files touched (4 src + sibling tests)

### 1. `packages/scoring/types.py` — NEW `SizingTier` + `SizingSection`; `BotConfig.sizing` field

- **NEW `class SizingTier(BaseModel)`** — `model_config = ConfigDict(frozen=True, strict=True)` (mirror `ExecutionSection:194`). Fields: `balance_min: Decimal`, `size: Decimal` (both USD notional per OQ-3=A; §B.1 `{ balance_min: 500, size: 700 }`). `Field(..., ge=0)` on both (no negative balance/size).
- **NEW `class SizingSection(BaseModel)`** — `model_config = ConfigDict(frozen=True, extra="forbid")` (mirror `RiskSection:329` / `ShadowVariant:244` net-new-feature rationale — `extra="forbid"` catches operator YAML typos at load). Fields:
  - `tiers: list[SizingTier]` — the balance ladder.
  - `score_multipliers: dict[str, Decimal]` — §B.1 `{ "4": 0.75, ... "9": 1.5 }`; keys kept as YAML-native `str` (the float-score→key mapping is T-527b compute logic, NOT schema — T-527a only stores the dict). Values `Decimal` (multiplier feeds T-527b qty math → §5.13).
  - `max_notional_per_symbol: dict[str, Decimal]` — §B.1 `{ default: 3000, BTCUSDT: 5000 }`; per-symbol USD-notional cap with a required `default` key.
  - **NO `tier_promotion` / `tier_demotion` fields** — OQ-2=A deferred; they stay absorbed by `BotConfig.extra="ignore"` exactly as `sizing` is today, until the NEW backlog task. Docstring states this explicitly (mirror the `BotConfig` docstring's "shadow was on this list pre-T-514" note style).
  - **`@model_validator(mode="after")`** structural guards (mirror `ShadowConfig` structural validators — these are schema-structural, NOT financial math): (a) `tiers` non-empty; (b) `tiers` strictly ascending by `balance_min` (guarantees a well-defined "lowest tier" for OQ-4=A sub-threshold skip + a deterministic tier scan for T-527b); (c) `max_notional_per_symbol` contains the `"default"` key (T-527b falls back to it for non-overridden symbols); (d) `score_multipliers` keys are all digit-strings (well-formed; the numeric-range/clamp semantics are T-527b's). Each raises `ValueError` with an operator-actionable message.
- **`BotConfig.sizing: SizingSection | None = None`** (new field after `risk:` line 374). Optional-sentinel: `None` = no tier sizing → T-527b leaves the static `execution.qty` path byte-unchanged. Mirrors `shadow: ShadowConfig | None = None` (`:373`, T-514 precedent), NOT `risk`'s `default_factory` (an empty-tiers `SizingSection()` is not a meaningful "disabled" state; `None` is the explicit backward-compat sentinel — and `tiers` is non-empty-required by the validator, so a defaulted empty section is invalid by construction).
- **Docstring touches (no logic):** `BotConfig` docstring (`:353-361`) — remove `sizing` from the "unmodeled top-level keys … never reach the ctor" list (now modeled), mirror the existing "`shadow` was on this list pre-T-514; T-514 promotes it" phrasing. `ExecutionSection` docstring (`:184-192`) — the "F4+ task replaces with risk-based sizing reading `sizing.*`" note now points to "T-527 (reified)".

### 2. `packages/scoring/yaml_loader.py` — NEW `_parse_sizing`; wire into loader

- **NEW `_parse_sizing(spec: dict[str, Any] | None) -> SizingSection | None`** — `if not spec: return None` (absent/empty → None, backward-compat). **Mirrors `_parse_risk:415`'s `RiskSection(**coerced)` splat** (NOT `_parse_shadow`'s explicit key-reads — see Status plan-vs-reality: splat is required so `SizingSection` `extra="forbid"` loudly rejects the OQ-2=A-deferred `tier_promotion`/typos at YAML load instead of silently dropping them): `coerced = dict(spec)`, then replace the 3 known nested keys IF present with `_to_decimal`-coerced versions (each `tiers[i].balance_min`/`.size`, each `score_multipliers` value, each `max_notional_per_symbol` value → `Decimal(str(v))`, exact, §5.13/§N1), leaving every other key (unknowns, `tier_promotion`, typos) untouched → `SizingSection(**coerced)`. `score_multipliers`/`max_notional_per_symbol` keys stay str (not coerced). A present-but-malformed block raises a loud pydantic error (NOT a silent fall-back to static qty — the operator wrote `sizing:`, they intend tier sizing).
- **Wire into `load_bot_config_from_string`** (`:469-479`): add `sizing=_parse_sizing(data.get("sizing"))` to the `BotConfig(...)` construction (next to `risk=_parse_risk(data.get("risk"))`).
- (No `_SIZING_DECIMAL_FIELDS` frozenset needed — unlike the flat `execution:`/`risk:` blocks, `sizing:` is nested; coercion is structural inside `_parse_sizing`, mirroring `_parse_shadow`'s inline nested coercion rather than the flat `_EXECUTION_DECIMAL_FIELDS` set.)
