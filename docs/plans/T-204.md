## Decisions committed (verbatim per §7.2 + §18.3 — no operator input needed)

Per the T-203 calibration note codified 2026-04-27, items where the brief is verbatim-clear are recorded here as committed defaults rather than surfaced as OQs.

1. **Schema follows §7.2 lines 1091-1106 verbatim, no deviation.** 6 columns: `id BIGSERIAL`, `occurred_at TIMESTAMPTZ NOT NULL`, `bot_id TEXT` (nullable), `correlation_id TEXT` (nullable), `event_type TEXT NOT NULL`, `payload JSONB NOT NULL`. Composite PK `(occurred_at, id)` per hypertable convention.

2. **Hypertable on `occurred_at` with 7-day chunks** per §7.2 line 1103 verbatim — `SELECT create_hypertable('trading_events', 'occurred_at', chunk_time_interval => interval '7 days')`. Mirror 0002 + 0004 + 0005 hypertable shape.

3. **Retention 365 days** per §18.3 line 2438 + line 1176 verbatim — `SELECT add_retention_policy('trading_events', INTERVAL '365 days')`. Audit trail kept for 365 days, then chunks dropped.

4. **NO compression policy.** §18.3 lines 1178-1183 specify compression for `ohlc_1m` + `features` ONLY. `trading_events` retention is shorter (365d) and storage is JSONB-heavy with high cardinality — compression would offer marginal gain at the cost of decompress overhead during audit replay queries. Anti-test asserts NO `policy_compression` row for trading_events while asserting `policy_retention` IS present (combined positive + negative test).

5. **2 indexes per §7.2 lines 1104-1105 verbatim**:
   - `te_bot_type` ON `(bot_id, event_type, occurred_at DESC)` — per-bot per-event-type time-range queries (audit replay, alerting filter).
   - `te_correlation` ON `(correlation_id)` — cross-service correlation lookup for end-to-end trace.

6. **NO foreign keys on any column.** §7.2 verbatim has no `REFERENCES` anywhere. `bot_id` is plain `TEXT` nullable (not FK on `bots(bot_id)`); `correlation_id` is plain `TEXT` nullable (no `correlations` table exists). Append-only event-stream design: events outlive their source rows (a bot may be archived but its trading_events linger 365d). Anti-FK assertion in test for `bot_id` mirrors T-202 W#2 / T-203 W#4 precedent.

7. **`bot_id` and `correlation_id` BOTH nullable** per §7.2 verbatim. System-level events (e.g., reconciliation startup logs) have no specific bot context; reconciliation events may not carry a correlation_id from upstream. Smoke test exercises both NULL and populated cases.

8. **Server defaults**: NONE. Brief specifies no DEFAULTs. All NOT-NULL columns require explicit values at INSERT time; nullable columns default to NULL implicitly.

9. **`occurred_at` has NO server_default** per §N1 invariant (no `CURRENT_TIMESTAMP` / `NOW()` in SQL). Application-side responsibility — every emitter (T-216b, T-218, T-219, T-220, T-221) sets `occurred_at = now_utc()` at emit time. Test sub-check asserts `column_default IS NULL` for `occurred_at`.

## Edge cases

- **`bot_id` nullable** per brief verbatim. System-level events (reconciliation startup, audit cron tick logs) carry `bot_id = NULL`. The composite index `te_bot_type` on `(bot_id, event_type, occurred_at DESC)` still works for both NULL and non-NULL cases (Postgres indexes NULLs by default).
- **`correlation_id` nullable** per brief verbatim. Some events (reconciliation, audit) may not carry an upstream correlation_id. The single-column `te_correlation` index works for both cases.
- **`occurred_at` no server_default** per §N1. Every emitter must set `occurred_at = now_utc()`. (T-202 / T-203 precedent.)
- **No FK on bot_id** per brief verbatim. Append-only event-stream design — events outlive their source rows. A bot archived in `bots` would still have its trading_events visible until 365d retention drops them. Mirror T-202 W#2 / T-203 W#4.
- **Retention vs compression contrast.** §18.3 has retention for trading_events but NOT compression. Sub-check (g) is the combined positive (retention IS present) + negative (compression IS NOT present) anti-test. Catches both regression directions: accidentally adding compression (storage bloat with marginal benefit) AND accidentally dropping retention (forever-accumulating audit table).
- **`event_type` is open-ended TEXT**, not enum. Brief enumerates known values via comment but explicitly ends with `| ...`. New event types land at downstream-task time without DDL change.
- **`payload` is JSONB**. Schema-level: free-form. Application-level: each `event_type` may pin a payload structure (Pydantic model) at the emitter site (T-216b/T-218/etc.).

## Additional operator-locked items folded into plan

Beyond OQ-1..OQ-3 defaults, operator's instruction sequence added:

**ALI-1 — Composite PK uniqueness sub-check** (mirror T-203 OQ-10 pattern). Sub-check (j) — added after smoke E2E. `pytest.raises(asyncpg.exceptions.UniqueViolationError)` on duplicate `(occurred_at, id)`. Since `id` is BIGSERIAL/Identity, normal INSERTs never collide on `id`. Test forces a collision by manually specifying `id` in a second INSERT to a known-occurred_at + known-id tuple already inserted. Two ways to do this:

- **Approach (a)**: First INSERT uses `RETURNING id` to capture the auto-generated id; second INSERT explicitly sets `(occurred_at, id) = (same, same)` triggering UniqueViolation.
- **Approach (b)**: Both INSERTs explicitly specify `id = 999999999` with same `occurred_at`; second one collides.

Plan ships approach (a) — mirrors T-203 OQ-10 idiom (capture a fresh tuple, then re-insert it to force collision). Cleaner than picking a magic id.
