## Goal

Add **replay-mode** support to existing `PaperExchange` adapter so it can be driven by an injected `HistoricalOHLCSource` (T-503) instead of a live `market.ohlc.1m.>` NATS subscription. Each historical candle is expanded into 3 sequential intra-candle segment-candles via `generate_intra_candle_path` (T-505); each segment is fed through the existing `_check_sl_tp_crosses` cross-detection + `_drain_sl_tp_fill` persistence pipeline. This **eliminates the T-213a Q4-A pessimistic SL-first simplification in replay mode** — bullish candles with both SL and TP in `[low, high]` correctly fire TP first (segment 1 = open→high), bearish candles correctly fire SL first (segment 1 = open→low) per TradingView "Replay" semantics.

Live mode codepath is **unchanged**: existing `start_consuming()` / `_on_candle()` / `_check_sl_tp_crosses` continue to subscribe NATS + process closed candles with pessimistic SL-first. Backwards compatibility is total — all existing live-mode tests must pass without modification, and `mode="live"` is the default constructor value so existing call sites need no changes.

Per OQ-3=A, replay mode is **in-memory only** for position state (no `paper_positions` hydrate, no DB writes during the replay). T-507 CLI orchestrator owns final-outcome persistence to `backtest_runs` / `backtest_trades` (per BRIEF §12.2:1965-1967) — out of T-506 scope.

### `run_replay()` (NEW)

```python
async def run_replay(self) -> None:
    """T-506 replay-mode entry point: consume historical_source to exhaustion.

    Iterates ``self._historical_source`` (T-503 HistoricalOHLCSource); for each
    ``OHLCRow`` expands into 3 intra-candle segments via T-505
    ``generate_intra_candle_path`` and feeds each segment through
    ``_check_sl_tp_crosses_replay`` + ``_drain_sl_tp_fill``. Returns when
    source is exhausted.

    Pre-conditions:
    - ``self._mode == 'replay'`` (raises ``RuntimeError`` otherwise).
    - ``self._historical_source is not None`` (asserted; ctor validation
      ensured this).
    - ``self._active_positions`` may be empty (replay starts fresh per
      OQ-3=A) OR pre-seeded by T-507 CLI before calling run_replay
      (hook-style; T-506 itself does NOT pre-seed).

    Per OQ-3=A: NO ``paper_positions`` hydrate, NO DB writes during
    replay. ``_drain_sl_tp_fill`` writes to ``paper_orders``/``paper_executions``
    /``paper_trades``/``paper_positions`` via existing T-213b code path —
    in replay mode T-507 CLI uses a SEPARATE backtest-trades persistence
    layer; the live-DB writes via _drain are tolerable as ephemeral side-
    effects (operator runs replay against dev DB; production replay sandbox
    out-of-scope for F5).

    OQ-3=A clarification (write-side): T-506 itself does not call DB writes
    directly; ``_drain_sl_tp_fill`` continues to write to live tables. T-507
    CLI runs against a dedicated dev-only DB or accepts the ephemeral data.
    A future enhancement could wire a "no-persist" flag to suppress drain
    writes — flagged out-of-scope per §0.8 anti-hypothetical.
    """
    if self._mode != "replay":
        raise RuntimeError(
            f"run_replay() requires mode='replay', got mode={self._mode!r}"
        )
    assert self._historical_source is not None  # ctor validation guaranteed
    async for ohlc in self._historical_source:
        await self._process_replay_candle(ohlc)
```

### `_process_replay_candle(ohlc)` (NEW private)

```python
async def _process_replay_candle(self, ohlc: OHLCRow) -> None:
    """Expand a real candle into 3 intra-candle segments + drive fill checks.

    Mirrors live-mode ``_on_candle`` minus the NATS-envelope decode +
    ``is_closed`` short-circuit (replay candles are always closed by
    construction). Populates BOTH ``_last_price`` and ``_last_candle``
    caches per live-mode parity — the latter is read by
    ``_compute_slippage`` via ``place_market_order``, so omission would
    KeyError on the first signal-driven order during replay.
    """
    self._last_price[ohlc.symbol] = ohlc.close
    # Cache the FULL real candle (not per-segment) for _compute_slippage.
    # Contained schema lie: OhlcCandlePayload.source is Literal["binance"];
    # T-503 may yield source="bybit" rows. We hardcode source="binance"
    # in the synthesised payload because _compute_slippage / slippage.*
    # NEVER read candle.source — verified via grep of all .source accesses
    # in packages/exchange/paper/. The lie is contained at the cache
    # boundary; no downstream consumer observes it. (Alternative B: refactor
    # _compute_slippage signature to take bare (high, low) — rejected
    # because it forces live-mode test churn that status.md drift-warning
    # explicitly cautions against. Alternative C: parallel
    # _last_segment_range dict — rejected as additional state +
    # mode-branch in _compute_slippage.)
    self._last_candle[ohlc.symbol] = OhlcCandlePayload(
        symbol=ohlc.symbol,
        bucket_start=ohlc.bucket_start,
        open=ohlc.open,
        high=ohlc.high,
        low=ohlc.low,
        close=ohlc.close,
        volume=ohlc.volume,
        is_closed=True,
    )
    o, p1, p2, c = generate_intra_candle_path(
        open=ohlc.open, high=ohlc.high, low=ohlc.low, close=ohlc.close
    )
    for seg_open, seg_close in ((o, p1), (p1, p2), (p2, c)):
        seg_low = min(seg_open, seg_close)
        seg_high = max(seg_open, seg_close)
        await self._check_sl_tp_crosses_replay(
            symbol=ohlc.symbol, low=seg_low, high=seg_high
        )
        while self._pending_sl_tp_fills:
            fill = self._pending_sl_tp_fills.pop(0)
            await self._drain_sl_tp_fill(fill)
```

## Internal flow — worked example (bullish candle)

Consider a bullish candle `(open=100, high=110, low=95, close=105)` with active position holding `sl_price=92` + `tp_price=108`:

1. `generate_intra_candle_path(o=100, h=110, l=95, c=105)` returns `(100, 110, 95, 105)`.
2. **Segment 1**: `(100 → 110)`. seg_low=100, seg_high=110. `_check_sl_tp_crosses_replay(symbol, 100, 110)`:
   - SL branch: `low=100 ≤ sl_price=92 ≤ high=110`? NO (92 < 100). Skip.
   - TP branch: `low=100 ≤ tp_price=108 ≤ high=110`? YES. Enqueue `PendingSLTPFill(kind="tp", trigger_price=108)`.
   - Drain: process TP fill via `_drain_sl_tp_fill` — closes/updates trade per T-213b code path.
3. **Segment 2**: `(110 → 95)`. seg_low=95, seg_high=110. After TP drain, position is closed → `_active_positions[symbol]` is deleted by drain path → `_check_sl_tp_crosses_replay` returns early at `position is None`. No double-fill.
4. **Segment 3**: `(95 → 105)`. seg_low=95, seg_high=105. Position is None → no-op.

Outcome: TP fired at price=108, SL NEVER fired (correctly suppressed by ordering). T-213a Q4-A pessimism would have fired SL first because `low=95 ≤ sl_price=92` is FALSE (note: even Q4-A wouldn't have fired SL here since 92 < 95). The interesting case is when both SL and TP are in `[low, high]`:

**Counter-example with both in range**: `(open=100, high=110, low=90, close=105)`, `sl_price=95`, `tp_price=108`:

- Q4-A live: candle has `low=90 ≤ sl_price=95 ≤ high=110` → SL fires (worst-case for paper P&L, per T-213a docstring line 270).
- T-506 replay:
  - Seg 1 (100→110): low=100, high=110. SL `95 ≤ 100`? NO. TP `108 in [100,110]`? YES → TP fires at 108. Position closed. Net P&L: positive (TP exit).
  - Subsequent segments no-op.

This is the **TradingView "Replay" semantic** — bullish candles fire TP first because price travels up before reaching the candle's low. T-506 unlocks this fidelity in backtest runs.
