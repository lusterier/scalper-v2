## Purpose

Ship the `alerting-svc` per BRIEF §9.7:1649-1672 + §B.5:3114-3158. New service that:

1. Subscribes to NATS `system.alerts` (per OQ-7=A — `trading.events` critical filter deferred F5+).
2. Renders alert payload via Jinja2 template (per BRIEF §9.7:1667 + §B.5).
3. Posts rendered message to Telegram Bot API per channel (system / trading / pnl / security).
4. Deduplicates same-event-type within 5-minute window (in-memory dict per OQ-4=A; reset on restart).
5. Retries critical alerts on Telegram API failure (3 retries exponential backoff per OQ-3=A).

Service is the **7th service in the repo** (after signal-gateway / market-data / feature-engine / execution / strategy-engine / analytics-api). Mirror existing service skeleton structure (T-309 strategy-engine + T-400 analytics-api).

**Explicitly out of scope for T-409**: `trading.events` critical filter (F5+ — needs severity classifier on `orders.events.>` stream); per-user mention; admin-write notifications (T-401b symbol_map / T-405 bot_config.apply / T-407 backtest_run.queued already audit-logged but no alerts; F5+ may add `lan_admin_write` alert rule once operator reviews UI noise level); 4 of 5 BRIEF §B.5 templates (operator opens individual tickets when needed).

### `make_alert_handler` flow (`consumer.py`)

```
async def _handler(envelope: MessageEnvelope) -> None:
    payload = envelope.payload
    event_name = payload.get("event")
    if not event_name or not isinstance(event_name, str):
        logger.warning("alert_payload_missing_event", payload=payload)
        return  # swallow per NatsClient handler contract

    rule = alerts_config.find_rule(event_name)
    if rule is None:
        # Unknown event_name: log + drop (no template configured)
        logger.info("alert_no_rule_match", event=event_name)
        return

    # Optional threshold gate (per BRIEF §B.5 pnl_audit_correction example)
    if rule.threshold and not _passes_threshold(payload, rule.threshold):
        logger.debug("alert_below_threshold", event=event_name, threshold=rule.threshold)
        return

    dedup_key = _compute_dedup_key(rule, payload)
    if dedup.is_duplicate(dedup_key, now=now_fn()):
        logger.debug("alert_deduped", event=event_name, dedup_key=dedup_key)
        return
    dedup.mark(dedup_key, now=now_fn())

    rendered = render_template(rule.template, payload, env=jinja_env)
    is_critical = rule.severity == "critical"
    await telegram_client.send(channel=rule.channel, text=rendered, is_critical=is_critical)
```

### `DedupTracker` class (`dedup.py`)

```python
class DedupTracker:
    """In-memory per-key timestamp map; lazy cleanup on access.

    Per OQ-4=A — F4 single-process scope; restart loses state (acceptable
    operational trade-off vs NATS KV bucket complexity). Cleanup happens
    on every is_duplicate() call: entries older than window_seconds removed
    O(n) — n is small (~10s of entries at single-operator scale).
    """

    def __init__(self, window_seconds: int) -> None: ...

    def is_duplicate(self, key: str, *, now: datetime) -> bool: ...

    def mark(self, key: str, *, now: datetime) -> None: ...

    def _cleanup(self, now: datetime) -> None:
        """Drop entries older than window_seconds. Idempotent."""
```

`_compute_dedup_key` per OQ-4=A: `sha256(event_name + sorted-payload-fields-as-canonical-string).hexdigest()` — payload-content-aware so two identical alerts dedupe but two distinct alerts of same type don't (e.g., two different `pnl_audit_correction` for different sub_accounts both fire).

### `TelegramClient` (`telegram.py`)

```python
class TelegramClient:
    """httpx.AsyncClient wrapper for Telegram Bot API per OQ-2=A.

    Retry policy per OQ-3=A: critical messages retry ``max_retries`` times
    with exponential backoff starting at ``initial_backoff_s`` (defaults
    3 retries × 1s/2s/4s); non-critical drop on first failure with WARN
    log. Both knobs flow through Settings DI (per BLOCKER #1 fix +
    L-001 active control — env-tunable for incident-time adjustments
    without redeploy, mirror T-408 SSE knobs Settings pattern).

    All sends are ``@non_idempotent`` per §N3 (Telegram doesn't guarantee
    server-side dedup; retried delivery may double-notify). Marker is
    documentation/grep-friendliness only per
    :mod:`packages.exchange.paper.persistence` precedent (lines 11-15) —
    no Protocol conformance test on ``TelegramClient`` (not a Protocol
    implementation; F5+ has no plan to make it one).
    """

    def __init__(
        self,
        *,
        token: str,
        channel_chat_ids: dict[str, str],
        max_retries: int,
        initial_backoff_s: float,
        logger: Any,
    ) -> None: ...

    @non_idempotent
    async def send(self, *, channel: str, text: str, is_critical: bool) -> None:
        chat_id = self._channel_chat_ids[channel]
        url = f"https://api.telegram.org/bot{self._token}/sendMessage"
        body = {"chat_id": chat_id, "text": text, "parse_mode": "HTML"}
        max_attempts = self._max_retries if is_critical else 1
        backoff = self._initial_backoff_s
        for attempt in range(1, max_attempts + 1):
            try:
                resp = await self._client.post(url, json=body, timeout=10.0)
                resp.raise_for_status()
                return  # success
            except (httpx.HTTPError, httpx.TimeoutException) as exc:
                if attempt == max_attempts:
                    severity = "ERROR" if is_critical else "WARN"
                    self._logger.warning(  # or .error if critical
                        "alerting_telegram_failed_critical" if is_critical else "alerting_telegram_failed",
                        channel=channel,
                        attempts=attempt,
                        error=str(exc),
                    )
                    return  # drop; no dead-letter in F4
                await asyncio.sleep(backoff)
                backoff *= 2  # exponential

    async def aclose(self) -> None:
        await self._client.aclose()
```

### `configs/alerts/templates/default.j2` (NEW catch-all)

Per OQ-5=A — minimal template renders any event into readable Telegram message. Per CONCERN #1 fix — payload truncated at 3500 chars to stay under Telegram 4096 message limit (with header overhead):

```jinja2
<b>[{{ severity | upper }}]</b> {{ event }}
{% if correlation_id %}<i>correlation_id:</i> <code>{{ correlation_id }}</code>{% endif %}
{% if published_at %}<i>at:</i> {{ published_at }} CEST{% endif %}
{% if payload %}<pre>{{ payload | tojson(indent=2) | truncate(3500, end='... [truncated]') }}</pre>{% endif %}
```

**Length safeguard rationale (CONCERN #1)**: Telegram Bot API rejects messages > 4096 chars with HTTP 400. Indented JSON of arbitrary `system.alerts` payload (esp. nested objects, stack traces) easily exceeds. `truncate(3500, end='... [truncated]')` reserves ~600 chars headroom for the `<b>...</b>` + `<i>...</i>` + `<pre></pre>` headers + correlation_id + timestamp lines. Per-template-author responsibility to keep messages bounded; the catch-all default ships with this guard.

Note: BRIEF §9.7:1665 specifies "CEST timestamps in display" — this template renders the timestamp as-is; F4+ may add jinja2 filter for UTC→CEST conversion. For F4 minimal scope, the publisher's `published_at` field is rendered verbatim with `CEST` literal suffix (the publishers emit UTC; UI/Telegram display layer is the conversion site; F4 acceptable: operator sees absolute time + can mentally add +2:00).

## Edge cases

1. **`alerts.yaml` missing / malformed**: lifespan raises `FileNotFoundError` / `ValueError` from `load_alerts_config`. Service fails to start (fail-fast per existing service convention).
2. **`channels[X].telegram_chat_id_env` not set in environment**: `load_alerts_config` validates at startup; raises `ValueError(f"channel {X!r} requires env var {var_name!r}")`. Lifespan crashes.
3. **`TELEGRAM_BOT_TOKEN` missing**: `Settings()` Pydantic raises ValidationError (required field, no default).
4. **NATS handler receives malformed event** (no `event` field, wrong type): handler logs `alert_payload_missing_event` WARN + returns (NatsClient swallow contract; one bad message doesn't kill subscription).
5. **Unknown event_name** (no rule matches): logs `alert_no_rule_match` INFO + returns. F4 wildcard rule catches everything; F5+ removing wildcard makes this path live.
6. **Threshold not met** (e.g., `pnl_audit_correction` with `abs_delta_usd < 10.0`): logs `alert_below_threshold` DEBUG + returns.
7. **Duplicate within dedup window**: logs `alert_deduped` DEBUG + returns; dedup state is per-process (restart loses state per OQ-4=A).
8. **Telegram API 4xx response** (e.g., 401 invalid token): `raise_for_status()` raises `httpx.HTTPStatusError`; caught in retry loop. Critical → 3× retry; non-critical → 1× drop.
9. **Telegram API 5xx response / timeout**: same retry path as 4xx (T-409 doesn't distinguish 4xx vs 5xx — both treated as transient; F4+ may differentiate to skip retries on 401/403).
10. **Telegram API rate limit (429)**: caught in retry loop; backoff respects `Retry-After` header is **NOT** implemented in F4 (acceptable — single-operator scale, ~5-10 alerts/hour worst-case far below Telegram bot limits).
11. **NATS bus disconnect mid-stream**: NatsClient handles reconnect transparently; subscription resumes.
12. **Critical alert burst — sequential dispatch** (per CONCERN #2): NatsClient handler runs sequentially per subscription. N critical alerts arriving in rapid succession serialize behind each other (worst-case `N × max_retries × backoff_sum` blocking time). At single-operator scale (~5-10 alerts/hr per plan estimate) this is not a concern. F5+ may dispatch each alert to `asyncio.create_task` for parallel sending if burst-handling becomes an issue (trade-off: loses ordering guarantees + complicates dedup race semantics).
13. **Lifespan shutdown with in-flight retry**: `bus.close()` drains incoming; `telegram_client.aclose()` closes httpx connection pool (any in-flight retry's `sleep` is cancelled by event loop teardown). Acceptable — at-most-once for in-flight retries.
14. **Dedup memory growth + cleanup-on-access bound** (per CONCERN #3): cleanup happens **only** on `is_duplicate` calls (lazy). Under heavy load (many distinct dedup keys per window) memory grows to ~entries × 64 bytes (sha256 hex + datetime). At 1000 distinct alerts/5min = 64 KB. If alerts STOP arriving, last-window-state is held indefinitely until next `is_duplicate` triggers cleanup. Worst-case 64KB held indefinitely after burst-then-quiet — bounded, not a leak; documented for future readers. F5+ may add periodic cleanup task if memory pressure surfaces.
15. **Env-var hot-reload — not supported** (per CONCERN #4): `TELEGRAM_BOT_TOKEN` + `TELEGRAM_CHAT_*` resolved once at lifespan startup. Mid-run env changes (e.g., systemd `Environment=` reload via `systemctl reload`) are NOT picked up — service restart required. F5+ may add SIGHUP handler if operator workflow demands hot-reload.
16. **`event: "*"` wildcard rule** (catch-all per OQ-5=A): `find_rule` matches first rule with `event == "*"` if no exact match. The wildcard literal is matched as a special-case sentinel in the lookup function (NOT a regex / glob); operator publishing literal `event: "*"` is a degenerate case (would be impossible to map to a specific rule). F4 explicit + intentional; F5+ may remove wildcard once specific rules cover the surface.
