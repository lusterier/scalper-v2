```python
@idempotent
async def kv_put(self, bucket: str, key: str, value: bytes) -> int:
    """Write `value` to JetStream KV `bucket` under `key`.

    Idempotent (§N3): NATS KV PUT replaces by key, last-write-wins.
    Same (bucket, key, value) yields the same final KV state regardless
    of call count. JS returns a monotonically increasing revision number;
    surfaced for callers that want to assert ordering or log-correlate.

    Bucket must be pre-provisioned by infra (T-012 streams.yaml). Missing
    bucket raises PublishError (config bug, fail loud per §0.4) — bus
    does not auto-create. Raises NotConnectedError from non-CONNECTED.
    """
```

### `packages/bus/schemas/feature_update.py`

```python
class FeatureUpdate(BaseModel):
    """§8.4 features.updated.<feature_name>.<symbol> payload, frozen.

    Mirrors §8.4 verbatim. value_num/value_bool/value_json are wire-level
    floats/bools/dicts. Exactly one of three is non-None — model_validator
    after-mode mirrors FeatureValue.__post_init__ invariant.

    value_json type stays `dict[str, object] | None` matching the domain
    contract `FeatureValue.value_json: Mapping[str, object]` (L-004) —
    open to non-float values; T-110c handles Decimal→float conversion
    for numeric sub-keys at the wire seam.
    """

    model_config = ConfigDict(frozen=True)

    schema_version: Literal["1.0"] = "1.0"
    feature_name: str
    symbol: str
    computed_at: datetime
    value_num: float | None = None
    value_bool: bool | None = None
    value_json: dict[str, object] | None = None
    source_version: str

    @field_validator("computed_at")  # mirror OhlcCandlePayload UTC validator
    @field_serializer("computed_at")  # ISO+00:00
    @model_validator(mode="after")  # exactly one of three


def subject_for(feature_name: str, symbol: str) -> str:
    """Build §8.1 line 1213 publish subject features.updated.<feature_name>.<symbol>.

    Centralised here so the spec literal lives in exactly one place
    (L-002 pattern). T-110c imports rather than constructing inline.
    """
    return f"features.updated.{feature_name}.{symbol}"
```

```python
    conn: _DbExecutor,

    feature_name: str,

    value_num: float | None,

    interval: str,

    Body materializes via comprehension over `await conn.fetch(...)` so
```

## Hand-off to T-110c (preconditions)

T-110c MUST satisfy these preconditions for `packages.db.queries.feature_engine` to function in production:

1. **JSONB codec on pool — TWO valid alternatives, T-110c plan-reviewer chooses one:**
   - **(a) Per-connection codec registration** — extend `packages/db/pool.py:create_pool` to accept `init: Callable[[Connection], Awaitable[None]] | None = None` and pass to asyncpg, OR per-acquire registration in T-110c FeaturePipeline before each `pool.acquire()` (verbose but no pool API change). T-110b integration tests use this approach (mirrors T-108 `test_0004_migration:55-62`).
   - **(b) `$N::jsonb` cast in SQL + `json.dumps()` outside codec mechanism** — precedent: `packages/db/queries/signal_gateway.py:96` uses `VALUES ($1, ..., $8::jsonb, ...)` with caller passing `json.dumps(value_json)`. Avoids global pool config; the cast is local to the query. Migration to this approach in T-110b would alter `insert_feature` signature to take `value_json: str | None` (pre-encoded JSON) — flagged but not pre-decided here.

   Without one of the above, production `insert_feature(value_json=...)` raises asyncpg `TypeError`. T-110b integration tests use (a) per-connection in fixtures.

3. **`subject_for` import** — `from packages.bus.schemas import subject_for` for §8.1 publish subject. NEVER inline-construct the `features.updated.…` subject string elsewhere (L-002 control).

4. **Workspace dep** — T-110c adds `scalper-v2-features` to `services/feature_engine/pyproject.toml.dependencies` on first OhlcCandle import.

Critical invariants:
- I1. interval validated against `_INTERVAL_TO_TABLE.keys()` allow-list **before** any f-string interpolation. SQL identifier safety guaranteed by allow-list.
- I2. Returned rows ASC by bucket_start so caller can `for row in result: registry.push(...)` in time order.
- I3. bucket_start always UTC-aware on output (TIMESTAMPTZ + asyncpg default tz handling).
- I4. source is a query parameter ($2), never interpolated.
- I5. Cagg time_bucket projection preserves bucket boundary alignment.
- I6. Empty result returns []; short result returns prefix; both are caller's responsibility.
