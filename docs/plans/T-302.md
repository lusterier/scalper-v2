```python
@dataclass(frozen=True, slots=True)
class RuleContext:
    """§10.6 RuleContext — the single argument to ``Condition.evaluate``.

    Carries the full evaluator-time context. Composite conditions
    (T-304) navigate sub-conditions which may reference different
    features within ``feature_snapshot``. Plugin conditions (T-305)
    receive the same shape per BRIEF §10.6:1809 verbatim.

    ``feature_ref`` is the rule's resolved feature reference (post
    template substitution per §10.3) — simple conditions look up
    ``feature_snapshot[feature_ref]`` to get THEIR feature value.
    Composite conditions ignore ``feature_ref`` and recurse into
    sub-conditions which carry their own resolved refs.
    """
    signal: SignalValidated
    feature_snapshot: Mapping[str, FeatureValue]
    feature_ref: str  # resolved name, e.g. "ind.btcusdt.15m.ema_20"


@runtime_checkable
class Condition(Protocol):
    """§10.2 + §10.6 Condition Protocol — implemented by ALL condition variants.

    ``@runtime_checkable`` enables ``isinstance(obj, Condition)`` for T-307
    evaluator dispatch.
    """
    type: str  # Pydantic discriminator field

    def evaluate(self, ctx: RuleContext) -> tuple[bool, dict[str, Any] | None]:
        """Return (outcome, error_info).

        - outcome: True if condition holds, False otherwise.
        - error_info: None on success; dict with "error" key on type-mismatch
          or unsupported-value-variant cases.
        """
```

## Validation rules

1. **Numeric conditions** (`gt`, `gte`, `lt`, `lte`, `between`) require `feature_value.value_num is not None`. If `value_num is None` → return `(False, {"error": "type_mismatch", "expected": "value_num", "got": <variant>})`.
2. **`equals` / `not_equals`** auto-detect variant via condition's `value` Pydantic-strict type: `Decimal` value compared against `feature_value.value_num`; `bool` value compared against `feature_value.value_bool`. `value_json` not supported (returns `(False, {"error": "value_json equality unsupported", "hint": "use plugin condition (T-305) for deep-dict equality"})`). Type-mismatch (e.g. `value: bool` but feature has `value_num` populated only) → return `(False, {"error": "type_mismatch", "expected": "value_bool", "got": "value_num"})`.
3. **`in`** condition: `values` list must be non-empty (Pydantic `Field(min_length=1)`). Membership test variant-detected from FIRST element of `values` (caller responsibility to keep variants homogeneous; mixed-variant lists are operator-config error).
4. **`between`**: `min ≤ max` enforced via `model_validator`. Boundary `feature_value == min` or `== max` returns `True` (inclusive per BRIEF §10.2:1698 verbatim "min ≤ feature ≤ max").
5. All conditions are `frozen=True, strict=True` per §5.3 + Pydantic strict-mode bool-vs-Decimal disambiguation.
6. **`feature_ref` missing from `ctx.feature_snapshot`**: each condition catches `KeyError` and returns `(False, {"error": "feature_missing", "feature_ref": ctx.feature_ref})`. Defensive against racy snapshot-build edge case.

## Edge cases

- **`feature_value.value_num is None` for numeric condition**: returns `(False, {"error": "type_mismatch"})` per Validation rule #1.
- **`feature_value.value_bool` against `gt/lt/etc.`**: same — numeric required.
- **Decimal precision**: comparisons use Pydantic-coerced `Decimal` for both sides. No float coercion. `Decimal(50000) > Decimal(49999.5)` returns True (exact).
- **`in` condition with mixed-variant values**: e.g. `values: [Decimal(1), Decimal(2), Decimal(3)]` against `feature_value.value_bool` → `(False, {"error": "type_mismatch"})`.
- **`between` condition with `min == max`**: valid (degenerate single-value); evaluates to `feature == min`.
- **`equals` with `value_json` feature**: returns `(False, {"error": "value_json equality unsupported"})`. Defers deep-dict equality to plugin condition.
- **`type` discriminator collision**: each condition's `type` Literal is unique per BRIEF §10.2 catalog. T-307 will compose `Annotated[Union[...], Field(discriminator="type")]`.

- **GteCondition(value=Decimal("50000"))** vs `FeatureValue(value_num=Decimal("50000"))` → `(True, None)` (inclusive `≥`).
- **BetweenCondition(min=Decimal("100"), max=Decimal("200"))**:
  - vs `value_num=Decimal("150")` → `(True, None)`.
  - vs `value_num=Decimal("100")` → `(True, None)` (inclusive lower boundary).
  - vs `value_num=Decimal("200")` → `(True, None)` (inclusive upper boundary).
  - vs `value_num=Decimal("99.999")` → `(False, None)`.
- **InCondition(values=[Decimal("1"), Decimal("2"), Decimal("3")])** vs `value_num=Decimal("2")` → `(True, None)`.
- **InCondition(values=[Decimal("1"), Decimal("2")])** vs `value_num=Decimal("3")` → `(False, None)`.
- **EqualsCondition(value=True)** vs `FeatureValue(value_bool=True)` → `(True, None)` (bool-variant detection). **Pydantic strict-mode pin**: `type(EqualsCondition(value=True).value) is bool` (NOT `Decimal("1")`) — without `strict=True` Pydantic smart-mode coerces `True → Decimal("1")` since `Decimal | bool` union tries Decimal first, silently breaking variant detection.
- **GtCondition(value=Decimal("50000"))** vs `FeatureValue(value_bool=True)` → `(False, {"error": "type_mismatch", ...})`.
- **BetweenCondition** with `min > max` → `ValueError` at construction (Pydantic model_validator).
- **GtCondition(value=Decimal("50000"))** with `feature_ref="missing"` not in snapshot → `(False, {"error": "feature_missing", "feature_ref": "missing"})` (defensive KeyError catch).
