## Purpose

**D9** (status.md / T-540 close-out RUN finding `b16d41a`): the **native** analytics-api (launched by `scripts/dev-up.sh`, not the container) emits structured logs with `service=signal-gateway` instead of `service=analytics-api` — a **§N2 violation** (structured JSON logs must carry the correct service identity; a wrong `service=` misattributes analytics-api's `system.log` / audit lines to signal-gateway, corrupting per-service log triage).

**Root cause (confirmed at plan-stage):**
- `services/analytics_api/app/config.py` `Settings(BaseSettings, extra="ignore")`: `service_name: str = "analytics-api"` — the default is correct; pydantic-settings maps it from env var `SERVICE_NAME` (uppercase). This is the established cross-service service-identity pattern (every service Settings reads `SERVICE_NAME`; `compose.yaml` sets it per-service — `:452 SERVICE_NAME: analytics-api`; `services/*/tests/test_config.py` assert the mapping).
- `scripts/dev-up.sh:24-29`: `if [[ -f .env ]]; then set -a; . "${REPO_ROOT}/.env"; set +a; fi` — sources `.env` with `set -a` (auto-export). `.env` (copied from `.env.example:18 SERVICE_NAME=signal-gateway` — signal-gateway's own correct value) puts `SERVICE_NAME=signal-gateway` into the exported environment.
- `scripts/dev-up.sh:78-80`: the analytics-api launch is `DATABASE_URL="…" NATS_URL="…" setsid nohup uv run uvicorn services.analytics_api.app.main:create_app …` — it explicitly pins `DATABASE_URL` + `NATS_URL` to override the sourced-`.env` values for the native process, but does **NOT** pin `SERVICE_NAME` → the inherited exported `SERVICE_NAME=signal-gateway` reaches analytics-api's `Settings()` → `service_name="signal-gateway"` → `get_logger(settings.service_name, …)` (`main.py:116`) → `structlog … .bind(service="signal-gateway", …)` (`packages/observability/logging.py:147`). The containerized path is unaffected (compose pins per-service `SERVICE_NAME`).
