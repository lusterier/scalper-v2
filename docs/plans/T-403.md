## Purpose

Two read-only endpoint groups for the dashboard's signal-feed + scoring-inspector sections:

1. **`/api/signals/*`** — signal feed paginated/filtered (BRIEF §9.6:1625 + §14.3:2061 live-signals-feed-last-50). Two endpoints: paginated list with filters + detail by id.
2. **`/api/scoring/*`** — per-signal scoring inspector reading rule-by-rule audit JSONB (BRIEF §9.6:1626 + §14.3:2066). One endpoint: `by-signal/{signal_id}` returns all bot evaluations for one signal.

Per §0.8 anti-hypothetical:
- NO `/api/scoring/` global paginated list — UI navigates from /api/signals/ list → drills into per-signal scoring (per OQ-4 default; add per opportunistic UI demand)
- NO scoring-evaluation-by-id detail endpoint — UI never needs single-bot evaluation in isolation
- NO inline join `signals + scoring_evaluations` — UI orchestrates client-side
- NO live SSE signal feed updates → T-408
- NO rich filters (correlation_id, idempotency_key) → opportunistic per OQ-2 default

T-403 is **read-only** (no admin writes); audit emission N/A.

    limit: int,
    offset: int,

    source: str | None,

    action: Action | None,
    ingestion_status: IngestionStatus | None,

    to_at: datetime | None,

    """Total count of signals matching filters."""


async def select_signal_by_id(
    conn: _DbExecutor,
    signal_id: int,
) -> SignalRow | None:
    """Return one signal row by id; None if missing.

    NOTE: Hypertable PK is composite (received_at, id) — this lookup
    has no chunk pruning predicate, so it walks every chunk. At MVP
    scale (10k-100k signals/month, 7-day chunks) this is acceptable;
    F5+ may add a `signals_id` btree index if perf bottleneck surfaces.
    """


async def select_scoring_evaluations_by_signal_id(
    conn: _DbExecutor,
    signal_id: int,

    NOTE: `se_bot_signal (bot_id, signal_id)` index per §7.2:1054 has

    index for `WHERE signal_id = $1` (signal_id is non-leading). This
    query walks every chunk of the hypertable, same as
    `select_signal_by_id`. At MVP scale acceptable; F5+ may add a
    separate `se_signal (signal_id)` index if perf bottleneck surfaces.
    ORDER BY bot_id ASC for deterministic UI ordering.

    `from_at` / `to_at` filter on `received_at` column (not closed_at like trades).
