## Purpose

Ship the durable, restart-surviving latch substrate for the daily-loss kill-switch (H-027) without yet wiring the trip logic. After T-525a1: a `bot_kill_switch_state` row can be written/read/cleared idempotently; a strategy-engine restart re-evaluates the latch at startup (clears a stale prior-UTC-day latch, retains a same-UTC-day latch); `RiskSection.daily_loss_limit_usd` exists as a typed Decimal config knob. T-525a2 then adds `sum_realized_pnl_since` + the `check_daily_loss_limit` gate that *reads* this substrate and *writes* the latch on trip.

```
### H-027 — Risk kill-switch latch must persist across restart + re-evaluate on startup

A latched risk kill-switch (daily loss limit — T-525a2; max drawdown — T-525b)
MUST persist in durable storage and be re-evaluated on service startup. The
latch lives in PG `bot_kill_switch_state` (migration 0018); an in-memory-only
latch would reset on a strategy-engine restart and silently re-enable a bot the
operator's risk limit had stopped — direct capital-loss exposure. On startup the
latch is reconciled: a daily latch whose `daily_anchor_date` precedes the current
UTC date is cleared (a new trading day); a same-UTC-day latch is retained
(restart must NOT reset an active stop).
**Test.** `test_kill_switch_latch_survives_simulated_restart_same_utc_day` (latch
written → fresh reconcile+read simulating restart → still tripped) +
`test_reconcile_clears_stale_prior_utc_day_latch`.
```

T-525a1 ships the **persistence + reconcile** half (the latch substrate + startup re-evaluation). T-525a2 ships the **writer** (the gate that trips the latch on a Decimal P&L threshold). ## Hand verification (latch state-machine + UTC-day reconcile — NO Decimal arithmetic in a1)

T-525a1 has no P&L math (that is T-525a2). The only verifiable logic is the reconcile/stale-day predicate. UTC clock.

`is_stale_daily_latch(state, now)` truth table:

| state.tripped | trip_reason | daily_anchor_date | now(UTC).date() | result |
|---------------|-------------|-------------------|-----------------|--------|
| False | — | — | 2026-05-16 | **False** (not tripped) |
| True | 'daily_loss_limit' | 2026-05-15 | 2026-05-16 | **True** (stale → clear) |
| True | 'daily_loss_limit' | 2026-05-16 | 2026-05-16 | **False** (same UTC day → retain) |
| True | 'max_drawdown' | 2026-05-15 | 2026-05-16 | **False** (non-daily reason — drawdown is hard-stop, T-525b; not auto-cleared by day rollover) |

`reconcile_kill_switch_on_startup` flow:
1. `select_kill_switch_state(bot_id)` → None → log `kill_switch.reconcile_no_state` info, return.
2. state present, `is_stale_daily_latch(state, now)` True → `clear_kill_switch(bot_id, updated_at=now)` + log `kill_switch.reconcile_cleared_stale_daily_latch` info (with prior anchor + reason).
3. state present, predicate False, `tripped` True → log `kill_switch.reconcile_latch_retained` warning (operator-visible: this bot is stopped) — H-027 same-day retain.
4. state present, `tripped` False → log `kill_switch.reconcile_not_tripped` info, return.
5. any `asyncpg.PostgresError | OSError | TimeoutError` in 1-4 → log `kill_switch.reconcile_failed` error to system.log, return (best-effort; do NOT raise into lifespan).

**Restart-survives (H-027) walk:** latch row `tripped=true, reason='daily_loss_limit', anchor=2026-05-15` written 10:31. Process killed 11:30, restarts 11:32 (same UTC day 2026-05-15). `reconcile_kill_switch_on_startup`: select → tripped, `is_stale_daily_latch`=False (anchor==today) → branch 3: retained, warn-logged. Next signal (T-525a2 gate, future) reads tripped=true → blocked. **Restart did NOT reset the stop.** Next UTC day 2026-05-16 first boot/signal: `is_stale_daily_latch`=True → branch 2: cleared. ✓ H-027.
