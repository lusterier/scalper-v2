## Purpose

Wire **two named compose services** — `strategy-engine-alpha` + `strategy-engine-beta` — into `compose.yaml` per BRIEF §18.1 lines 2374-2384 verbatim shape. Each service uses the same Docker image (`scalper-v2/strategy-engine:dev`) parameterized by `BOT_ID` env var. T-313 E1 dvoj-bot integration test will spin up both.

T-310b shipped the consumer body + `BOT_ID`-required Settings (T-309). T-311 wires that into compose so the operator can `docker compose up strategy-engine-alpha strategy-engine-beta` and have two functionally-isolated bot processes running per BRIEF §3.1:245 ("strategy-engine is N processes, one per bot. Each bot has its own config, consumer group, and state. A crash in one bot does not affect others.").

## Public interface change

### `compose.yaml` services (NEW)

```yaml
strategy-engine-alpha:
  build:
    context: .
    dockerfile: services/strategy_engine/Dockerfile
  image: scalper-v2/strategy-engine:dev
  restart: unless-stopped
  networks: [backend]
  env_file:
    - path: /etc/scalper-v2/secrets.env
      required: false
  environment:
    SERVICE_NAME: strategy-engine
    LOG_LEVEL: INFO
    NATS_URL: nats://nats:4222
    BOT_ID: alpha
    BOT_CONFIG_DIR: /app/configs/bots
    PLUGIN_REGISTRY_PATH: /app/configs/plugin_registry.yaml
    BOT_CONFIRM_LIVE: ${BOT_ALPHA_CONFIRM_LIVE:-}
  depends_on:
    postgres: { condition: service_healthy }
    nats: { condition: service_healthy }
    nats-init: { condition: service_completed_successfully }
  healthcheck:
    test: ["CMD", "python", "-c", "import urllib.request,sys; urllib.request.urlopen('http://127.0.0.1:8000/health', timeout=2); sys.exit(0)"]
    interval: 10s
    timeout: 3s
    retries: 5
    start_period: 15s

strategy-engine-beta:
  # same shape, BOT_ID: beta + BOT_CONFIRM_LIVE: ${BOT_BETA_CONFIRM_LIVE:-}
```

## §16.5 Live-mode safeguard (BRIEF §16.5:2245)

Brief §16.5 says "Starting a bot with `exchange.mode: live` requires env var `BOT_CONFIRM_LIVE=yes` in the service's environment. Startup logs a loud warning `LIVE MODE ENGAGED` and sends a Telegram alert."

**T-311 ships only the env passthrough** per §18.1:2379 verbatim (`BOT_CONFIRM_LIVE: ${BOT_<ALPHA|BETA>_CONFIRM_LIVE}` in compose `environment:` block). The runtime enforcement (lifespan fail-fast when `bot_config.exchange.mode == "live"` AND `os.environ.get("BOT_CONFIRM_LIVE") != "yes"`, plus `LIVE MODE ENGAGED` warning + Telegram alert) is **deferred to a follow-up task** per §0.8 anti-hypothetical. Today:

- Strategy-engine reads `bot_config.exchange.mode` (T-310a `ExchangeSection.mode: Literal["live","testnet","paper"]`) but does NOT enforce the safeguard at startup.
- Execution-service reads `OrderRequest.exchange_mode` per-message but does NOT enforce.
- Telegram alerting is part of `services/alerting/` (F4+ scope per §3.1; service directory exists empty).

Without runtime enforcement, the env-var is dormant — operators can set `BOT_CONFIRM_LIVE=yes` and have it reach the container, but no code today checks it. T-311 ships the env wiring per brief §18.1 verbatim shape so a follow-up task can land the lifespan check in a single small diff (estimated ~15 LOC + tests). **This is a hazard-bound deferral**: if operator switches to `exchange.mode: live` before the runtime check ships, the live-mode safeguard is non-functional. Compose-level visibility (`BOT_CONFIRM_LIVE` in compose `environment:`) is documented operator-facing reminder; implementation to follow. Tracked as T-F3+ backlog item.

No new §20 hazards introduced. No regression of H-001..H-026.

## Open questions

**OQ-1 (resolved by reading brief):** Should compose use volume mount `./configs/bots:/app/configs/bots:ro` per BRIEF §9.4:1530 ("via mounted volume") or rely on Dockerfile-baked configs?

**Decision:** Dockerfile-baked (T-309 already COPYs configs/). Mounted-volume hot-reload is F5+ roadmap per §9.4:1547.

**TASKS.md deviation note**: TASKS.md line 108 (T-311 backlog stub) reads "mounted `configs/bots/` volume". §9.4:1547 ("Config is read only at startup in v1. Config change = restart container.") makes mount **functionally equivalent** to image-bake in F3 — both require container restart for config change. Image-bake is simpler (no volume management; configs are part of image immutability) and matches T-309 Dockerfile already-shipped contract. Operator may add `volumes: ["./configs/bots:/app/configs/bots:ro"]` to compose.dev.yaml strategy-engine-alpha/beta blocks for local hot-reload-via-restart workflow without rebuilding image (single-line override). Deferred to operator preference per §0.8 anti-hypothetical; F5+ true hot-reload (no restart) replaces both approaches.

**OQ-2 (resolved by reading brief):** Should compose ship 1 strategy-engine service or 2?

**Decision:** 2 (alpha + beta) per BRIEF §18.1:2374-2383 verbatim. T-313 E1 dvoj-bot test needs both. Adding more than 2 (gamma, delta...) is operator-driven post-§B.1 example.

**OQ-3 (resolved by reading T-309 Dockerfile):** Does Dockerfile already COPY plugins/?

**Decision:** Yes — T-309 Dockerfile lines 75-76 already `COPY configs/ /app/configs/` + `COPY plugins/ /app/plugins/`. T-311 just adds `configs/plugin_registry.yaml` to the existing COPY. Plugin Python source lands in T-312.
