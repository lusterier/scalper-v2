## Purpose

Ship the data structures + plumbing T-310b will consume to build `OrderRequest` payloads from `SignalValidated` + scoring decisions:

1. **BotConfig** gains `exchange`/`signals`/`execution` sections (§B.1 verbatim) so per-bot YAML carries the fields T-310b needs to populate `OrderRequest` (qty, leverage, sl_pct, tp_pct, tp_qty_pct, be_trigger, be_sl_level, trail_pct, exchange_mode).
2. **SignalRejected schema + subject** for §9.4 step 3f shadow-tracking publish on reject decisions.
3. **`select_signal_id_by_idempotency_key` query helper** so T-310b can resolve `signals.id BIGINT` from `SignalValidated.idempotency_key: str` (schema doesn't carry id; OQ-1 Path A).

T-310a is **pure infra** — no consumer body, no NATS subscribe, no scoring_evaluations write. Per §0.8 anti-hypothetical, every shipped surface is consumed by T-310b verbatim.

### `select_signal_id_by_idempotency_key`

```python
@idempotent
async def select_signal_id_by_idempotency_key(
    conn: _DbExecutor,
    *,
    idempotency_key: str,
    received_at_lower_bound: datetime,
) -> int | None:
    """Return ``signals.id`` for a previously-inserted webhook, or ``None``.

    Requires ``received_at_lower_bound`` for hypertable chunk pruning
    (Timescale needs the partitioning column in WHERE for chunk-skip).
    Caller passes ``now - max_signal_age_seconds`` (T-310b default 600s,
    configurable via Settings).

    The ``signals_idempotency`` UNIQUE index (migration 0002) provides
    O(log N) lookup; index is on ``(idempotency_key, received_at)``
    composite so the WHERE clause uses both.
    """
    row = await conn.fetchrow(
        """
        SELECT id FROM signals
        WHERE idempotency_key = $1
          AND received_at >= $2
        ORDER BY received_at DESC
        LIMIT 1
        """,
        idempotency_key,
        received_at_lower_bound,
    )
    return int(row["id"]) if row is not None else None
```

## Edge cases

- **Empty `signals:` block in YAML** — defaults to `SignalsSection()` (ttl 120, no filter). Tested.
- **Missing `exchange:` block in YAML** — Pydantic ValidationError surfaces at lifespan startup. Tested.
- **`source_filter: []` empty list** — accepted (empty whitelist = reject every source). T-310b consumer body will treat empty list per documented contract; T-310a only validates structure.
- **`qty` precision** — `Decimal("0.001")` via `_to_decimal` helper avoids `Decimal(0.001)` float-init footgun (existing T-308 pattern; reused).
- **`signal_id` lookup hot path** — DB hop per signal at scoring time. Each strategy-engine instance pays 1 round trip per signal — not bursty (signals are seconds-apart per H-008 TTL). The composite UNIQUE index ensures O(log N) lookup. F4+ may extend SignalValidated with `signal_id: int` to eliminate the hop (OQ-1 Option B); T-310a goes Path A per operator decision 2026-05-02.
- **`received_at_lower_bound`** — Timescale hypertable WHERE without partition column → full-scan all chunks. Lower-bound default 600s in T-310b; T-310a contract just enforces the parameter's presence in SQL.

### Hypertable lookup SQL — hand-traced

Plan: `SELECT id FROM signals WHERE idempotency_key=$1 AND received_at >= $2 ORDER BY received_at DESC LIMIT 1`. The composite UNIQUE index `signals_idempotency (idempotency_key, received_at)` per migration 0002:101 supports the `WHERE idempotency_key=$1 AND received_at >= $2` predicate via index range scan. ORDER BY DESC + LIMIT 1 is a top-K pattern Postgres + Timescale execute as index-scan-backwards; chunk pruning eliminates non-relevant chunks before the scan.

If multiple rows share the same `idempotency_key` (which shouldn't happen given the UNIQUE constraint, but the unique window is per-`received_at` — duplicate keys across far-apart `received_at` exist only if signal-gateway re-ingests same key after `signal_ttl_seconds`), the `ORDER BY received_at DESC LIMIT 1` returns the most recent — which is what T-310b wants (the in-flight signal).
