## Purpose

Two read+write endpoint groups for the dashboard's strategy editor + audit log sections:

1. **`/api/configs/*`** â€” bot config view (current + version history) + validate + apply (BRIEF Â§9.6:1630 + Â§14.3:2064). 4 endpoints. Apply is the **second admin write surface** in repo (after T-401b symbol_map CRUD); same atomic-tx contract per H-022 + Â§16.8.
2. **`/api/audit/*`** â€” audit log viewer (BRIEF Â§9.6:1631 + Â§14.3:2067). 2 endpoints â€” paginated chronological list + detail by id. Reads `audit_events` rows that T-401b symbol_map CRUD writes + T-405's own bot_config.apply writes.

T-405 also lands a small refactor of T-308 `yaml_loader.py`: extract `load_bot_config_from_string(yaml_text, *, plugin_registry)` so validate endpoint accepts raw YAML body without temp-file detour.

Per Â§0.8 anti-hypothetical:
- NO config rollback endpoint (POST /api/configs/{bot_id}/rollback) â€” operator can re-apply a historic version's YAML manually (idempotent path through validateâ†’apply); explicit rollback is opportunistic per UI demand
- NO config diff endpoint (server-side YAML diff between versions) â€” UI computes diff client-side from version detail responses
- NO YAML upload via multipart/form-data â€” JSON body with `yaml_text` field is sufficient
- NO audit log retention purge endpoint â†’ operator-driven SQL or F5+ scheduled job
- NO live SSE audit stream â†’ T-408
- NO export-audit-CSV endpoint â†’ opportunistic UI demand

T-405 is a deliberate single-task ship per OQ-7=B; LOC ratio +79% accepted with operator awareness. T-405a/T-405b split was offered but rejected because audit reader is already-fully-constrained by audit_events schema (T-401a shipped) and pairs naturally with configs writes that emit the rows.

**5-helper same-conn pin** (WG#2): `select_bot_config_current` + `select_max_bot_config_version` + `insert_bot_config` + `update_bot_config_applied` + `insert_audit_event` ALL must execute on one conn instance inside same tx. Test pin mirror T-401b WG#2 verbatim (`expected_conn = await_aenter_value(mock_pool)` + `assert mock.await_args.args[0] is expected_conn` Ă— 5).

`after_state` excludes `config_yaml` (TEXT, ~5-50 KB) â€” audit row stores only `id` + `bot_id` + `version` + `applied_at` + `applied_by` + `config_hash` + `notes` (7 fields, no config_yaml). Full YAML retrievable from `bot_configs` table by version. Reduces audit_events JSONB bloat. `before_state` applies same exclusion symmetrically.

**Concurrent apply race policy (WG#3)**: race is **detected, NOT prevented**. Two concurrent applies on same bot_id both compute `max_version=3 â†’ next_version=4`; first INSERT wins, second raises `asyncpg.UniqueViolationError` â†’ router catches â†’ 409 Conflict. Pre-read inside tx for read-after-write consistency on `before_state` (no `SELECT ... FOR UPDATE` â€” adds lock contention without eliminating race; UniqueViolation as belt-and-braces is sufficient).

**Validation failure path**: `load_bot_config_from_string(yaml_text)` raises `ValueError` BEFORE tx opens â†’ router catches â†’ 422 with `{"detail": str(e)}`. tx never opened, no rollback needed.

**bot_id URL-vs-body mismatch path**: if `parsed_config.bot_id != url_bot_id` â†’ 409 Conflict BEFORE tx opens. Defensive against accidental shadow-apply on wrong bot. Status code 409 (vs 422) chosen because the request body is structurally valid Pydantic â€” the conflict is between two well-formed names; semantic mismatch is closer to "resource conflict" than "schema validation".
