## Bug (production-surfaced, live-diagnostic attempt 2026-05-18, manual-test-006)

`POST .../v5/position/set-leverage 200 OK` then `bus_handler_failed correlation_id=manual-test-006 error="retCode=110043 retMsg=leverage not modified" subject=orders.requests.demo` — NO `/v5/order/create` (handler aborted at set-leverage).

Root cause code-confirmed: `client.py:188-193` raises on ANY non-zero retCode (`_translate_retcode` → `_RETCODE_RATE_LIMIT={10006,10016}`/`_AUTH={10003,10004}`/`_REJECT={10001,10005}` else `ExchangeError`). 110043 ∈ none → `ExchangeError("retCode=110043 retMsg=leverage not modified")` raised. `adapter.py:176-200 set_leverage` (`@idempotent`, LRU-cached) catches only `RateLimitError` → 110043 propagates → placement `bus_handler_failed` → no order.

**Bybit retCode 110043 = "Set leverage has not been modified"** — requested leverage already equals current; Bybit makes no change. Idempotent no-op = success-equivalent (desired state holds), **endpoint-unique** (Bybit emits 110043 ONLY for `/v5/position/set-leverage`). Universally benign in Bybit V5 integrations.

**Compounding defect (LRU cache never populated):** `set_leverage` sets `self._leverage_cache[key]=now` ONLY on the post-`try/except` line (success-only). 110043 → raise → cache never populated. In-memory cache lost on every execution-service restart → after any restart, first order per symbol → cache-miss → set-leverage call → exchange-side leverage already at target (persists on Bybit) → 110043 → fail. Steady state: every order on a symbol with leverage already set fails at set-leverage (first per symbol after each routine execution-service restart). Latent: paper has no set-leverage; FIRST real order per symbol set it fresh (retCode=0 — manual-test-004); every subsequent → 110043. Next never-run-live-path layer (alerting httpx → T-547 → T-548 → T-549b → T-550 → T-552 → T-553 → THIS).
