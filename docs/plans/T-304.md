# T-304: Composite conditions (and, or, not, when_then_else)

**Status**: DRAFT for plan-reviewer (2026-05-02)
**Phase**: F3 (fifth numbered F3 task)
**Trace**: F3 §10.2 lines 1703-1706 (4 of 14 condition types — boolean composition)

## Purpose

Land 4 composite condition variants per BRIEF §10.2 lines 1703-1706:

- `and`: all sub-conditions true
- `or`: any sub-condition true
- `not`: negation of single sub-condition
- `when_then_else`: ternary — evaluate `when`; if True, dispatch to `then` branch; else dispatch to `else_` branch

Each composite navigates sub-conditions via duck-typed `sub.evaluate(ctx)` (`@runtime_checkable Condition` Protocol from T-302). Sub-conditions are validated at construction via Pydantic `model_validator` checking `isinstance(sub, Condition)` — leverages T-302's runtime-checkable Protocol. Type-level: sub-condition fields are `BaseModel` (broadest valid Pydantic type that captures all condition classes); runtime-level: isinstance(Condition) check catches type errors at construct time.

After T-304, 14 of 14 §10.2 condition variants exist. T-305 will add plugin variant + registry; T-307 evaluator will compose `ConditionUnion = Annotated[Union[...], Field(discriminator="type")]` over all 14+.

class WhenThenElseCondition(BaseModel):
    """§10.2 ``when_then_else`` — ternary: evaluate `when`; dispatch to `then_` or `else_`."""
    model_config = ConfigDict(frozen=True, strict=True)

Pydantic field-name caveat: Python keywords `not`, `else` cannot be field names. The `type` discriminator string stays exactly per BRIEF §10.2 catalog (`"not"`, `"when_then_else"`); class fields use `condition` (for not) and `then_` / `else_` with trailing underscore (PEP-8 conflict-resolution convention) for when_then_else.

## Validation rules

1. **`AndCondition.conditions` / `OrCondition.conditions`** must be non-empty (`Field(min_length=1)`); operator-config error caught at construction.
2. **All sub-condition fields** validated via `model_validator(mode="after")` calling `isinstance(sub, Condition)` (T-302 `@runtime_checkable Protocol`). Failure → `ValueError` at Pydantic construct time (operator YAML config error).
3. **Short-circuit semantics**:
   - `and`: iterate `conditions`; on first False, return `(False, None)` without evaluating remainder. All True → `(True, None)`.
   - `or`: iterate `conditions`; on first True, return `(True, None)`. All False → `(False, None)`.
   - `not`: invert sub's outcome. `(True, _) → (False, None)`; `(False, _) → (True, None)`.
   - `when_then_else`: evaluate `when`; if outcome True → dispatch to `then_.evaluate(ctx)` and return its tuple; if False → dispatch to `else_.evaluate(ctx)` and return its tuple.
4. **Sub-condition error handling**: composite is **pure boolean combinator** — sub-condition `error_info` (dict) is **swallowed** at composite level. AND/OR/NOT treat sub's outcome as raw boolean regardless of error_info presence. T-307 evaluator's `on_error` policy operates at the TOP-level `rule.condition.evaluate` call, not at sub-level.
   - Edge case for `not`: `(False, {"error": ...}) → (True, None)`. The error's underlying state is "unknown / failed", but composite reports True per pure-boolean-inversion semantics. Operator who wants strict error propagation uses non-composite condition.
   - Edge case for `when_then_else`: `when` returns `(False, error) → dispatch to else_ branch`. Error is swallowed; behavior is well-defined per ternary.
5. All composites are `frozen=True, strict=True`.
6. **Recursive depth**: not bounded. Operator YAML may nest arbitrarily deep. Stack overflow at ~1000+ depth on default CPython is operator-config error (Pydantic doesn't enforce depth).

## Edge cases

- **Empty `conditions` list**: `AndCondition(conditions=[])` rejected at construction via `Field(min_length=1)` (same for Or).
- **Sub-condition NOT satisfying Condition Protocol**: e.g. `AndCondition(conditions=[42])` rejected at construction via `model_validator` isinstance check.
- **`not` with sub returning error**: returns `(True, None)` per pure-boolean inversion. Documented as known surprise; operator who wants strict error propagation avoids `not` on potentially-erroring sub.
- **`when_then_else` with `when` erroring**: dispatches to `else_` branch (boolean False after error swallow). Documented.
- **Deeply nested composite (100+ levels)**: works — Python recursion limit handles it. Beyond ~1000 levels, RecursionError (operator-config error; not a code bug).
- **Mixed condition types as sub-conditions**: an AndCondition can contain GtCondition + RisingCondition + nested OrCondition — all are valid Condition Protocol implementers per @runtime_checkable.

## Hand verification

T-304 has **boolean logic** — `and` short-circuit, `or` short-circuit, `not` invert, ternary dispatch. No arithmetic, no Decimal compare at this layer (delegated to sub-conditions). Math-validator gate is in scope (`packages/scoring/`) but content is pure boolean composition.

Hand-fixtures (per condition):

- **AndCondition([GtCondition(50000), LtCondition(60000)])** vs feature_value=Decimal("55000") → `(True, None)` (both subs True).
- **AndCondition([GtCondition(50000), LtCondition(60000)])** vs feature_value=Decimal("70000") → `(False, None)` (second sub False; first sub True). Short-circuit pin: first sub evaluated, second sub evaluated and returns False, AND returns (False, None).
- **AndCondition([GtCondition(50000), LtCondition(60000)])** vs feature_value=Decimal("40000") → `(False, None)` (first sub False; second NOT evaluated per short-circuit).
- **OrCondition([EqualsCondition(Decimal("100")), EqualsCondition(Decimal("200"))])** vs feature_value=Decimal("100") → `(True, None)` (first sub True; second NOT evaluated).
- **OrCondition([GtCondition(70000), LtCondition(30000)])** vs feature_value=Decimal("50000") → `(False, None)` (both False).
- **NotCondition(GtCondition(50000))** vs feature_value=Decimal("40000") → `(True, None)` (GtCondition False → NotCondition True).
- **NotCondition(GtCondition(50000))** vs feature_value=Decimal("60000") → `(False, None)`.
- **WhenThenElseCondition(when=GtCondition(50000), then_=LtCondition(60000), else_=GtCondition(40000))** vs feature_value=Decimal("55000") → when=True → dispatch then_ → 55000<60000 True → `(True, None)`.
- **WhenThenElseCondition(...)** vs feature_value=Decimal("65000") → when=True (>50000) → dispatch then_ → 65000<60000 False → `(False, None)`.
- **WhenThenElseCondition(...)** vs feature_value=Decimal("45000") → when=False → dispatch else_ → 45000>40000 True → `(True, None)`.
- **NotCondition(some_sub_with_error)** → if sub returns `(False, {"error": "feature_missing"})` → NotCondition returns `(True, None)`. Documented surprise.
- **AndCondition / OrCondition / NotCondition / WhenThenElseCondition** with non-Condition sub (e.g. raw dict, integer, string) → Pydantic ValidationError at construction.
- Recursive: `AndCondition([OrCondition([Gt(1), Lt(0)]), NotCondition(Equals(5))])` → 2-level nesting works.
