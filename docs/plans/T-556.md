## Problem (desk-diagnostic + Phase-1 live-diagnostic, empirically confirmed)

`services/execution/app/placement.py` step 6 calls `adapter.get_fill_price(symbol, exchange_order_id)` в†’ Bybit `/v5/execution/list?orderId=...`. On Bybit **demo** the execution record propagates to that endpoint with latency в‰€ **2.8 s** (measured 2026-05-18 via one operator-pre-authorized controlled demo order: `/v5/order/create` 200 OK `10:38:01.416` в†’ fill first available `10:38:04.211`; **7 GET attempts**, first 6 returned empty `list` в†’ `None`).

Shipped defaults `execution_fill_price_retry_attempts=3` / `execution_fill_price_retry_backoff_s=0.1` в‰€ 0.6 s window в†’ the loop gives up ~2.2 s before the fill appears в†’ `FillPriceUnresolvedError` в†’ DLQ в†’ orphan position (exchange has the position, DB persistence aborts mid-handler).

This is the **None-path** (empty `list`, fill genuinely not yet propagated) вЂ” **distinct** from T-216c / H-032 (transient *exception* path: `(AuthError, NetworkTimeout, RateLimitError)`), which stays byte-unchanged.
