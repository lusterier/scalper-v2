## Connection / transaction model + partial-failure semantics (Bod A — WG1/WG2)
ONE `async with pool.acquire() as conn:` per sub-account wrapping the ENTIRE per-sub-account work (the N×M `insert_funding_fee` fan-out + the 1 `insert_trading_event` emit), inside the per-sub-account `funding_fee_poll.persist_failed` try/except. **NO `async with conn.transaction():`** (deliberate). This is a REASONED divergence from the T-531 mirror's per-insert `pool.acquire()` (equity_snapshot.py:93) — documented in-code as a mechanically-required structural difference (T-527b2b `_resolve_sub_account`-relocation class, NOT a silent refactor): T-531 has ONE write-kind per bot → per-insert acquire natural; T-532b has a per-sub-account dual-write GROUP (N fan-out rows + 1 emit) → one acquire per sub-account is the correct grouping (avoids N×M+1 acquire churn).
Partial-failure (some funding_fees rows written, emit fails; or vice-versa) is ACCEPTED monitoring degradation, NOT a correctness bug — verbatim the documented T-531/ADR-0006 posture (funding_fees + the funding_settlement_window event are a MONITORING time-series; financial truth is the T-220 cumulative-delta audit, ADR-0006 H-017). SELF-HEALS next tick: the WINDOWED pull (window=10800 ≫ interval=300 → consecutive windows OVERLAP) re-pulls + re-emits a missed/partial settlement. Each `funding_settlement_window` event is a SELF-CONTAINED windowed snapshot (NOT additive across emits — verbatim T-220 `trade_pnl_deltas` cumulative-snapshot per ADR-0006: a reader reads the LATEST per sub-account, NEVER sums the emit series). The overlapping-window dup in raw funding_fees rows is documented append-trail noise (surrogate-id-distinct, T-531 "near-dup monitoring row" class; a cumulative reader uses the trading_events snapshot or DISTINCTs the rows — NEVER SUM(funding_fees) for truth). §N3: both writers @non_idempotent append-only, no retry (ADR-0007 D7) — consistent with the self-healing overlap.
