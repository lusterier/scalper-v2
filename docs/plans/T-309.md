## Public interface

### `Settings` (env-backed)

```python
class Settings(BaseSettings):
    model_config = SettingsConfigDict(extra="ignore")

    service_name: str = "strategy-engine"
    log_level: LogLevel = "INFO"
    http_port: int = 8000

    nats_url: str = "nats://nats:4222"
    database_url: str  # required, no default

    # T-309 skeleton — bot identity + config paths
    bot_id: str  # required; loads configs/bots/<bot_id>.yaml at startup
    bot_config_dir: str = "/app/configs/bots"
    plugin_registry_path: str = "/app/configs/plugin_registry.yaml"
```

**`bot_id` is required** (no default). Per BRIEF §9.4:1530 — "One Docker container per active bot, parameterized by `BOT_ID` env var." Missing → Pydantic ValidationError at lifespan startup, before uvicorn binds port (mirrors `database_url` precedent).

**`bot_config_dir` + `plugin_registry_path` defaults match Dockerfile WORKDIR + the standard mounted-volume convention from BRIEF §9.4:1530 + §10.6.** Configurable for tests + local dev.

### `create_app(settings: Settings | None = None) -> FastAPI`

Factory pattern. Mirrors execution-service / feature-engine verbatim. Composition split:
- **Sync primitives** (Settings, logger, prometheus registry) attached in factory body before lifespan runs.
- **Async resources** (pool, bus, plugin_registry, bot_config, FeatureResolver) attached in lifespan inside `async with`.

### Lifespan order (load-bearing)

```
1. plugin_registry = load_plugin_registry(Path(settings.plugin_registry_path))
2. bot_config_path = Path(settings.bot_config_dir) / f"{settings.bot_id}.yaml"
   bot_config = load_bot_config(bot_config_path, plugin_registry=plugin_registry)
3. pool = await create_pool(database_url, application_name="strategy-engine")
4. bus = NatsClient(servers=[nats_url], name=service_name, logger=logger)
   await bus.connect()
5. resolver = FeatureResolver(bus=bus, pool=pool, bound_logger=logger)
6. app.state attach: pool, bus, plugin_registry, bot_config, resolver
7. logger.info("service_started", http_port=..., bot_id=..., rules_count=len(bot_config.scoring.rules))
8. yield
```

**Reverse shutdown (load-bearing):**
```
9. await bus.close()  — bus first per T-200 Q2 publish-after-persist (T-310 will publish OrderRequest
                        post scoring_evaluations INSERT; bus must drain before pool closes so any
                        in-flight publish that touches the pool finishes against an open pool)
10. await pool.close()
11. logger.info("service_stopped")
```

**Plugin registry + bot config loaded BEFORE pool/bus** — both are pure file-I/O (YAML parse + Pydantic narrowing), no async resources held. Failure to load = service crash at startup, before NATS connection (fail-fast per L-001 / WG#7 of T-216a precedent).

### `/health`, `/ready`, `/metrics`

- `/health` — always 200, no deps (mirror execution-service verbatim)
- `/ready` — 200 only when bus.state==CONNECTED **and** pool.acquire(timeout=1.0) succeeds; else 503 with `reason="bus"|"db"` (precedence bus→db, mirror execution T-214 verbatim shape)
- `/metrics` — Prometheus scrape endpoint via `make_metrics_asgi_app` mount

T-309 ships **two** reasons: `bus`, `db`. **No third reason** for "scoring rules empty" or "bot config missing" — those crash the lifespan at startup, by definition the service cannot be `ready` without them.

## Edge cases

- **Missing `configs/bots/<bot_id>.yaml`** → `FileNotFoundError` from `Path.open()` at lifespan step 2 → service crash. Acceptable: bot misconfiguration must surface immediately, not at first signal.
- **Missing `configs/plugin_registry.yaml`** → `FileNotFoundError` from `load_plugin_registry`. Same rationale.
- **Empty plugin registry** (`rules: []`) → returns `{}`, valid (no plugin conditions in bot YAML) → bot loads cleanly. Already tested in T-305.
- **Bot YAML uses plugin condition not in registry** → `ValueError` from `parse_condition` → service crash. Already covered by T-308.
- **Empty `bot_config.scoring.rules`** → ScoringConfig allows `[]`; T-310 will handle (every signal → passthrough/no-execute). T-309 just attaches.
- **Pool acquire timeout in `/ready`** — 1.0 s ceiling per execution T-214 precedent.

## Open questions

**OQ-1 (resolved by reading BRIEF):** Should T-309 attach `bot_config` + `plugin_registry` + `resolver` to `app.state`, or load them lazily on first signal in T-310?

**Decision: load at startup, attach to app.state.**

Rationale:
- BRIEF §9.4:1546-1547 — "Config is read only at startup in v1. Config change = restart container." → eager load aligns with v1 contract.
- Fail-fast on bad config = standard pattern (BRIEF §5.11 + execution Settings precedent).
- T-310 needs zero re-discovery; consumer body just reads from app.state.

**OQ-2 (resolved by reading BRIEF):** Does `/ready` need a third reason for "bot config not loaded" or "rules empty"?

**Decision: no.** Per analysis under "Public interface > `/health`, `/ready`, `/metrics`": those errors crash the lifespan, by which point `/ready` itself isn't reachable. Two reasons (bus, db) is sufficient.

**OQ-3 (resolved by reading T-308 source):** Does `load_bot_config` accept a `pathlib.Path` or `str`?

**Decision: `Path`.** Confirmed via `packages/scoring/yaml_loader.py:332` signature `path: Path`. Lifespan code constructs `Path(settings.bot_config_dir) / f"{bot_id}.yaml"`.

**OQ-4 (resolved by reading T-301):** Does T-309 need to add an `insert_scoring_evaluation` deps provider for T-310?

**Decision: no, T-309 doesn't add it.** Per §0.8 anti-hypothetical, T-310 owns the consumer body and will add the deps provider in its plan if needed. T-309 ships the 5 standard providers (pool, bus, settings, logger, **plus bot_config**); the bot_config provider is justified because BRIEF §9.4 main-loop step 3 reads scoring rules per signal — every code path in T-310 will need it, so adding it now in the deps surface keeps T-310's plan smaller. **Defer adding `get_resolver` provider to T-310** since the resolver is invoked from the consumer body (one call site), not a request handler.

## Hand verification

T-309 is pure skeleton — no financial math, no Decimal arithmetic, no ratio computation. Math-validator will return `VERIFIED — out of scope` immediately.

The lifespan ordering itself has one verifiable property:

**Claim:** Plugin registry must load BEFORE bot config (load_bot_config(plugin_registry=...)).

**Verification:** Trace the call:
1. `plugin_registry: dict[(str, str), type[Rule]] = load_plugin_registry(...)`
2. `bot_config: BotConfig = load_bot_config(path, plugin_registry=plugin_registry)`

Step 2 passes the registry from step 1 as kwarg. If a bot YAML uses a `plugin` condition, `parse_condition` looks up `(name, version)` in this dict (per `yaml_loader.py:_parse_plugin`). Reversed order is invalid Python (NameError on `plugin_registry`). → **Test #7 (`test_lifespan_loads_plugin_registry_first`) pins this via call_log assertion.**

**Claim:** Bus must close BEFORE pool on shutdown.

**Verification:** Per T-200 Q2 publish-after-persist: T-310 will publish `OrderRequest` post `scoring_evaluations` INSERT (pool query). If pool closed first, in-flight publish referencing pool state would race against pool teardown. Standard precedent: execution main.py:306-329 + feature-engine main.py:177-180. → **Test #8 (`test_lifespan_closes_bus_before_pool`) pins this via call_log + side_effect.**
