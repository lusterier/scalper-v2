### L-011 forward-pointer for T-510b / T-511 / T-513 write-helper owners

When adding `insert_shadow_variant` / `insert_shadow_rejected` / `update_*_terminal` helpers, **pass Python dicts directly** to asyncpg for the `meta` JSONB binding — no `json.dumps` at call site. Reason: shadow.py write helpers run under **execution-service** which currently does NOT register `_register_jsonb_codec` (per L-011 grep — only analytics-api + feature-engine register the codec); so today they're **safe-by-accident** (single-encoding via SQL `$N::jsonb` cast on text payload works). **BUT** if execution-service later adds codec registration in F5+ for any reason (e.g., reading scoring_evaluations.feature_snapshot dict-shape via the codec), the write helpers would silently double-encode just like audit_events did at F4 E1 smoke. Pre-emptive convention: use `_to_jsonable(value)` helper from `packages/db/queries/audit.py` (or sibling) to pre-stringify UUID/datetime/Decimal in the dict before binding. This way the codec encoder sees only JSON-native types regardless of registration state.

### Indexes (4 btree, 2 per table)

- `shadow_variants_parent` — `(parent_trade_id)` — drill-down "all variants for trade X" (T-516 reader).
- `shadow_variants_bot_active` — `(bot_id) WHERE terminated_at IS NULL` — partial index for restart-recovery scan (T-512 reads active variants per bot).
- `shadow_rejected_signal` — `(signal_id)` — lookup by source signal.
- `shadow_rejected_bot_active` — `(bot_id) WHERE terminated_at IS NULL` — partial index for 60-min observation task restart-recovery (T-513).

NOT a hypertable for either table per OQ-2=A — low volume (<500 variants/day per single operator at typical trade frequency; <100 rejected signals/day).

### Spec-clarification — §7.2 header deviation justified

BRIEF §7.2 lines 1083 + 1087 mark both shadow tables as `(hypertable)` in section headers (e.g. *"`shadow_rejected` — tracking of rejected signals (hypertable)"* / *"`shadow_variants` — parallel rule simulations (hypertable)"*). These section headers are stub-pointers to §13 and contain no DDL. **§7.2 line 838 (the authoritative hypertable-list) does NOT include `shadow_variants` or `shadow_rejected`.** This plan mirrors the T-501 `backtest_trades` precedent (which also resolved the same §7.2-header-vs-line-838 discrepancy in favor of NOT-hypertable + OQ-2=A volume rationale): expected volume <500 variants/day + <100 rejected/day produces <50k total rows in 1 year — well below TimescaleDB chunking threshold. If F5+ runtime measurement reveals volume exceeding capacity, hypertable promotion is a forward-only optimization migration in F5+ (no T-510a regression). Plan-doc captures this deviation explicitly so drift-checker / brief-reviewer in future tasks reading §7.2-headers don't flag spec-drift.

### FK cascade behavior

`shadow_variants.parent_trade_id` → `trades(id) ON DELETE CASCADE` per OQ-3=A. `trades` is a regular table with simple `id` PK (`migrations/versions/0005_orders_trades_executions.py:124-130`); FK + cascade are clean.

`shadow_rejected.signal_id` is **plain BIGINT NOT NULL with no FK** per OQ-6=A. Rationale: `signals` is a hypertable with composite PK `(received_at, id)` (`migrations/versions/0002_signals_hypertable.py:91`); PostgreSQL rejects FK creation on `id` alone (no UNIQUE constraint on the column). All 4 existing tables holding `signal_id` reference (`0005_orders_trades_executions.py`:96+137 / `0008_paper_tables.py`:77+118 / `0010_scoring_evaluations.py`:55 / `0013_backtest_trades.py`:59) deliberately omit the FK; `0010` carries an inline comment *"No FK on bot_id / signal_id — audit-stream stands alone"*. T-510a follows this convention. Orphan-row risk is theoretical: signals are hypertable-retention-policy-driven (not operator-deletable surface), and a 60-min observation window outlives only the most aggressive retention setting. Composite-FK alternative (option B) was rejected as adding +1 column + +1 INSERT-side write-time lookup with no precedent in repo.
