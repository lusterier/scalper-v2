## Purpose

Read-heavy REST + SSE service that will, in T-401..T-408, expose paginated PG queries to the dashboard UI per BRIEF §9.6. T-400 ships **only the skeleton**: composition root + lifespan + `/health`/`/ready`/`/metrics`. Per §0.8 anti-hypothetical, no `/api/*` endpoints, no SSE backplane, no NATS subscribe — those land in their owning tasks (T-401..T-408).

Service is a **single-instance singleton** (BRIEF §2.2 line 234 — "1 instance"), not per-bot like strategy-engine. No `BOT_ID` env var.

## Public interface

### `Settings` (env-backed)

```python
class Settings(BaseSettings):
    model_config = SettingsConfigDict(extra="ignore")

    service_name: str = "analytics-api"
    log_level: LogLevel = "INFO"
    http_port: int = 8000

    nats_url: str = "nats://nats:4222"
    database_url: str  # required, no default
```

**5 fields total** — strictly less than T-309 (T-309 has 8 fields incl. `bot_id`, `bot_config_dir`, `plugin_registry_path`, `signal_max_age_seconds`). T-400 has no per-bot identity (singleton) and no scoring config paths.

`extra="ignore"` tolerates sibling-service env vars (`BOT_ID`, `MARKET_DATA_SYMBOLS`, `BOT_<ID>_BYBIT_API_KEY`, …) in `.env.example` / `secrets.env` / `compose` env_file.

### `create_app(settings: Settings | None = None) -> FastAPI`

Factory pattern. Mirrors strategy-engine / execution-service / feature-engine verbatim. Composition split:
- **Sync primitives** (Settings, structlog logger, Prometheus registry) attached in factory body before lifespan runs.
- **Async resources** (pool, bus) attached in lifespan inside `async with`.

### Lifespan order (load-bearing)

```
1. pool = await create_pool(database_url, application_name="analytics-api")
2. bus = NatsClient(servers=[nats_url], name=service_name, logger=logger)
   await bus.connect()
3. app.state attach: pool, bus
4. logger.info("service_started", http_port=...)
5. yield
```

**Reverse shutdown (load-bearing):**
```
6. await bus.close()  — bus first per T-200 Q2 publish-after-persist convention.
                        T-400 ships no publishers, but T-408 SSE handler will fan out
                        NATS messages while in-flight; closing bus first drains those
                        before pool teardown so any future publish-after-DB-read path
                        completes against an open pool. Inheriting the convention from
                        T-309 / execution main.py:306-329 / feature-engine main.py:177-180.
7. await pool.close()
8. logger.info("service_stopped")
```

### `/health`, `/ready`, `/metrics`

- `/health` — always 200, no deps (verbatim T-309 mirror)
- `/ready` — 200 only when `bus.state == CONNECTED` **and** `pool.acquire(timeout=1.0)` succeeds; else 503 with `reason="bus"|"db"` (precedence bus → db; verbatim T-309 mirror)
- `/metrics` — Prometheus scrape endpoint via `make_metrics_asgi_app` mount (verbatim T-309 mirror)

T-400 ships `bus` + `db` reasons. **No third reason** — analytics-api skeleton has no other deps.

## Edge cases

- **Missing `DATABASE_URL`** → Pydantic ValidationError at `Settings()` construction → service crash before uvicorn binds. Acceptable: misconfiguration must surface immediately.
- **NATS unreachable at startup** → `bus.connect()` raises → lifespan crash. Acceptable per fail-fast convention.
- **Pool acquire timeout in `/ready`** — 1.0 s ceiling per T-309 / T-214 precedent.
- **Bus disconnect mid-runtime** → `/ready` flips to 503 reason=bus until reconnect; container stays up (still serves `/health` + `/metrics`).
