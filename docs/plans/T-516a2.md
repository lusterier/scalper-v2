## Edge cases

1. **404 on drill-down (paper trade not found)** — same shape as `trades.$tradeId.tsx:64-76`: detect 404 via `apiFetch` error message substring, render "Paper trade #{id} not found" + Back link.
2. **`signal_id` = null** — paper trade with manual or reconcile-driven signal-less ancestry; signal + scoring queries gated by `signalId !== null && signalId !== undefined` (mirror `trades.$tradeId.tsx:49-51`); placeholder text "No signal (manual or reconcile-driven trade)".
3. **Zero results in paginated list** — DataTable `emptyMessage="No paper trades match filters"`.
4. **`status === "error"`** — backend `TradeStatus` allows `error` status; UI status filter includes `"error"` option same as live.
5. **`exit_price` = null on open paper trade** — TradeSummary renders "—" via `trade.exit_price ?? "—"` (mirror `trades.$tradeId.tsx:153`).
6. **`closed_at` = null on open paper trade** — backend ORDER BY `closed_at DESC NULLS FIRST` (per analytics-api router); list renders `formatUtcDateTime(trade.closed_at)` only when non-null else "—".
7. **Discriminated union prop drift risk** — if backend `PaperTradeResponse` ever diverges from `TradeResponse` (e.g., paper-only field added in F6+), the union prop becomes lossy. Mitigation: api-types definition stays as TWO distinct interfaces (no `type PaperTrade = Trade` alias) so divergence triggers TypeScript compile errors at usage sites; component prop accepts the broader union shape but reads only the 12 fields rendered in TradeSummary (no field-by-field destructuring beyond rendered subset).
