# Review lessons

Patterns the review system has caught. Each lesson encodes a watch-out point that reviewers should now check explicitly going forward. Reviewers consult this file at start of every invocation.

Format:
```
## L-NNN (T-XXX, <reviewer> <verdict>, <date>)
Pattern: <one-sentence description>
Active control: <what to check>
```

---

## L-001 (T-105, brief-reviewer SHIP nice-to-have, 2026-04-26)
Pattern: Polling intervals, pause durations, and similar hardcoded numbers in business logic violate §N9 (configurability invariant) even if they are sensible defaults.
Active control: At plan-reviewer stage, scan the plan for any numeric literals controlling timing, pacing, or periodic behavior. Require Settings/env exposure unless the value is mathematically fixed (e.g., "alpha = 2/(period+1)" formula constants).

## L-002 (T-107b, plan-reviewer REVISE, 2026-04-26)
Pattern: `name_template` literals in indicator implementations may drift from §B.2 spec when the indicator's `__init__` exposes a parameter that is hardcoded in the spec literal (e.g., `vwap_session` — session is fixed in spec, parameterizing it in the template breaks naming convention).
Active control: At plan-reviewer stage for any task touching `packages/features/builtins/`, explicitly check each indicator's `name_template` literal against §B.2 verbatim. Parametrized substitutions (`{symbol}`, `{interval}`) are universal; instance-baked parameters in the template (period, std_dev, etc.) are only valid where §B.2 shows them.

## L-003 (T-107b, plan-reviewer CONCERN, 2026-04-26)
Pattern: Helper functions extracted from indicator code (e.g., `_sma_seeded_ema_series`) need explicit alignment documentation showing they produce the same values as the standalone indicator class on the same input. Without this, a math drift between two implementations of the "same" computation can silently break golden tests.
Active control: At plan-reviewer stage, when a plan introduces a helper used by multiple indicators or by an indicator-internal computation, require a "golden cross-check" test in the plan that asserts helper output equals standalone indicator output on identical input.

## L-004 (T-107b, plan-reviewer CONCERN, 2026-04-26)
Pattern: `value_json` in FeatureValue is `Mapping[str, object]` (per T-106 contract) — this means dict literals work but the resulting FeatureValue is **not** hashable since dict isn't hashable. Plans involving `value_json` outputs should explicitly note this in the test strategy so a hash-check test exists.
Active control: At plan-reviewer stage, for any indicator producing `value_json`, require a test verifying that FeatureValue with `value_json` payload behaves correctly with respect to the documented `Mapping[str, object]` contract.

## L-005 (T-108, plan-reviewer REVISE, 2026-04-26)
Pattern: `sa.Float()` in SQLAlchemy/Alembic produces PostgreSQL `real` (4-byte single precision), NOT `DOUBLE PRECISION` (8-byte). Use `sa.Double()` for `DOUBLE PRECISION` columns. Spec literal `DOUBLE PRECISION` (e.g., §7.2 line 907 for features.value_num) requires `sa.Double()`, otherwise the migration silently degrades precision and integration tests of column type will fail.
Active control: At plan-reviewer stage, for any migration whose plan specifies `DOUBLE PRECISION`, verify the SQLAlchemy type is `sa.Double()`, not `sa.Float()`. At brief-reviewer stage, grep staged migration code for `sa.Float()` and flag if any `DOUBLE PRECISION` columns exist in the same migration.

## L-006 (general, observed across T-105/T-107a/T-107b, 2026-04-26)
Pattern: LOC estimates from Claude Code's plan drafts are systematically 20-40% optimistic for integration-heavy tasks (T-105 backfill: 240 → 344 = +43%; T-107b indicators: 216 → 280 = +30%). Skeleton or scaffold tasks (T-106, T-107a after split) match estimates more closely.
Active control: At plan-reviewer stage, if a plan's LOC estimate exceeds 280 LOC for an integration-heavy task, recommend pre-emptive split. If a plan's LOC estimate is 200-280 LOC, note it as "watch for cap pressure mid-write" but allow.

## L-007 (T-107, operator decision pre-plan-reviewer, 2026-04-26)
Pattern: Mid-write splits (Claude Code reaching 400-LOC cap mid-implementation and forced to split) are operationally expensive — they require git history rewrite, branch creation, and disrupt the implementation flow. Pre-emptive splits at planning time are far cheaper.
Active control: At plan draft stage, if the plan describes 5+ distinct components (e.g., 6 indicators) or has explicit "if I exceed cap mid-write I'll split" contingency language, recommend pre-emptive split into two tasks before invoking plan-reviewer.

## L-008 (T-218a, plan-reviewer REVISE BLOCKER #1, 2026-05-01)
Pattern: Plan-doc SQL literals using Python type names (`Decimal '0'`, `Datetime '...'`, `Bool 'true'`) are NOT valid PostgreSQL syntax — PG knows `numeric` / `timestamp` / `boolean` (lowercase, optionally with explicit cast `::numeric` or typed-literal `NUMERIC '0'`). The Python `decimal.Decimal` class name is not a PG type identifier and the SQL parser will raise `syntax error at or near "Decimal"` at runtime. Mock-level tests don't catch this because they intercept `conn.execute` before SQL is parsed.
Active control: At plan-reviewer stage, grep plan-doc + WG section for SQL literals containing Python type names (`Decimal '`, `Datetime '`, `Bool '`, etc.) — flag as BLOCKER. Require either implicit cast (bare `0` for NUMERIC, bare `true` for boolean), explicit cast (`0::numeric`, `'2026-01-01'::timestamptz`), or typed literal (`NUMERIC '0'`, `TIMESTAMPTZ '...'`). Additionally require a DB-level integration test pin against testcontainer PG when the helper exercises a non-trivial SQL expression (COALESCE/CASE/CAST), gated on `POSTGRES_TEST_DSN` env var per existing F1 integration pattern, otherwise mock-only tests pass while real path explodes.

## L-009 (T-409, ci-full pip-audit failure post-merge, 2026-05-04)
Pattern: Local pre-commit hooks do NOT run pip-audit; only ci-full does. New deps pinned at plan-time may carry unfixed CVEs (jinja2==3.1.4 had 3 CVEs with patches in 3.1.5 + 3.1.6 published before T-409 plan was drafted). Plan-reviewer Gate 1 + brief-reviewer Gate 3 + math-validator Gate 4 don't audit dep CVEs — gap surfaces only on master after CI-full run (~2 min latency post-push), requiring follow-up `fix(T-NNN)` commit on master.
Active control: At plan-reviewer stage, when a plan adds a NEW external dep (per §0.9), require explicit "verified latest patch version with no open CVEs at plan time" sentence in §0.9 justification — Claude Code can run `uv pip install --dry-run "<pkg>"` + `pip-audit --dry-run` locally before APPROVE. At brief-reviewer stage, run `uv run pip-audit --skip-editable` on staged uv.lock if any new dep added; reject SHIP if any unresolved CVE reported. Local pre-commit can't add pip-audit hook (~5s slow + network) without operator opt-in; gate is reviewer-side.

## L-010 (T-420, brief-reviewer FIX FIRST, 2026-05-05)
Pattern: Shared fetch wrapper that always calls `res.json()` will silently break the first 204/empty-body endpoint, while mocked tests pass because they short-circuit the json-parse path. T-420 introduced `DELETE /api/symbol-map/{id}` which backend returns 204 + empty body (per `services/analytics_api/app/routers/symbol_map.py:245`); `apiFetch` `return res.json()` would throw SyntaxError on empty payload, mutation falls into onError, query invalidation NEVER fires, deleted row stays visible — operator believes delete failed while backend actually deleted. Tests passed because they mocked `apiFetch` directly, bypassing `fetch`+`res.json()` path.
Active control: When introducing a NEW DELETE / PUT-no-body / 202-Accepted-empty mutation in a UI task, brief-reviewer MUST verify the fetch wrapper handles 204 + empty Content-Length BEFORE res.json() — short-circuit with `if (res.status === 204 || res.headers.get("content-length") === "0") return undefined as T;`. Plan-reviewer should flag this preemptively when reviewing plans that introduce DELETE/PUT-no-body. Test must exercise REAL apiFetch via fetch-level mock (`vi.fn() as typeof globalThis.fetch` returning `new Response(null, {status: 204})`), not apiFetch-level mock — apiFetch-level mocks bypass the json-parse trap entirely.

## L-016 (T-512a, drift-checker DRIFT pass-1 → SHIP via operator §0.3 over-cap waiver, 2026-05-08)
Pattern: Replay-recovery / restart-resume FSM tasks (shadow / rejected-signal / future task-restart) systematically under-budgeted at plan-time by ~150-180% — even after pre-emptive split per L-007. T-512a plan-reviewer pass-2 APPROVE budget was ~225 LOC src; reality post-implementation was ~625 LOC src (~155% over §0.3 400 cap; ~180% over plan target). Distribution: per-variant defensive parent-state check both modes (~25 LOC) + WG#5 window cap pre-check (~20 LOC) + WG#3 timer carry-over pre-check (~25 LOC) + WG#6 per-variant compute timeout wrap with diagnostic logging fields cursor_rows_consumed/last_bucket_start/elapsed_compute_ms/elapsed_io_ms (~50 LOC) + WG#11 replay variant_state seed for live continuation tracking (~15 LOC) + WG#12 own_sub subscribe live-continuation closure (~30 LOC) + WG#18 cascade-delete-race log format (~15 LOC) + structured logging events module-level (~50 LOC) + module docstring + imports + helpers (~100 LOC). All scope-aligned per drift-checker DRIFT verdict — no scope creep, plain calibration miss. 2nd data point with T-511b1 (shadow-worker FSM 70-80% over) confirms L-014 multiplier applies to ALL replay/restart-recovery FSM tasks, not just per-variant FSM body.
Active control: Plan-reviewer at gate 1 MUST flag any plan with budget <500 LOC src for replay-recovery / restart-resume FSM tasks (shadow restart per BRIEF §13.4, rejected-signal restart per §13.5, future signal-replay or risk-engine FSM tasks) as optimistic. Multiplier guidance for replay-recovery: enumerate plan's WG/AC items + count "defensive paths" (parent-state checks per mode, window cap, timer carry-over, per-variant timeout, structured logging diagnostic fields, cascade-delete race logging, live-continuation closure, replay state seeding); each defensive path contributes 15-30 LOC realistic. Specifically: if plan enumerates ≥2 mode-specific dispatches (live + paper) + ≥3 defensive cap/timeout/window pre-checks + ≥1 diagnostic logging event with structured fields + ≥1 nested closure for live-continuation, realistic budget is 500-700 LOC src for the replay module alone, NOT 150-225. Brief-reviewer at gate 3 MUST verify any §0.3 over-cap is accompanied by explicit operator waiver in commit msg per L-014 active control. Drift-checker at gate 2 MUST distinguish "scope drift" (DRIFT verdict — operator must REVISE) from "plan-budget calibration miss" (DRIFT verdict but operator may waive — replay-recovery pattern is the canonical case for this).

## L-015 (T-511b2a follow-up, ci-full integration regression post-merge, 2026-05-08)
Pattern: When a NEW migration modifies a table introduced in an earlier migration, the integration tests for the earlier migration (which run against ``migrated_db_dsn`` fixture upgraded to head) inherit the new schema state. Tests asserting old column shape, INSERT statements with NOT NULL columns, or FK behaviour will fail at head. T-511b2a's migration 0015 added ``parent_kind: TEXT NOT NULL`` to ``shadow_variants`` (introduced in 0014); 3 tests in ``test_0014_migration.py`` failed in ci-full: column shape (14 → 15 cols), 2 INSERTs (``NotNullViolationError`` on parent_kind), FK cascade test (FK dropped in 0015 → no cascade fires; original assertion expected ``after == 0``). Local pre-commit cannot catch this — integration migration tests are env-gated on ``POSTGRES_TEST_DSN`` and only run in ci-full (~2 min latency post-merge). L-009 sibling: ci-full-only checks surface failures post-merge requiring follow-up ``fix(T-NNN)`` commits.
Active control: When a plan adds a migration that modifies a table introduced in an earlier migration (drop column / add NOT NULL / drop FK / type change), plan-reviewer at Gate 1 MUST require a "Sibling migration test impact" section listing every earlier ``test_NNNN_migration.py`` whose assertions touch the modified table + the specific assertion that needs updating (column shape tuple / INSERT bind args / FK cascade behaviour). Brief-reviewer at Gate 3 MUST grep staged diff for ``test_NNNN_migration.py`` files matching the modified table's earlier migration + verify the relevant assertion lines match post-head state. If sibling test changes are not in scope of the migration task, plan must explicitly defer with a "Follow-up sibling test fix anticipated" note + estimate of ci-full latency. Math-validator + drift-checker do NOT catch this (env-gated tests skip locally). The plan-stage gate is the most reliable prevention point.

## L-013 (T-510b + T-520 cherry-pick — pre-emptive `_to_jsonable` wrapper as codec-state-immune JSONB-writer convention, 2026-05-07)
Pattern: F4 E1 audit_events double-encode regression (L-011) + audit_events `Object of type UUID is not JSON serializable` regression (intermediate `c241c15` fix) showed that `packages.db.queries.*` write helpers binding `$N::jsonb` parameters are sensitive to two orthogonal failure modes — (a) raw Python dict + asyncpg-without-codec → `InterfaceError: invalid input for query argument`; (b) pre-`json.dumps`-encoded text + asyncpg-WITH-codec → double-encode (string-scalar JSONB stored). The original L-011 active control split convention by per-service codec registration state (analytics-api / feature-engine register codec → pass dicts directly; signal-gateway / strategy-engine do not → use `json.dumps`), creating cross-service drift risk: when a service flips codec-registration state in F5+, all its existing JSONB writers silently flip into the corresponding bug class. T-510b (shadow.py for execution-service) + T-520 cherry-pick (signal_gateway.insert_signal) shipped a *codec-state-immune* convention: every JSONB writer wraps the dict via `_to_jsonable(value)` (recursive UUID/datetime/Decimal → str pre-stringification — defined at `packages/db/queries/audit.py:55`); the outer `json.dumps(...)` wrapper is added only when the consuming service does NOT register the codec. The `_to_jsonable` wrapper is the constant element across both modes; the `json.dumps(...)` wrapper is the codec-state-conditional bit.
Active control: When introducing or modifying any JSONB writer in `packages.db.queries.*` that binds `$N::jsonb` parameters, MUST: (1) wrap the dict via `from packages.db.queries.audit import _to_jsonable  # noqa: PLC2701` (option (c) explicit private-import per T-510b operator default A; alternatives extracted-shared-module / promoted-public flagged in plan-doc); (2) For services that do NOT register `_register_jsonb_codec` (signal-gateway / execution-service / strategy-engine today): emit `json.dumps(_to_jsonable(payload))` as the bind value. (3) For services that DO register codec (analytics-api / feature-engine today; grep `_register_jsonb_codec`): pass `_to_jsonable(payload)` dict directly to asyncpg; codec encoder serialises via internal `json.dumps`. (4) Module docstring MUST document the switch trigger when consumer service flips codec-registration state. Brief-reviewer must reject any `json.dumps(payload)` (raw, no `_to_jsonable` wrapper) on JSONB writers — pre-emptive wrapper is the standard convention going forward, regardless of current codec state. Plan-reviewer must surface this convention as Write-time guidance for any new task introducing a JSONB writer in `packages.db.queries.*`.

## L-012 (T-501 / T-505 ci-full regression — relative downgrade target breaks across head changes, 2026-05-07)
Pattern: Migration integration tests using `alembic downgrade -1` (relative target) silently break across master when a NEW migration lands — the relative target now points one revision later than the test author intended. T-501 added migration 0013 → `alembic head` shifted from 0012 to 0013 → `test_migration_0012_downgrade_preserves_pgcrypto` (which calls `downgrade -1` after `upgrade head` expecting to roll back 0012 → 0011) actually rolled back 0013 → 0012, leaving `backtest_runs` undropped, and the assertion `assert table_row is None` failed in master ci-full ~2 min after the chore(tasks) commit landed (`c11c767` ci-full failure). Local pre-commit only runs unit tests; integration migration tests are env-gated on `POSTGRES_TEST_DSN` and don't run in pre-commit, so the regression surfaced post-merge with no warning at brief-reviewer time. T-501 own test_0013 had the same latent fragility (its `downgrade -1` works today because head==0013 and target==0012, but would break when migration 0014 lands).
Active control: When writing migration integration tests that exercise `alembic downgrade`, ALWAYS use an explicit revision target (`downgrade <revision_id>`, e.g. `downgrade 0011`) instead of relative `-N` syntax. The test's intent is "rollback exactly this migration"; explicit target encodes intent and is robust against future migrations changing `head`. Plan-reviewer must flag any new migration test using `downgrade -1` / `downgrade -N` as a BLOCKER and require explicit target. Brief-reviewer must grep new migration test files for `downgrade.*-[0-9]` patterns. The fix in T-501/T-505 cohort retroactively rewrites both `test_0012_migration.py` (relative -1 → explicit 0011) + `test_0013_migration.py` (relative -1 → explicit 0012) for consistency. Future task `test_0014_migration.py` MUST start with explicit `downgrade 0013` from day one.

## L-011 (F4 E1 smoke double-encode under registered JSONB codec, 2026-05-07)
Pattern: When the connection has a registered JSONB codec (`conn.set_type_codec("jsonb", encoder=json.dumps, decoder=json.loads)` — present in analytics-api + feature-engine lifespans per T-401a / T-110d), pre-serialising a dict with `json.dumps(...)` BEFORE passing it to asyncpg as a `$N::jsonb` parameter triggers DOUBLE encoding: the codec re-runs `json.dumps` on the already-stringified value, storing a JSON string scalar (escaped `"{\"id\":1,...}"`) instead of a JSONB object. Read-back through the codec's `json.loads` decoder returns a string, which Pydantic dict-coercion then drops to `null` in the API response — so the reader sees empty `before_state`/`after_state` while the DB *does* contain (corrupted) data. Surfaced live during F4 E1 smoke at two layers: (a) `POST /api/backtests/` 500 when `_audit_state_dict(BacktestRunRow)` returned a dict whose UUID/datetime fields tripped the unguarded `json.dumps` (no `default=`); (b) `bot_config.apply` audit row showing `after_state: null` in the audit-log viewer because of the double-encode. Sibling `packages/db/queries/signal_gateway.py:110` uses the same pre-serialise pattern but its caller (signal-gateway service) does NOT register the codec, so single-encoding works there — the audit helper inherited the pattern from a different-service precedent and broke under analytics-api's codec. Existing mock-based tests captured the encoded string and assumed it round-tripped correctly; they never simulated the codec's second pass.
Active control: For any JSONB writer in `packages/db/queries/` that runs under a service registering `set_type_codec("jsonb", encoder=json.dumps)` (analytics-api, feature-engine — grep `_register_jsonb_codec`), pass Python dicts DIRECTLY to asyncpg (no `json.dumps` at the call site); pre-convert UUID / datetime / Decimal in the dict via a `_to_jsonable(v)` helper so the codec encoder sees only JSON-native types. Plan-reviewer must flag any new `audit_state_dict` / `payload_dict` helper that pipes into `$N::jsonb` and require: (1) call site passes dict, not `json.dumps(dict)`; (2) regression test that captures `bind_args` and asserts `isinstance(captured, dict)` (not str), AND that emulates the codec by calling `json.dumps(captured)` to prove no `TypeError`. Brief-reviewer must reject `json.dumps(state) if state is not None else None` patterns in JSONB-writer query helpers — even with `default=str` they double-encode under registered codec. Sibling `signal_gateway.insert_signal` is currently safe-by-accident (no codec on signal-gateway service) but should switch to the same pattern if/when the service registers a codec — flagged as latent at L-011 publish time.

## L-017 (fix(T-218b-open-fill-qty-bug), operator-discovered shipped-code bug, 2026-05-08)
Pattern: Test fixtures using artificial "post-update" mock values to bypass downstream state-checks can hide bugs in the pre-update logic. `test_process_open_fill_orders_lookup_to_open_branch` (test_dispatcher.py:366-386 pre-fix) used `_ps_row(remaining_qty=Decimal("5"))` with comment `"# post-update non-zero (no close trigger)"` — that comment assumed `update_position_state_after_fill` ran and decremented qty. The fixture value was artificial — realistic placement-time pre-update value would be the FULL qty (matching event qty), NOT `5`. The test asserted `reconcile_close.assert_not_called()` but did NOT assert `update_position_state_after_fill.assert_not_called()` — so the actual subtract bug (open fill decrementing remaining_qty in placement-time scenario) was masked behind the "post-update fake". Sibling B1-fix test `test_process_invokes_reconcile_close_when_remaining_zeroes` ENCODED the bug as expected behavior (exec_type='open' + remaining_qty=0 → reconcile_close called) — repurposed at fix-time as H-030 defensive-guard pin (assertion inverted to `assert_not_called`).
Active control: For state-mutation tests on functions that read state THEN mutate it, test fixtures MUST use REALISTIC pre-mutation entering values (e.g. placement-time `remaining_qty = full qty`, NOT artificial "post-update" placeholders). Test assertions MUST pin BOTH the mutation behavior (explicit `update_X.assert_called_once` / `assert_not_called` on EVERY mutating helper) AND the post-condition (downstream effect such as `reconcile_close` not called). Plan-reviewer Gate 1 MUST flag any test fixture with comments like `"post-update non-zero"` or `"artificial value to bypass X"` without explicit assertion pinning the SHOULD-NOT-BE-CALLED helper. Brief-reviewer Gate 3 MUST grep for `assert_not_called` count in tests touching `services/execution/` dispatcher / paper-exchange / reconcile paths — every state-mutation test must pin BOTH "what was called" AND "what was NOT called" sides of the contract.

## L-018 (fix(T-218c-paper-dispatcher-skip), operator-discovered shipped-code bug, 2026-05-08)
Pattern: Plan analysis claiming "bug dormant in <mode>" or "<mode> not affected by this issue" is itself a hazard if not backed by direct code-reading evidence. fix(T-218b) plan claimed "PaperExchange synthetic-fill flow does NOT emit ExecutionEvent for open fills via stream_executions (only SL/TP synthetic fills emit)" — this was reasoning, NOT verified evidence. Direct reading of `paper/adapter.py:820-831` shows `_persist_open` enqueues ExecutionEvent on every open fill (plus close at :929-940 + synthetic SL/TP at :1184-1194). The plan-doc's incorrect claim led to T-218b being framed as "live-mode-only" while paper mode had its own separate kill-path (`dispatcher.py:188` unattributable_fill RuntimeError) which would fire BEFORE the H-030 conditional even reached the `update_position_state_after_fill` call site. The H-030 fix is correct for live; H-031 (T-218c) addresses paper independently. Both fixes are needed.
Active control: For any plan claim of the shape "bug dormant in <mode>", "<mode> not affected by this issue", or "<feature> not exercised in <X> path", plan-reviewer Gate 1 MUST require explicit code citation (file:line + grep evidence) showing the relevant code path is not exercised. Reasoning-only "dormant" claims must be flagged as BLOCKER until code-verified. Brief-reviewer Gate 3 + math-validator Gate 4 should similarly verify any "X is dormant" claim against grep evidence at commit time. Operator-driven audit catches such claims; review system should catch them earlier at plan time. The pattern generalizes: any "no-impact" claim about a code path needs code-citation evidence, not reasoning.
